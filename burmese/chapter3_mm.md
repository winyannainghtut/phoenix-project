အခန်း ၃ (CHAPTER 3)

• အင်္ဂါနေ့၊ စက်တင်ဘာ ၂ ရက်

ကျွန်တော် Patty နဲ့ Wes တို့နောက်ကနေ NOC ကို ဖြတ်ပြီး cubicles တွေ အများကြီးရှိတဲ့ နေရာထဲကို လိုက်လာခဲ့တယ်။ ပြွတ်သိပ်နေတဲ့ cubicle ခြောက်ခုကို ပေါင်းထားတဲ့ အလုပ်ခွင် အကျယ်ကြီးတစ်ခုထဲကို ကျွန်တော်တို့ ရောက်သွားကြတယ်။ နံရံတစ်ဖက်မှာတော့ ကီးဘုတ်တစ်ခုနဲ့ LCD မော်နီတာ လေးလုံးပါတဲ့ စားပွဲကြီးတစ်လုံး ရှိနေပါတယ်၊ ဒါက Wall Street trading desk တစ်ခုနဲ့ တူနေပါတယ်။ နေရာတိုင်းမှာ စာဗာ (server) တွေ စုပုံနေပြီး အားလုံးက မီးလေးတွေ မှိတ်တုတ်မှိတ်တုတ် ဖြစ်နေကြတယ်။ စားပွဲရဲ့ နေရာတိုင်းမှာလည်း မော်နီတာတွေ အများကြီး ရှိနေပြီး graphs (ဂရပ်) တွေ၊ login windows တွေ၊ code editors တွေ၊ Word documents တွေနဲ့ ကျွန်တော် နားမလည်တဲ့ applications တွေ အများကြီးကို ပြသထားပါတယ်။

Brent ကတော့ ပတ်ဝန်းကျင်ကို လုံးဝ သတိမထားမိဘဲ ပြတင်းပေါက်တစ်ခုထဲမှာ စာရိုက်နေပါတယ်။ သူ့ဖုန်းကနေ NOC conference line (အစည်းအဝေးလိုင်း) အသံကို ကြားနေရတယ်။ အသံကျယ်ကျယ်နဲ့ ဖွင့်ထားတဲ့ စပီကာဖုန်းက ဘေးပတ်ဝန်းကျင်က လူတွေကို အနှောင့်အယှက် ဖြစ်စေမလားဆိုတာ သူ လုံးဝ စိုးရိမ်ပုံ မရပါဘူး။

“ဟေး Brent။ ခဏလောက် အားလား” လို့ Wes က သူ့ပုခုံးပေါ် လက်တင်ပြီး အသံကျယ်ကျယ်နဲ့ မေးလိုက်တယ်။

“ခဏ စောင့်လို့ ရမလား” လို့ Brent က မော့တောင် မကြည့်ဘဲ ပြန်ပြောတယ်။ “ကျွန်တော် အခု တကယ့်ကို အလုပ်ရှုပ်နေတာ။ SAN issue (SAN ပြဿနာ) ကို လုပ်နေတာလေ၊ သိတယ်မဟုတ်လား။”

Wes က ကုလားထိုင်တစ်လုံးကို ဆွဲထိုင်လိုက်တယ်။ “အေး၊ အဲဒီအကြောင်း ပြောဖို့ ငါတို့ ဒီကို လာတာ။”

Brent လှည့်လာတဲ့အခါ Wes က “မနေ့ညကအကြောင်း ပြန်ပြောပြဦး။ ဘာဖြစ်လို့ SAN upgrade က payroll run failure (လစာပေးစနစ် ကျရှုံးမှု) ကို ဖြစ်စေတယ်လို့ မင်း ကောက်ချက်ချလိုက်ရတာလဲ” လို့ ဆက်မေးလိုက်တယ်။

Brent က မျက်လုံးတစ်ချက် ဝေ့လိုက်ပြီး— “အားလုံး အိမ်ပြန်သွားတဲ့နောက်မှာ ကျွန်တော် SAN အင်ဂျင်နီယာ တစ်ယောက်ကို firmware upgrade လုပ်ဖို့ ကူညီပေးနေခဲ့တာ။ ကျွန်တော်တို့ ထင်ထားတာထက် အများကြီး ပိုကြာသွားတယ်—tech note (နည်းပညာ မှတ်စု) ထဲကအတိုင်း ဘာမှ ဖြစ်မလာဘူး။ တော်တော်လေးကို ခက်ခဲခဲ့ပေမဲ့ ညနေ ၇ နာရီလောက်မှာတော့ နောက်ဆုံး ပြီးသွားခဲ့တယ်။

“ကျွန်တော်တို့ SAN ကို reboot လုပ်လိုက်တယ်၊ ဒါပေမဲ့ self-tests တွေ အကုန်လုံး fail ဖြစ်ကုန်ရော။ ကျွန်တော်တို့ ဘာမှားသွားလဲဆိုတာကို ၁၅ မိနစ်လောက် ရှာဖွေခဲ့ကြသေးတယ်။ အဲဒီအချိန်မှာပဲ payroll run fail ဖြစ်သွားတဲ့အကြောင်း အီးမေးလ်တွေ ရောက်လာတော့တာပဲ။ အဲဒီမှာတင် ကျွန်တော်က ‘Game Over’ ပဲလို့ ပြောလိုက်တာ။

“ကျွန်တော်တို့တွေဗားရှင်း (version) တွေ နောက်ကျနေတာ အရမ်းများသွားပြီ။ SAN vendor ကလည်း ကျွန်တော်တို့ လုပ်လိုက်တဲ့ upgrade path (အဆင့်မြှင့်တင်မှု လမ်းကြောင်း) မျိုးကို စမ်းသပ်ထားဖူးမှာ မဟုတ်ဘူး။ ကျွန်တော် မင်းကို ဖုန်းဆက်ပြီး ဒါကို ရပ်လိုက်ချင်တယ်လို့ ပြောခဲ့တာပေါ့။ မင်း ခေါင်းညိတ်လိုက်တာနဲ့ ကျွန်တော်တို့ rollback (နောက်ဆုတ်ခြင်း) ကို စတင်ခဲ့ကြတာ။

“အဲဒီမှာပဲ SAN Crashed (SAN ပြတ်တောက်သွားခြင်း) ဖြစ်သွားတာပဲ” လို့ သူက ကုလားထိုင်ပေါ် မှီထိုင်ရင်း ပြောတယ်။ “အဲဒါက payroll တင်မကဘဲ တခြား server တွေ အများကြီးကိုလည်း အတူတူ ဆွဲချသွားခဲ့တာ။”

“ငါတို့ SAN firmware ကို upgrade လုပ်ဖို့ စဉ်းစားနေတာ နှစ်နဲ့ချီနေပြီ၊ ဒါပေမဲ့ မလုပ်ဖြစ်ခဲ့ဘူး” လို့ Wes က ကျွန်တော့်ဘက် လှည့်ပြီး ရှင်းပြတယ်။ “တစ်ခါတော့ လုပ်ခါနီးဖြစ်သွားသေးတယ်၊ ဒါပေမဲ့ ပြုပြင်ထိန်းသိမ်းဖို့ အချိန် (maintenance window) အလုံအလောက် မရခဲ့ဘူး။ စွမ်းဆောင်ရည် (performance) ကလည်း တစ်နေ့ထက်တစ်နေ့ ပိုပိုပြီး ညံ့လာလို့ အရေးကြီးတဲ့ application တွေ တော်တော်များများကို ထိခိုက်လာတဲ့အထိပဲ။ ဒါကြောင့် နောက်ဆုံးတော့ မနေ့ညက မဖြစ်ဖြစ်အောင် လုပ်မယ်ဆိုပြီး upgrade လုပ်လိုက်ကြတာပေါ့။”

ကျွန်တော် ခေါင်းညိတ်လိုက်တယ်။ အဲဒီအချိန်မှာပဲ ကျွန်တော့်ဖုန်း မြည်လာတယ်။

Ann ဆီကဖြစ်လို့ ကျွန်တော် စပီကာဖုန်း ဖွင့်လိုက်တယ်။

“ရှင် အကြံပြုသလိုပဲ၊ မနေ့က payroll database ထဲက data တွေကို ကျွန်မတို့ စစ်ဆေးပြီးပါပြီ။ ပြီးခဲ့တဲ့ လစာပေးတဲ့ ကာလတုန်းကတော့ အဆင်ပြေပါတယ်။ ဒါပေမဲ့ အခုအကြိမ်မှာတော့ factory hourlies (နာရီစား ဝန်ထမ်းတွေ) ရဲ့ Social Security numbers (ဆိုရှယ်စီကျူရီတီ နံပါတ်) တွေ အားလုံးက ဘာတွေမှန်း မသိအောင် ဖြစ်နေတယ်။ ပြီးတော့ သူတို့ရဲ့ အလုပ်လုပ်တဲ့ နာရီတွေနဲ့ လစာ နေရာမှာလည်း သုညတွေပဲ ဖြစ်နေတယ်။ ဘယ်သူမှ ဒီလိုမျိုး တစ်ခါမှ မမြင်ဖူးကြဘူး။”

“Field တစ်ခုတည်းကပဲ အဲဒီလို ဖြစ်နေတာလား” လို့ ကျွန်တော်က အံ့သြတကြီးနဲ့ မေးလိုက်တယ်။ “ ‘ဂီဘရစ်’ (gibberish) ဆိုတာ ဘာကို ပြောတာလဲ။ အဲဒီ နေရာတွေမှာ ဘာတွေ ရေးထားလို့လဲ။”

သူက screen ပေါ်မှာ မြင်နေရတာကို ရှင်းပြဖို့ ကြိုးစားနေပါတယ်။ “အင်း… အဲဒါတွေက ဂဏန်းတွေလည်း မဟုတ်ဘူး၊ စာလုံးတွေလည်း မဟုတ်ဘူး။ နှလုံးသားပုံတွေ၊ စပိတ်ပုံတွေနဲ့ တွန့်လိမ်နေတဲ့ စာလုံးတွေ… ပြီးတော့ umlauts (ဂျာမန်စာလုံးပါ အစက်လေးများ) ပါတဲ့ နိုင်ငံခြား စာလုံးတွေ အများကြီးပဲ… ပြီးတော့ space (နေရာလွတ်) တစ်ခုမှ မပါဘူး။ အဲဒါက အရေးကြီးသလားဟင်။”

Ann က line noise (လိုင်းပေါ်က အမှားအယွင်းများ) တွေကို အသံထွက်ဖတ်ဖို့ ကြိုးစားနေတာကို Brent က ခိုးရယ်နေတာကြောင့် ကျွန်တော် သူ့ကို ခပ်တည်တည်ပဲ ကြည့်လိုက်တယ်။ “ကျွန်တော်တို့ မြင်ယောင်လို့ ရပါပြီ” လို့ ပြောလိုက်တယ်။ “ဒါက တကယ် အရေးကြီးတဲ့ သဲလွန်စပဲ။ အဲဒီ corrupted data (ပျက်စီးနေသော အချက်အလက်) တွေပါတဲ့ spreadsheet ကို ကျွန်တော့်ဆီ ပို့ပေးလို့ ရမလား။”

သူက ပို့ပေးမယ်လို့ သဘောတူပါတယ်။ “ဒါနဲ့၊ အခု database တွေ အများကြီး ကျနေတာလား။ ဆန်းတော့ ဆန်းသား။ မနေ့ညကတော့ တက်နေသေးတယ်လေ။”

Wes က Brent တစ်ခုခု မပြောမိအောင် နှုတ်ပိတ်လိုက်ရင်း တစ်ခုခုကို တိုးတိုးလေး ညည်းလိုက်တယ်။

“အင်း… ဟုတ်ကဲ့။ အဲဒီ ပြဿနာကို ကျွန်တော်တို့ သိပါတယ်၊ အခုလည်း ဖြေရှင်းနေပါတယ်” လို့ ကျွန်တော်က အမူအရာ တည်တည်နဲ့ပဲ ပြန်ပြောလိုက်တယ်။

ဖုန်းချလိုက်တဲ့အခါမှာတော့ ကျွန်တော် သက်ပြင်းချလိုက်မိတယ်၊ ပြဿနာတွေကို ဖြေရှင်းနေရတဲ့ လူတွေကို စောင့်ရှောက်ပေးတဲ့ နတ်ဘုရား တစ်ပါးပါးကိုလည်း စိတ်ထဲကနေ ကျေးဇူးတင်လိုက်မိတယ်။

“Database ထဲက field တစ်ခုတည်းကပဲ ပျက်စီးသွားတာ ဟုတ်လား။ ကဲ… သူငယ်ချင်းတို့၊ အဲဒါကတော့ SAN failure ကြောင့် ဖြစ်တာ မဟုတ်တာ သေချာသလောက်ပဲ” လို့ ကျွန်တော် ပြောလိုက်တယ်။ “Brent၊ မနေ့က SAN upgrade အပြင် payroll run failure ကို ဖြစ်စေနိုင်မယ့် တခြား ဘာတွေ ဖြစ်ခဲ့သေးလဲ။”

Brent က ကုလားထိုင်ပေါ်မှာ မှီထိုင်ရင်း စဉ်းစားနေပါတယ်။ “အင်း… ခုမှ သတိရတယ်။ မနေ့က timekeeping application ရဲ့ developer တစ်ယောက်က ကျွန်တော့်ဆီ ဖုန်းဆက်ပြီး database table structure (ဒေတာဘေ့စ် ဇယား တည်ဆောက်ပုံ) အကြောင်း ထူးထူးဆန်းဆန်း မေးခွန်းတစ်ခု လာမေးသေးတယ်။ အဲဒီတုန်းက ကျွန်တော်က Phoenix test VM ကို လုပ်နေတာဆိုတော့ အလုပ်ကို မြန်မြန် ပြန်လုပ်လို့ရအောင် သူ့ကို အဖြေအမြန်ပေးပြီး လွှတ်လိုက်တာ။ သူက app ကို ဖျက်ဆီးမယ့် အလုပ်တစ်ခုခု လုပ်လိုက်တာလို့များ မင်း ထင်နေတာလား။”

Wes က NOC conference call လိုင်းဖွင့်ထားတဲ့ စပီကာဖုန်းဆီကို ချက်ချင်း လှည့်လိုက်ပြီး အသံပြန်ဖွင့်လိုက်တယ်။ “ဟေး လူတို့၊ ဒါ Wes ပါ။ ငါနဲ့အတူ Brent၊ Patty တို့ ရှိနေသလို ငါတို့ရဲ့ အထက်လူကြီးသစ် Bill Palmer လည်း ရှိနေတယ်။ Steve Masters က သူ့ကို IT Ops တစ်ခုလုံးရဲ့ တာဝန်ခံအဖြစ် ခန့်လိုက်ပြီ။ ဒါကြောင့် လူတို့ သေချာ နားထောင်ကြနော်။”

ငါ့ရဲ့ ရာထူးအသစ်ကို စနစ်တကျ ကြေညာချင်တဲ့ ဆန္ဒကတော့ ပိုပြီး ဝေးကွာသွားသလိုပါပဲ။

Wes က ဆက်ပြောတယ်၊ “စက်ရုံတွေမှာ သုံးနေတဲ့ timekeeping application ကို developer တစ်ယောက်ယောက်က တစ်ခုခု ပြောင်းလဲလိုက်တာမျိုး သိကြတဲ့လူ ရှိလား။ Brent ဆီကို database tables တွေ ပြောင်းလဲတဲ့ကိစ္စနဲ့ ပတ်သက်ပြီး တစ်ယောက်ယောက်က ဖုန်းဆက်မေးတယ်လို့ ပြောတယ်လေ။”

စပီကာဖုန်းထဲကနေ အသံတစ်ခု ထွက်လာတယ်၊ “ဟုတ်ကဲ့၊ စက်ရုံတွေနဲ့ connectivity issues (ချိတ်ဆက်မှု ပြဿနာ) တက်နေတဲ့ တစ်ယောက်ကို ကျွန်တော် ကူညီပေးခဲ့သေးတယ်။ သူက timekeeping app ကို ထိန်းသိမ်းနေတဲ့ developer တစ်ယောက်လို့ ကျွန်တော် ထင်ပါတယ်။ သူက အခုအပတ်မှာ John အမြန်လုပ်ခိုင်းနေတဲ့ security application (လုံခြုံရေးဆော့ဖ်ဝဲ) တစ်ခုကို တပ်ဆင်နေတာတဲ့။ သူ့နာမည်က Max လား မသိဘူး။ သူ့ရဲ့ ဆက်သွယ်ရမယ့် အချက်အလက်တွေ တစ်နေရာရာမှာ ရှိဦးမယ် ထင်တယ်… သူက ဒီနေ့ပဲ ခွင့်ယူတော့မှာမို့လို့ ဒါကို အမြန် လုပ်နေရတာလို့ ပြောပါတယ်။”

အခုမှ လမ်းကြောင်း အမှန်ပေါ် ရောက်သွားပြီ။

Developer တစ်ယောက်က ခွင့်မယူခင်မှာ အရေးပေါ် အပြောင်းအလဲတစ်ခုကို ဇွတ်အတင်း လုပ်သွားတာပေါ့—ဒါကလည်း ကျွန်တော်တို့ရဲ့ CISO (Chief Information Security Officer) ဖြစ်တဲ့ John Pesche ခိုင်းတာ ဖြစ်နိုင်ပါတယ်။

ဒီလို အခြေအနေမျိုးက developer တွေအပေါ် ထားရှိတဲ့ ကျွန်တော့်ရဲ့ သံသယကို ပိုပြီး ခိုင်မာစေပါတယ်။ သူတို့က တစ်ခုခုကို ပေါ့ပေါ့ဆဆ လုပ်ပြီး ဖျက်ဆီးပစ်တတ်ကြတယ်၊ ပြီးရင်တော့ ပျောက်သွားကြပြီး Operations သမားတွေကပဲ အဲဒီ နောက်ဆက်တွဲတွေကို လိုက်ရှင်းရတော့တာပေါ့။

Developer ထက် ပိုပြီး အန္တရာယ်ရှိတဲ့ အရာကတော့ Security (လုံခြုံရေး) နဲ့ ပေါင်းထားတဲ့ developer ပါပဲ။ ဒီနှစ်ခု ပေါင်းသွားပြီဆိုရင်တော့ အကြောင်းပြချက်ရော၊ လုပ်ရပ်ရော၊ အခွင့်အရေးရော အားလုံး ရှိသွားပါပြီ။

ကျွန်တော် ထင်တာကတော့ ကျွန်တော်တို့ရဲ့ CISO က Development manager တစ်ယောက်ကို တစ်ခုခု လုပ်ခိုင်းလိုက်မယ်၊ အဲဒီနောက်မှာ developer က တခြားတစ်ခုခု ထပ်လုပ်လိုက်ပြီး နောက်ဆုံးမှာတော့ payroll run ပျက်စီးသွားတာ ဖြစ်ပါလိမ့်မယ်။

Information Security သမားတွေက လူတွေကို အမြဲ ဖိအားပေးတတ်ပြီး တခြားဌာနတွေအပေါ် ဘယ်လို သက်ရောက်မှုရှိမလဲဆိုတာကို ဂရုမစိုက်ဘဲ အရေးပေါ် လုပ်ခိုင်းတတ်ကြတယ်၊ အဲဒါကြောင့်ပဲ သူတို့ကို အစည်းအဝေးတွေမှာ သိပ်မဖိတ်ကြတာပေါ့။ တစ်ခုခုကို မပြီးမြောက်အောင် လုပ်ဖို့ အကောင်းဆုံးနည်းလမ်းကတော့ သူတို့ကို အခန်းထဲမှာ ထည့်ထားလိုက်ဖို့ပါပဲ။

သူတို့က အမြဲတမ်း အကြောင်းပြချက် ပေါင်းစုံနဲ့ ကျွန်တော်တို့ လုပ်သမျှ အရာတိုင်းက security hole (လုံခြုံရေး အပေါက်) တွေ ဖြစ်လာနိုင်ပြီး အာကာသက hackers တွေက အဲဒါကို အသုံးချပြီး ကုမ္ပဏီတစ်ခုလုံးကို ဖျက်ဆီးမယ်၊ ကုမ္ပဏီရဲ့ code တွေ၊ မူပိုင်ခွင့်တွေ၊ ခရက်ဒစ်ကတ် နံပါတ်တွေနဲ့ မိသားစု ဓာတ်ပုံတွေကိုတောင် ခိုးယူသွားလိမ့်မယ်လို့ ပြောလေ့ရှိကြပါတယ်။ ဒါတွေက တကယ်ဖြစ်နိုင်တဲ့ အန္တရာယ်တွေ ဖြစ်နိုင်ပေမဲ့ သူတို့ရဲ့ စူးစူးဝါးဝါးနဲ့ မောက်မောက်မာမာ တောင်းဆိုမှုတွေက လက်တွေ့မှာ ကျွန်တော်တို့ ပတ်ဝန်းကျင်ရဲ့ လုံခြုံရေးကို ဘယ်လို တိုးတက်စေမလဲဆိုတာကိုတော့ ကျွန်တော် ဆက်စပ်လို့ မရပါဘူး။

“ကဲ အားလုံးပဲ” လို့ ကျွန်တော်က ခိုင်မာတဲ့ လေသံနဲ့ ပြောလိုက်တယ်။ “ဒီ payroll run failure က ရာဇဝတ်မှု ခင်းကျင်းရာ နေရာတစ်ခုလိုပဲ၊ ငါတို့ကတော့ စုံထောက်တွေပေါ့။ SAN ကတော့ တရားခံ မဟုတ်တော့ပါဘူး၊ ဒါပေမဲ့ ကံမကောင်းစွာနဲ့ပဲ ငါတို့ရဲ့ စစ်ဆေးမှုတွေလုပ်ရင်း သူ့ကို မတော်တဆ ထိခိုက်လိုက်မိပြီ။ Brent၊ မင်းကတော့ ထိခိုက်သွားတဲ့ SAN ကို ဆက်ပြီး ပြင်လိုက်ဦး—အမြန်ဆုံး ပြန်သုံးလို့ရအောင် လုပ်ဖို့ လိုတယ်။

“Wes နဲ့ Patty၊ ငါတို့ အဓိက စောင့်ကြည့်ရမယ့် လူတွေကတော့ Max နဲ့ သူ့ရဲ့ မန်နေဂျာပဲ” လို့ ကျွန်တော်က ပြောတယ်။ “ဘာပဲလုပ်ရလုပ်ရ သူတို့ကို ရှာပါ၊ ဖမ်းထားပါ၊ သူတို့ ဘာလုပ်လိုက်သလဲဆိုတာကို ရှာဖွေပါ။ Max က ခွင့်ယူထားတာလည်း ငါ ဂရုမစိုက်ဘူး။ သူ တစ်ခုခု မှားယွင်းသွားတာ သေချာတယ်၊ ငါတို့ ညနေ ၃ နာရီမှာ ဒါကို အမှန်ပြင်ရမယ်။”

ကျွန်တော် ခဏစဉ်းစားလိုက်တယ်။ “ငါ John ကို သွားရှာလိုက်ဦးမယ်။ မင်းတို့ထဲက ဘယ်သူ လိုက်ချင်သေးလဲ။”

Wes နဲ့ Patty တို့က John ကို စစ်ဆေးမယ့်နေရာမှာ ဘယ်သူ ပါဝင်မလဲဆိုတာကို ငြင်းခုံနေကြတယ်။ Patty က “ကျွန်မပဲ သွားသင့်တယ်။ John ရဲ့ လူတွေကို စည်းကမ်းရှိအောင် ကျွန်မ နှစ်ပေါင်းများစွာ ထိန်းကျောင်းလာခဲ့တာ။ သူတို့က ဘယ်တော့မှ ငါတို့ရဲ့ process (လုပ်ငန်းစဉ်) အတိုင်း မလုပ်ကြဘူး၊ အဲဒါက အမြဲတမ်း ပြဿနာ တက်စေတယ်။ Steve နဲ့ Dick တို့က သူ့ကို ဒီလိုမျိုး လုပ်ရပ်အတွက် အပြစ်ပေးတာကို ကျွန်မ တကယ် မြင်ချင်တယ်” လို့ တင်းတင်းမာမာ ပြောလိုက်တယ်။

ဒါက တကယ်ကို ခိုင်လုံတဲ့ အကြောင်းပြချက် ဖြစ်ပုံရပါတယ်၊ Wes က “အေးပါ၊ မင်းပဲ သွားလိုက်တော့။ သူ့အတွက် ငါ တကယ်ပဲ စိတ်မကောင်း ဖြစ်မိပါရဲ့” လို့ ပြောလိုက်လေရဲ့။

ကျွန်တော် ပြောလိုက်တဲ့ စကားလုံးတွေအတွက် ရုတ်တရက် နောင်တရမိသွားတယ်။ ဒါက လူသတ်သမားကို လိုက်ရှာနေတာမျိုး မဟုတ်သလို၊ လက်စားချေဖို့လည်း ကျွန်တော် မရည်ရွယ်ပါဘူး။ ကျွန်တော်တို့ လိုအပ်တာက ပြဿနာ မတက်ခင်အထိ ဖြစ်ခဲ့သမျှ အရာအားလုံးရဲ့ တိကျတဲ့ timeline တစ်ခုပဲ ဖြစ်ပါတယ်။

မနေ့ညက မသင့်တော်တဲ့ ကောက်ချက်တွေကို ဇွတ်ချလိုက်မိလို့ SAN failure ဖြစ်ခဲ့ရတာပါ။ ဒီလိုမျိုး အမှားတွေကို ငါတို့ ထပ်မလုပ်ရတော့ဘူး။ ငါ တာဝန်ယူနေတဲ့ အချိန်မှာတော့ ဒါမျိုး မဖြစ်စေရဘူး။

Patty နဲ့ ကျွန်တော် John ဆီ ဖုန်းဆက်တဲ့အခါ Patty ရဲ့ screen ပေါ်က ဖုန်းနံပါတ်ကို ကျွန်တော် မျက်မှောင်ကြုတ်ပြီး ကြည့်လိုက်မိတယ်၊ ကျွန်တော့် မိန်းမ ပြောသလို မျက်မှန် တပ်ရတော့မယ့် အချိန် ရောက်ပြီလားလို့ တွေးမိရင်းပေါ့။ အသက် လေးဆယ် ဆိုတာလည်း နီးနေပြီဆိုတဲ့ အသိက ထပ်ပြီး သတိပေးနေသလိုပါပဲ။

ဖုန်းနံပါတ် နှိပ်လိုက်တဲ့အခါ တစ်ခါတည်းနဲ့တင် ဖုန်းကိုင်လိုက်ပြီး— “John ပြောနေပါတယ်” တဲ့။

ကျွန်တော်က payroll နဲ့ SAN failure အကြောင်း သူ့ကို အမြန် ပြောပြလိုက်ပြီး “မနေ့က timekeeping application ကို တစ်ခုခု ပြောင်းလဲလိုက်တာမျိုး ရှိသလား” လို့ မေးလိုက်တယ်။

သူက “အဲဒါ တကယ် စိတ်မကောင်းစရာပဲ။ ဒါပေမဲ့ ငါတို့က မင်းတို့ရဲ့ midrange systems တွေကို ဘာမှ မပြောင်းလဲခဲ့ဘူးလို့ ငါ အာမခံနိုင်ပါတယ်။ မကူညီနိုင်တာ စိတ်မကောင်းပါဘူး” လို့ ပြန်ပြောတယ်။

ကျွန်တော် သက်ပြင်းချလိုက်မိတယ်။ အခုလောက်ဆို Steve ဒါမှမဟုတ် Laura က ကျွန်တော့်ရဲ့ ရာထူးတိုးတဲ့ သတင်းကို ထုတ်ပြန်လိုက်ပြီလို့ ထင်နေခဲ့တာ။ လူတိုင်းနဲ့ စကားပြောတိုင်း ကျွန်တော့်ရဲ့ ရာထူးအသစ်အကြောင်းကိုပဲ လိုက်ရှင်းပြနေရတာက ကျွန်တော့် ကံကြမ္မာ ဖြစ်ပုံရပါတယ်။

ဒီသတင်းကို ကျွန်တော်ကိုယ်တိုင်ပဲ ကြေညာလိုက်ရင် ပိုလွယ်သွားမလားလို့တောင် စဉ်းစားမိတယ်။

“Luke နဲ့ Damon တို့ မရှိတော့ဘူး ဟုတ်လား” John က အံ့သြသွားပုံရတယ်။ “Compliance audit finding (လိုက်နာမှုဆိုင်ရာ စစ်ဆေးတွေ့ရှိချက်) ကြောင့် သူတို့ နှစ်ယောက်စလုံးကို Steve က တကယ် အလုပ်ဖြုတ်လိမ့်မယ်လို့ မထင်ထားခဲ့ဘူး။ ဒါပေမဲ့ ဘယ်သူ သိနိုင်မှာလဲ။ ဒီမှာလည်း အပြောင်းအလဲတွေ စတင်နေပြီ ထင်တယ်။ ဒါကို သင်ခန်းစာယူပါ Bill။ မင်းတို့ Operations သမားတွေက security issue (လုံခြုံရေး ပြဿနာ) တွေအပေါ်မှာ အချိန်ဆွဲနေလို့ မရတော့ဘူး! ဒါ ငါ့ရဲ့ မိတ်ဆွေတစ်ယောက်အနေနဲ့ ပေးတဲ့ အကြံပေးချက်ပါပဲ…

“အဲဒီအကြောင်း ပြောရမယ်ဆိုရင် ပြိုင်ဘက်တွေက ငါတို့အပေါ်မှာ ဘယ်လိုမျိုး အမြဲတမ်း ကျော်တက်နေတာလဲဆိုတာ ငါ သံသယ ရှိနေတယ်” လို့ သူက ဆက်ပြောတယ်။ “စကားပုံတစ်ခု ရှိသလိုပေါ့၊ တစ်ကြိမ်ဖြစ်ရင် မတော်တဆ၊ နှစ်ကြိမ်ဖြစ်ရင် တိုက်ဆိုင်မှု၊ သုံးကြိမ်ဖြစ်ရင်တော့ ဒါဟာ ရန်သူရဲ့ လုပ်ရပ်ပဲ။ ငါတို့ အရောင်းဝန်ထမ်းတွေရဲ့ အီးမေးလ် စနစ်တွေ hacked လုပ်ခံထားရတာလည်း ဖြစ်နိုင်တယ်။ အဲဒါကြောင့် ငါတို့ ဒီလောက်အထိ အရှုံးပေါ်နေတာပေါ့။”

John ကတော့ စကားတွေ ဆက်ပြောနေပေမဲ့ ကျွန်တော့်စိတ်ထဲမှာတော့ Luke နဲ့ Damon တို့ဟာ security ကိစ္စတစ်ခုခုကြောင့် အလုပ်ဖြုတ်ခံရတာ ဖြစ်နိုင်တယ်ဆိုတဲ့ စကားမှာပဲ တန့်နေမိတယ်။ ဒါက ဖြစ်နိုင်ပါတယ်—John က Steve တို့၊ ဘုတ်အဖွဲ့တို့၊ စာရင်းစစ်တွေတို့နဲ့ အမြဲ ဆက်ဆံနေရတာကိုး။

ဒါပေမဲ့ Steve က သူတို့ကို အလုပ်ဖြုတ်တဲ့ အကြောင်းရင်းတွေပြောတုန်းက John အကြောင်းရော Information Security အကြောင်းကိုရော ဟစပ်လို့တောင် မပြောခဲ့ပါဘူး—Phoenix ကိုပဲ အဓိက အာရုံစိုက်ဖို့ ပြောခဲ့တာပါ။

ကျွန်တော် Patty ကို မေးခွန်းထုတ်တဲ့ ပုံစံနဲ့ ကြည့်လိုက်မိတယ်။ သူကတော့ မျက်လုံးတစ်ချက် ဝေ့လိုက်ပြီး ခေါင်းကို လက်ညှိုးနဲ့ လှည့်ပြတယ် (ရူးနေတယ်လို့ ဆိုလိုတာပါ)။ John ရဲ့ စိတ်ကူးယဉ်မှုတွေကတော့ တကယ် ရူးကြောင်ကြောင်နိုင်တယ်လို့ သူ ထင်မှာ သေချာပါတယ်။

“Steve က ရုံးဖွဲ့စည်းပုံ အသစ်အကြောင်း ဘာမှ ရှင်းပြမထားဘူးလား” လို့ ကျွန်တော်က တကယ် သိချင်စိတ်နဲ့ မေးလိုက်မိတယ်—John ကတော့ information security ကို အမြဲတမ်း ဦးစားပေး အဆင့်မှာ ထားလေ့မရှိဘူးဆိုပြီး အမြဲ ညည်းတွားလေ့ ရှိတာပါ။ သူက CIO နဲ့ တန်းတူ VP ရာထူးကို ရချင်ကြောင်း၊ အဲဒီလိုမှပဲ အကျိုးစီးပွား ပဋိပက္ခတွေကို ဖြေရှင်းနိုင်မယ်လို့ ပြောလေ့ရှိတာပါ။

ကျွန်တော် သိရသလောက်တော့ သူ အောင်မြင်ခဲ့တာ မရှိပါဘူး။

Luke နဲ့ Damon တို့က John ကို အလုပ်တကယ် လုပ်နေတဲ့ လူတွေကြားထဲ မနှောင့်ယှက်နိုင်အောင် တတ်နိုင်သမျှ ဖယ်ထုတ်ထားကြတာ လူတိုင်း သိကြပါတယ်။ John ကတော့ သူတို့ ဘယ်လောက်ပဲ တားတား အစည်းအဝေးတွေကိုတော့ မဖြစ်ဖြစ်အောင် တက်နိုင်နေတုန်းပါပဲ။

“ဘာလဲ။ ဘာဖြစ်နေလဲဆိုတာ ငါ လုံးဝ မသိဘူး” လို့ သူက စိတ်တိုတိုနဲ့ ပြောလိုက်တယ်၊ ကျွန်တော့် မေးခွန်းက သူ့ကို ထိသွားပုံရတယ်။ “ငါကတော့ အမြဲတမ်း မှောင်ထဲမှာပဲ နေခဲ့ရတာပါ။ ဒီတစ်ခါလည်း ငါကပဲ နောက်ဆုံးမှ သိရမယ့်လူ ဖြစ်ပုံရပါတယ်။ မင်း မပြောမချင်း ငါက ငါ့ရဲ့ အထက်လူကြီးက Luke လို့ပဲ ထင်နေခဲ့တာ။ အခု သူ မရှိတော့ဘူးဆိုရင် ငါ ဘယ်သူ့ဆီ သတင်းပို့ရမှန်းတောင် မသိတော့ဘူး။ မင်း Steve ဆီက ဖုန်းရခဲ့လို့လား။”

“ဒါတွေက ငါ့ရာထူးထက် အများကြီး ပိုမြင့်တဲ့လူတွေ ဆုံးဖြတ်တဲ့ ကိစ္စတွေပါ—ငါလည်း မင်းလိုပဲ ဘာမှ မသိပါဘူး” လို့ ကျွန်တော်က ဟန်ဆောင်ပြီး ပြန်ပြောလိုက်တယ်။ စကားလမ်းကြောင်း အမြန်ပြောင်းဖို့အတွက် “timekeeping app အပြောင်းအလဲအကြောင်း မင်း ဘာသိလဲ” လို့ မေးလိုက်မိတယ်။

“ငါ Steve ဆီ ဖုန်းဆက်ပြီး ဘာဖြစ်နေလဲဆိုတာ မေးကြည့်ဦးမယ်။ သူကတော့ Information Security ဆိုတာ ရှိမှန်းတောင် မေ့နေပုံရတယ်” လို့ သူက ဆက်ပြောနေပြန်တော့ payroll အကြောင်း ငါတို့ ပြောဖြစ်ကြပါဦးမလားလို့တောင် တွေးမိသွားတယ်။

ကျွန်တော် စိတ်သက်သာရာ ရသွားတာကတော့ နောက်ဆုံးမှာ သူက “အိုကေ၊ ဟုတ်ပြီ၊ မင်း Max အကြောင်း မေးနေတာပေါ့။ ငါတို့မှာ data တွေကို သိမ်းဆည်းတဲ့ နေရာမှာ PII (Social Security numbers လိုမျိုး လူတစ်ဦးချင်းစီကို ဖော်ပြနိုင်တဲ့ အချက်အလက်တွေ) သိမ်းတဲ့ ကိစ္စနဲ့ ပတ်သက်ပြီး အရေးကြီးတဲ့ audit issue (စစ်ဆေးမှု ပြဿနာ) တစ်ခု ရှိနေခဲ့တယ်။ မင်း သိတဲ့အတိုင်း SSNs တွေ၊ မွေးနေ့တွေ စတာတွေကိုပေါ့။ European Union ဥပဒေရော အမေရိကန် ပြည်နယ် ဥပဒေတွေကပါ အဲဒီလို data တွေကို သိမ်းထားတာမျိုးကို တားမြစ်ထားကြတယ်။ ငါတို့ ဒါနဲ့ ပတ်သက်ပြီး စစ်ဆေးတွေ့ရှိချက် အကြီးအကျယ် ရခဲ့တာ။ ငါ့ အဖွဲ့အနေနဲ့ ဒီကုမ္ပဏီကို ကယ်တင်ဖို့နဲ့ ထပ်ပြီး အပြောခံရတာမျိုး မဖြစ်အောင် ငါ လုပ်ရတာပေါ့။ အဲဒါက တကယ်ကို သတင်းတွေရဲ့ ရှေ့ဆုံးစာမျက်နှာမှာ ပါသွားနိုင်တဲ့ ကိစ္စလေ၊ သိတယ်မဟုတ်လား။”

သူက ဆက်ပြောတယ်၊ “အချက်အလက်တွေကို tokenized လုပ်ပေးနိုင်တဲ့ product တစ်ခုကို ငါတို့ ရှာတွေ့ထားတယ်၊ အဲဒါဆိုရင် SSNs တွေကို သိမ်းထားစရာ မလိုတော့ဘူး။ အဲဒါကို လွန်ခဲ့တဲ့ တစ်နှစ်လောက်ကတည်းက တပ်ဆင်ဖို့ လုပ်ခဲ့တာ၊ ဒါပေမဲ့ ငါ ဘယ်လောက်ပဲ တောင်းဆိုတောင်းဆို ဘာမှ ဖြစ်မလာခဲ့ဘူး။ အခုတော့ အချိန် မရှိတော့ဘူး။ PCI (Payment Card Industry) စားရင်းစစ်တွေက ဒီလထဲမှာ လာတော့မှာဆိုတော့ timekeeping အဖွဲ့နဲ့ ပူးပေါင်းပြီး ဒီအလုပ်ကို အမြန်ဆုံး ပြီးအောင် လုပ်လိုက်ရတာပေါ့။”

ကျွန်တော် ဖုန်းကို ကိုင်ရင်း ဘာပြောရမှန်းမသိ ဆွံ့အသွားမိတယ်။

တစ်ဖက်မှာတော့ John ရဲ့ လက်ထဲမှာ သက်သေရှိနေပြီဖြစ်လို့ ကျွန်တော် အရမ်း ဝမ်းသာသွားတယ်။ John ပြောတဲ့ SSN field ဆိုတာ Ann ပြောပြတဲ့ corrupted data နဲ့ အတိအကျ တိုက်ဆိုင်နေတာကိုး။

ဒါပေမဲ့ တခြားတစ်ဖက်မှာတော့– “ငါ နားလည်တာ မှန်မမှန် ပြောပြပါဦး…” လို့ ကျွန်တော် ဖြည်းဖြည်းချင်း မေးလိုက်တယ်။ “မင်းက audit finding တစ်ခုကို ပြင်ဖို့အတွက် ဒီ tokenization application ကို သွားထည့်လိုက်တယ်၊ အဲဒါက payroll run fail ဖြစ်အောင် လုပ်လိုက်ပြီး Dick နဲ့ Steve တို့ကို တကယ်ကို ဒေါသထွက်စေခဲ့တာပေါ့၊ ဟုတ်လား။”

John က စိတ်ဆိုးမာန်ဆိုးနဲ့ ပြန်ဖြေတယ်၊ “ပထမဆုံးအနေနဲ့၊ tokenization security product ကြောင့် ဒီပြဿနာ ဖြစ်ပေါ်လာတာ မဟုတ်ဘူးဆိုတာ ငါ အပြည့်အဝ ယုံကြည်တယ်။ ဒါဟာ မဖြစ်နိုင်တဲ့ ကိစ္စပဲ။ vendor က ဒါက စိတ်ချရပါတယ်လို့ ငါတို့ကို အာမခံထားသလို ငါတို့လည်း အားလုံးကို အသေးစိတ် စစ်ဆေးပြီးသားပါ။ ဒုတိယအချက်အနေနဲ့ကတော့ Dick နဲ့ Steve တို့ ဒေါသထွက်နေကြတာက အကြောင်းပြချက် ရှိပါတယ်—Compliance (လိုက်နာမှု) ဆိုတာ ရွေးချယ်စရာ မဟုတ်ဘူး။ ဒါဟာ ဥပဒေပဲ။ ငါ့အလုပ်က သူတို့ကို ထောင်ဝတ်စုံ (orange jumpsuits) တွေ ဝတ်ရမယ့် ဘေးကနေ ကာကွယ်ပေးဖို့ပဲ၊ အဲဒါကြောင့် ငါ လုပ်သင့်တာကို ငါ လုပ်ခဲ့တာ။”

“ ‘ထောင်ဝတ်စုံ’ ဟုတ်လား။”

“ထောင်ထဲမှာ ဝတ်ရတာမျိုးလေ” လို့ သူက ပြောတယ်။ “ငါ့အလုပ်က ကုမ္ပဏီရဲ့ အကြီးအကဲတွေကို ဥပဒေတွေ၊ စည်းမျဉ်းတွေနဲ့ သဘောတူညီချက်တွေ အားလုံးကို လိုက်နာနေအောင် လုပ်ဆောင်ပေးဖို့ပဲ။ Luke နဲ့ Damon တို့က အရမ်း ပေါ့ဆကြတာ။ သူတို့က စာရင်းစစ်နဲ့ လုံခြုံရေးပိုင်းမှာ ဖြစ်သင့်တာထက် အများကြီး လျှော့ပြီး လုပ်ခဲ့ကြတယ်။ ငါသာ အခုလို မလုပ်ခဲ့ရင် ငါတို့အားလုံး အခုလောက်ဆို ထောင်ထဲ ရောက်နေကြလောက်ပြီ။”

ငါတို့က payroll failure အကြောင်း ပြောနေတာပါ၊ စိတ်ကူးယဉ် ရဲအဖွဲ့က ထောင်ထဲ ထည့်မယ့်အကြောင်း ပြောနေတာ မဟုတ်ပါဘူး။

“John၊ တစ်ခုခုကို production ထဲကို အပြောင်းအလဲလုပ်ပြီး ထည့်တော့မယ်ဆိုရင် ဘယ်လို လုပ်ဆောင်ရမလဲဆိုတဲ့ process (လုပ်ငန်းစဉ်) တွေ၊ procedure (လုပ်ထုံးလုပ်နည်း) တွေ ရှိပါတယ်” လို့ Patty က ပြောလိုက်တယ်။ “မင်း အဲဒါတွေကို ကျော်ပြီး လုပ်ခဲ့လို့ အခုလို ပြဿနာ အကြီးအကျယ် တက်နေတာပေါ့၊ ငါတို့လည်း အဲဒါတွေကို လိုက်ပြင်နေရပြီ။ ဘာလို့ မင်း process အတိုင်း မလုပ်ခဲ့တာလဲ။”

“ဟား! ကောင်းလိုက်တာ Patty” John က တဟားဟား ရယ်လိုက်တယ်။ “ငါ process အတိုင်း လုပ်ခဲ့တာပဲ။ မင်းရဲ့ လူတွေ ငါ့ကို ဘာပြောလဲ သိလား။ နောက်ထပ် deployment window (တပ်ဆင်လို့ရမယ့် အချိန်) က နောက်ထပ် လေးလနေမှ ရနိုင်မယ်တဲ့။ ဟယ်လို? စာရင်းစစ်တွေက နောက်အပတ်ကျရင် ရောက်လာတော့မှာလေ!”

သူက တင်းတင်းမာမာ ဆက်ပြောတယ်၊ “မင်းတို့ရဲ့ အောက်ခြေအဆင့် အဝိုင်းအဝိုင်း လည်နေတဲ့ process ထဲမှာ ပိတ်မိနေဖို့ဆိုတာ လုံးဝ မဖြစ်နိုင်တဲ့ ကိစ္စပဲ။ မင်းသာ ငါ့နေရာမှာဆိုရင်လည်း ဒီလိုပဲ လုပ်မှာပဲ။”

Patty မျက်နှာ နီမြန်းလာတယ်။ ကျွန်တော်ကတော့ အေးအေးဆေးဆေးပဲ “Dick ရဲ့ အပြောအရဆိုရင် ငါတို့မှာ timekeeping app ကို ပြန်ကောင်းအောင် လုပ်ဖို့ လေးနာရီတောင် မပြည့်တော့ဘူး။ အခုတော့ SSNs တွေကို ထိခိုက်စေတဲ့ အပြောင်းအလဲတစ်ခု ရှိခဲ့တယ်ဆိုတာ သိရပြီဆိုတော့ ငါတို့ လိုအပ်တာကို ရပြီလို့ ထင်ပါတယ်။”

ကျွန်တော် ဆက်ပြောတယ်၊ “ဒီအလုပ်မှာ ကူညီပေးခဲ့တဲ့ Max က ဒီနေ့ ခွင့်ယူထားတယ်။ Wes ဒါမှမဟုတ် Brent က မင်းဆီကို ဖုန်းဆက်ပြီး ဒီ tokenization product အကြောင်း ပိုသိရအောင် မေးလိမ့်မယ်။ မင်းတို့ အတတ်နိုင်ဆုံး ကူညီပေးလိမ့်မယ်လို့ ငါ မျှော်လင့်ပါတယ်။ ဒါက တကယ် အရေးကြီးတယ်။”

John က သဘောတူတဲ့အခါမှာတော့ ကျွန်တော် ကျေးဇူးတင်စကား ပြောလိုက်တယ်။ “ဒါနဲ့ နောက်ဆုံး မေးခွန်းတစ်ခု။ ဘာဖြစ်လို့ ဒီ product က ပြဿနာ မဖြစ်စေနိုင်ဘူးလို့ မင်း ထင်နေရတာလဲ။ မင်း အဲဒီ အပြောင်းအလဲကို စမ်းသပ်စစ်ဆေး (test) ခဲ့သေးလား။”

ဖုန်းထဲမှာ ခဏတာ တိတ်ဆိတ်သွားပြီးနောက် John က “မရှိဘူး၊ ငါတို့ စမ်းသပ်လို့ မရခဲ့ဘူး။ Test environment (စမ်းသပ်ရန် ပတ်ဝန်းကျင်) ဆိုတာ မရှိဘူးလေ။ မင်းတို့လူတွေ လွန်ခဲ့တဲ့ နှစ်ပေါင်းများစွာကတည်းက ဘတ်ဂျက် တောင်းခဲ့တယ်လို့ ကြားဖူးပေမဲ့…”

ကျွန်တော် သိထားသင့်တာပေါ့။

“အင်း… ဒါ သတင်းကောင်းပါပဲ” လို့ John ဖုန်းချသွားတဲ့နောက် Patty က ပြောတယ်။ “ဒါကို ပြင်ဖို့ကတော့ မလွယ်နိုင်ပေမဲ့ အနည်းဆုံးတော့ ဘာဖြစ်နေလဲဆိုတာ ငါတို့ သိသွားပြီလေ။”

“John ရဲ့ tokenization change က change schedule (အပြောင်းအလဲစာရင်း) ထဲမှာ ရှိခဲ့လား” လို့ ကျွန်တော် မေးလိုက်တယ်။

သူက ရယ်မောခြင်း မရှိဘဲ ရယ်ပြတယ်။ “အဲဒါကိုပဲ ငါ ရှင့်ကို ပြောဖို့ ကြိုးစားနေတာ။ John က ငါတို့ရဲ့ change process ထဲကို လာခဲပါတယ်။ တကယ်တော့ လူအများစုကလည်း မလာကြပါဘူး။ ဒါက တကယ့်ကို Wild West (တောရိုင်း နယ်မြေ) လို ဖြစ်နေတာ။ ငါတို့ကတော့ လက်တည့်စမ်းပြီးပဲ လုပ်နေကြတာပေါ့။”

သူက ကာကွယ်တဲ့ လေသံနဲ့ ဆက်ပြောတယ်၊ “ငါတို့ဆီမှာ စနစ်ကျတဲ့ process တွေ ပိုလိုအပ်နေသလို အထက်ပိုင်းကလည်း IT ပိုင်းဆိုင်ရာ သင်တန်းတွေနဲ့ ထောက်ပံ့မှုတွေ ပိုပေးဖို့ လိုတယ်။ လူတိုင်းက အလုပ်တစ်ခုကို ပြီးမြောက်ဖို့ဆိုတာ လုပ်လိုက်ရုံပဲလို့ ထင်နေကြတာ။ အဲဒါက ကျွန်မရဲ့ အလုပ်ကို တကယ် ခက်ခဲစေတာပေါ့။”

ကျွန်တော့်ရဲ့ အဖွဲ့ဟောင်းမှာတုန်းကတော့ တစ်ခုခု ပြောင်းလဲမယ်ဆိုရင် တကယ်ကို စည်းကမ်းရှိခဲ့ကြပါတယ်။ တခြားလူတွေကို အသိမပေးဘဲ ဘယ်သူမှ မလုပ်ကြသလို၊ ငါတို့ရဲ့ အပြောင်းအလဲက တခြားလူကို မထိခိုက်စေအောင်လည်း အသေအချာ လုပ်ဆောင်ခဲ့ကြတာပါ။ အခုလို ဘာမှန်းမသိဘဲ မျက်စိမှိတ် လုပ်နေရတာမျိုးက ကျွန်တော့်အတွက်တော့ မရင်းနှီးပါဘူး။

“တစ်ခုခု မှားတိုင်း မှားတိုင်း အမြဲတမ်း လိုက်ပြီး လိုက်စစ်နေဖို့ အချိန် မရှိတော့ဘူး” လို့ ကျွန်တော်က စိတ်ပျက်လက်ပျက်နဲ့ ပြောလိုက်တယ်။ “ပြီးခဲ့တဲ့ သုံးရက်အတွင်းက လုပ်ခဲ့သမျှ ပြောင်းလဲမှုတွေ အားလုံးကို စာရင်းလုပ်ပေးပါ။ တိကျတဲ့ timeline မရှိရင် ဘာကြောင့် ဘာဖြစ်လဲဆိုတာ ငါတို့ သိနိုင်မှာ မဟုတ်ဘူး၊ အဲဒါဆိုရင် နောက်ထပ် ပြတ်တောက်မှုတွေ (outages) ထပ်ဖြစ်ဖို့ပဲ ရှိတော့မယ်။”

“အကြံကောင်းပဲ” လို့ သူက ခေါင်းညိတ်ပြတယ်။ “လိုအပ်ရင်တော့ IT က လူတိုင်းဆီ အီးမေးလ် ပို့ပြီး သူတို့ ဘာတွေ လုပ်ခဲ့ကြလဲဆိုတာကို လိုက်မေးရမှာပေါ့၊ ငါတို့ schedule ထဲမှာ မရှိတာတွေကို ရှာဖို့အတွက်လေ။”

“လူတိုင်းဆီ အီးမေးလ် ပို့မယ် ဟုတ်လား။ ပြောင်းလဲမှုတွေကို မှတ်တမ်းတင်တဲ့ စနစ် (system) တစ်ခုတောင် မရှိဘူးလား။ ရှင်တို့ရဲ့ ticketing system ဒါမှမဟုတ် change-authorization system ကရော အလုပ်မဖြစ်ဘူးလား” လို့ မေးလိုက်မိရင်း ကျွန်တော် အံ့သြသွားမိတယ်။ ဒါက စုံထောက်အဖွဲ့က လန်ဒန်က လူတိုင်းဆီကို အီးမေးလ် ပို့ပြီး လူသတ်မှု ဖြစ်တဲ့နေရာမှာ ဘယ်သူရှိနေလဲလို့ လိုက်မေးနေတာနဲ့ အတူတူပါပဲ။

“အိပ်မက်ကနေ နိုးလိုက်ပါဦး” လို့ သူက ကျွန်တော့်ကို အသစ်အသစ် တစ်ယောက်လိုမျိုး ကြည့်ပြီး ပြောလိုက်တယ်၊ ကျွန်တော်ကလည်း အသစ်တစ်ယောက် ဖြစ်နေတာကိုး။ “နှစ်ပေါင်းများစွာ ကျွန်မက လူတွေကို change management process တွေ၊ tools တွေကို သုံးဖို့ တိုက်တွန်းခဲ့တာပါ။ ဒါပေမဲ့ John လိုပဲ ဘယ်သူမှ မသုံးကြဘူး။ Ticketing system ကလည်း အဲဒီအတိုင်းပါပဲ၊ ဖြစ်သလိုပဲ သုံးနေကြတာ။”

အခြေအနေက ကျွန်တော် ထင်ထားတာထက် အများကြီး ပိုဆိုးနေတာပဲ။

“ကဲ… မင်း လုပ်သင့်တာကို လုပ်လိုက်ပါတော့” လို့ ကျွန်တော်က စိတ်ပျက်မှုကို မဖုံးကွယ်နိုင်ဘဲ ပြောလိုက်တယ်။ “timekeeping system ကို ထောက်ပံ့ပေးနေတဲ့ developer အားလုံးရော၊ system administrators တွေနဲ့ networking သမားတွေ အားလုံးဆီကိုပါ ပို့လိုက်ပါ။ သူတို့ရဲ့ မန်နေဂျာတွေကိုလည်း ဖုန်းဆက်ပါ၊ ဘယ်လောက်ပဲ သေးငယ်တဲ့ အပြောင်းအလဲ ဖြစ်ပါစေ၊ ငါတို့ သိဖို့ လိုအပ်တယ်လို့ ပြောလိုက်ပါ။ John ရဲ့ လူတွေကိုလည်း မမေ့နဲ့ဦး။”

Patty ခေါင်းညိတ်ပြတဲ့အခါ ကျွန်တော်က “ကြည့်ဦး၊ မင်းက change manager ပါ။ ငါတို့ အခုထက် ပိုကောင်းအောင် လုပ်မှ ရမယ်။ ငါတို့မှာ ဘာတွေ ဖြစ်ပျက်နေလဲဆိုတာကို သိနေဖို့ လိုတယ်၊ အဲဒါကြောင့် ငါတို့ဆီမှာ အလုပ်ဖြစ်တဲ့ change management process တစ်ခု လိုအပ်တယ်။ အပြင်မှာ တကယ် ဘာတွေ ဖြစ်နေလဲဆိုတာကို သိရအောင် လူတိုင်းဆီက အပြောင်းအလဲတွေကို စုစည်းပါ” လို့ ပြောလိုက်တယ်။

အံ့သြစရာကောင်းတာက Patty ရဲ့ မျက်နှာက ပျက်သွားတယ်။ “ကြည့်ပါ Bill၊ ကျွန်မ ဒါကို အရင်ကလည်း ကြိုးစားဖူးပါတယ်။ ဘာဖြစ်လာမလဲဆိုတာ ကျွန်မ ပြောပြမယ်။ CAB (Change Advisory Board) ဆိုတာကြီးက တစ်ကြိမ် ဒါမှမဟုတ် နှစ်ကြိမ်လောက်ပဲ တွေ့ကြလိမ့်မယ်။ ပတ်ပိုင်း အနည်းငယ်အတွင်းမှာပဲ လူတွေက သူတို့ အလုပ်ရှုပ်နေလို့ဆိုပြီး မတက်တော့ဘဲ နေကြလိမ့်မယ်။ ဒါမှမဟုတ် အချိန်မီ ပြီးအောင် လုပ်ရမယ်ဆိုတဲ့ အကြောင်းပြချက်နဲ့ ခွင့်ပြုချက် မစောင့်ဘဲ လုပ်ချင်ရာ လုပ်ကြလိမ့်မယ်။ ဘယ်လိုပဲဖြစ်ဖြစ် တစ်လအတွင်းမှာ အဲဒါက ပြန်ပျက်သွားလိမ့်မှာပဲ။”

“ဒီတစ်ခါတော့ မဖြစ်စေရဘူး” လို့ ကျွန်တော်က တင်းတင်းမာမာ ပြောလိုက်တယ်။ “နည်းပညာ ခေါင်းဆောင်တွေ အားလုံးဆီကို အစည်းအဝေး ဖိတ်စာ ပို့လိုက်ပါ၊ ဒါဟာ မတက်မဖြစ် အစည်းအဝေး ဖြစ်ကြောင်းလည်း အသိပေးလိုက်ပါ။ တကယ်လို့ သူတို့ကိုယ်တိုင် မတက်နိုင်ရင် ကိုယ်စားလှယ် လွှတ်ရမယ်။ နောက်ထပ် အစည်းအဝေးက ဘယ်တော့လဲ။”

“မနက်ဖြန်ပါ” လို့ သူက ပြောတယ်။

“ကောင်းပြီ” လို့ ကျွန်တော်က တကယ်ပဲ အားတက်သရောနဲ့ ပြောလိုက်တယ်။ “အဲဒီ အစည်းအဝေးကို ကျွန်တော် တကယ်ပဲ မျှော်လင့်နေပါတယ်။”

ကျွန်တော် အိမ်ပြန်ရောက်တဲ့အချိန်မှာ သန်းခေါင်ကျော်နေပြီ။ စိတ်ပျက်စရာတွေနဲ့ ပြည့်နှက်နေတဲ့ တစ်နေ့တာလုံးရဲ့ နောက်မှာ ကျွန်တော် အရမ်း ပင်ပန်းနေပါပြီ။ ကြမ်းပြင်ပေါ်မှာ မီးပုံးပျံတွေ ရှိနေသလို၊ ထမင်းစားစားပွဲပေါ်မှာလည်း ဝိုင်ပုလင်း တစ်ဝက်လောက် ကျန်နေတာကို တွေ့ရတယ်။ နံရံမှာတော့ “ဂုဏ်ယူပါတယ် ဖေဖေ!” လို့ ရေးထားတဲ့ ဆေးရောင်စုံ ပိုစတာလေး တစ်ခု ရှိနေပါတယ်။

ဒီနေ့ မွန်းလွဲပိုင်းက ကျွန်တော့် ဇနီး Paige ဆီ ဖုန်းဆက်ပြီး ရာထူးတိုးတဲ့အကြောင်း ပြောတုန်းက သူက ကျွန်တော့်ထက်တောင် ပိုပြီး ပျော်နေခဲ့တာပါ။ သူက အိမ်နီးနားချင်းတွေကို ဖိတ်ပြီး အောင်ပွဲခံဖို့တောင် စီစဉ်ခဲ့တာပေါ့။ အရမ်း နောက်ကျမှ အိမ်ပြန်ရောက်တာဖြစ်လို့ ကျွန်တော့်အတွက် လုပ်ထားတဲ့ ပါတီကိုတောင် ကျွန်တော် မမီလိုက်ပါဘူး။

ညနေ ၂ နာရီမှာတော့ Patty က ပြီးခဲ့တဲ့ သုံးရက်အတွင်းက လုပ်ခဲ့တဲ့ အပြောင်းအလဲ ၂၇ ခုထဲမှာ John ရဲ့ tokenization change နဲ့ SAN upgrade တို့ကသာ payroll failure နဲ့ ဆက်စပ်နိုင်ခြေ အရှိဆုံး ဖြစ်တယ်လို့ အကျိုးသင့်အကြောင်းသင့် ပြောပြခဲ့တယ်။

ဒါပေမဲ့ Wes နဲ့ သူ့အဖွဲ့ကတော့ SAN ကို ပြန်ကောင်းအောင် မလုပ်နိုင်သေးဘူး။

ညနေ ၃ နာရီမှာတော့ Ann နဲ့ Dick တို့ကို ကျွန်တော် သတင်းဆိုး ပေးခဲ့ရတယ်၊ အစီအစဉ် B ကို လုပ်ဖို့ကလွဲလို့ တခြား ရွေးချယ်စရာ မရှိတော့တဲ့အကြောင်းပေါ့။ သူတို့ရဲ့ စိတ်ပျက်မှုနဲ့ အားမလိုအားမရ ဖြစ်မှုတွေကို သူတို့ရဲ့ မျက်နှာမှာ အထင်းသား မြင်နေရပါတယ်။

ည ၇ နာရီကျမှသာ timekeeping application ကို ပြန်ကောင်းအောင် လုပ်နိုင်ခဲ့ပြီး၊ ည ၁၁ နာရီမှာတော့ SAN ကို နောက်ဆုံးမှာ online ပြန်ဖြစ်အောင် လုပ်နိုင်ခဲ့ပါတယ်။

VP of IT Operations အနေနဲ့ ကျွန်တော့်ရဲ့ ပထမဆုံးနေ့ဟာ သိပ်ပြီး မလှပခဲ့ပါဘူး။

အလုပ်က မဆင်းခင်မှာ Steve၊ Dick နဲ့ Ann တို့ဆီကို အခြေအနေ အကျဉ်းချုပ်ကို အီးမေးလ် ပို့ခဲ့ပြီး ဒီလိုမျိုး အမှားမျိုး နောက်ထပ် မဖြစ်စေရဘူးလို့ ကတိပေးခဲ့ပါတယ်။

ကျွန်တော် အပေါ်ထပ်ကို တက်ခဲ့တယ်၊ သွားတိုက်တာ ပြီးသွားတဲ့အခါ အိပ်ရာဝင်ခါနီးမှာ Paige နိုးမသွားအောင် ဂရုစိုက်ရင်း ဖုန်းကို နောက်ဆုံးတစ်ခေါက် စစ်ဆေးလိုက်တယ်။ ကုမ္ပဏီရဲ့ PR manager ဆီက အီးမေးလ်တစ်စောင် တွေ့လိုက်ရတဲ့အခါ ကျွန်တော် ဆဲမိသွားတယ်၊ “သတင်းဆိုးပဲ။ မနက်ဖြန် ငါတို့ သတင်းစာမျက်နှာဖုံးမှာ ပါတော့မယ် ထင်တယ်…” တဲ့။

ကျွန်တော် အိပ်ရာပေါ်မှာ ထိုင်ရင်း အဲဒီ သတင်းကို အနီးကပ် ဖတ်လိုက်မိတယ်။

*Elkhart Grove Herald Times*
*Parts Unlimited က လစာပေးခြင်းမှာ အမှားအယွင်းများ ဖြစ်ပေါ်၊ ဒေသခံ သမဂ္ဂ ခေါင်းဆောင်က ဒါဟာ ‘လက်ခံနိုင်စရာ မရှိဘူး’ ဟု ဆို*

မော်တော်ကား အစိတ်အပိုင်း ထုတ်လုပ်ရေး ကုမ္ပဏီ Parts Unlimited ဟာ သူ့ရဲ့ ဝန်ထမ်းတွေကို လစာတွေ အပြည့်အဝ မပေးနိုင်ခဲ့ဘဲ၊ အချို့ ဝန်ထမ်းတွေဆိုရင် လစာ လုံးဝ မရရှိခဲ့ကြဘူးလို့ ကုမ္ပဏီတွင်း မှတ်တမ်းတစ်ခုအရ သိရပါတယ်။ ဒေသခံ ကုမ္ပဏီတစ်ခုဖြစ်တဲ့ Parts Unlimited က ၎င်းတို့ရဲ့ နာရီစား ဝန်ထမ်းအချို့ကို လစာ မှန်ကန်စွာ မပေးနိုင်ခဲ့တာကို ဝန်ခံထားပြီး အချို့မှာမူ လုပ်ခလစာ လုံးဝ မရရှိခဲ့ကြပါဘူး။ ဒါဟာ ငွေသားစီးဆင်းမှု ပြဿနာကြောင့် မဟုတ်ဘဲ payroll system failure (လစာပေးစနစ် ကျရှုံးမှု) ကြောင့်သာ ဖြစ်တယ်လို့ ကုမ္ပဏီဘက်က ငြင်းဆိုထားပါတယ်။

ယခင်က အောင်မြင်ခဲ့တဲ့ ဒေါ်လာ ၄ ဘီလီယံတန် ကုမ္ပဏီဟာ ပြီးခဲ့တဲ့ သုံးလပတ်တွေမှာ ဝင်ငွေ လျော့နည်းလာခဲ့ပြီး အရှုံးတွေနဲ့ ရင်ဆိုင်နေရတာပါ။ ဒီလို ဘဏ္ဍာရေး အဖြစ်အပျက်တွေဟာ အထက်ပိုင်း စီမံခန့်ခွဲမှုရဲ့ အမှားတွေကြောင့် ဖြစ်တယ်လို့ အချို့က ဝေဖန်နေကြပြီး ဒေသခံ အလုပ်သမားတွေကြားမှာလည်း အလုပ်အကိုင် မလုံခြုံမှုတွေ ဖြစ်ပေါ်လာစေပါတယ်။

မှတ်တမ်းအရဆိုရင်တော့ ဒီ payroll failure ဟာ ဘာကြောင့်ပဲ ဖြစ်ဖြစ် ဝန်ထမ်းတွေအနေနဲ့ လစာ ရရှိဖို့အတွက် ရက်ပေါင်း သို့မဟုတ် ရက်သတ္တပတ်ပေါင်းများစွာ စောင့်ဆိုင်းရနိုင်ပါတယ်။

“ဒါက ကုမ္ပဏီရဲ့ စီမံခန့်ခွဲမှုပိုင်းမှာ မကြာသေးမီ နှစ်တွေအတွင်း ဖြစ်ပေါ်ခဲ့တဲ့ အမှားအယွင်းတွေထဲက နောက်ဆုံးတစ်ခုပါပဲ” လို့ Nestor Meyers ရဲ့ အဓိက လေ့လာသုံးသပ်သူ Kelly Lawrence က ဆိုပါတယ်။

Parts Unlimited ရဲ့ CFO Dick Landry ကတော့ ဒီ payroll ပြဿနာ၊ စာရင်းအင်း အမှားတွေနဲ့ စီမံခန့်ခွဲမှု စွမ်းဆောင်ရည်တွေနဲ့ ပတ်သက်ပြီး *Herald Times* ရဲ့ ဖုန်းခေါ်ဆိုမှုတွေကို ပြန်လည် ဖြေကြားခြင်း မရှိသေးပါဘူး။

Landry ကိုယ်စား ထုတ်ပြန်လိုက်တဲ့ ကြေညာချက်မှာတော့ ဒီ “glitch” (နည်းပညာ ချို့ယွင်းချက်) အတွက် စိတ်မကောင်းဖြစ်ကြောင်းနဲ့ ဒီလို အမှားမျိုး နောက်ထပ် မဖြစ်စေရဘူးလို့ ကတိပြုထားပါတယ်။

*Herald Times* အနေနဲ့ ဒီသတင်းရဲ့ နောက်ဆက်တွဲတွေကို ဆက်လက် တင်ပြသွားမှာ ဖြစ်ပါတယ်။

ကျွန်တော် အရမ်း ပင်ပန်းနေပြီဖြစ်လို့ ဘာမှ ထပ်မလုပ်နိုင်တော့ဘူး၊ မီးတွေကို ပိတ်လိုက်ရင်းနဲ့ မနက်ဖြန်ကျရင် Dick ကို လူကိုယ်တိုင် သွားတွေ့ပြီး တောင်းပန်ရမယ်လို့ စိတ်ထဲမှာ မှတ်သားထားလိုက်တယ်။ မျက်လုံးမှိတ်ပြီး အိပ်ဖို့ ကြိုးစားခဲ့ပါတယ်။

တစ်နာရီ ကြာသွားတဲ့အထိ ကျွန်တော် အမိုးကို ကြည့်ရင်းနဲ့ လုံးဝ အိပ်မပျော်နိုင်သေးပါဘူး။

အခန်း ၃၂ (CHAPTER 32)

• တနင်္လာနေ့၊ နိုဝင်ဘာ ၁၀ ရက်

နောက်ထပ် နှစ်ပတ်တာ အချိန်တွေကတော့ SWAT team ရဲ့ အလုပ်တွေ၊ Wes နဲ့ Patty တို့ရဲ့ အလုပ်တွေနဲ့တင် အမြန် ကုန်ဆုံးသွားခဲ့ပါတယ်။

ကျွန်တော် developers (ဆော့ဖ်ဝဲလ် တိုးတက်အောင် လုပ်သူများ) နဲ့ နေ့တိုင်း ထိတွေ့ အလုပ်လုပ်ခဲ့ရတာဟာ ဆယ်စုနှစ်တစ်ခုမက ကြာခဲ့ပြီလေ။ သူတို့တွေ ဘယ်လောက်အထိ ထူးဆန်းတတ်သလဲဆိုတာ ကျွန်တော် မေ့နေခဲ့တာ။ ကျွန်တော့်အမြင်မှာတော့ သူတို့ဟာ အင်ဂျင်နီယာတွေထက်စာရင် အလွတ်တန်း ဂီတသမားတွေ (indie musicians) နဲ့ ပိုတူနေသလိုပါပဲ။

ကျွန်တော့်ခေတ်တုန်းကတော့ developers တွေဟာ အင်္ကျီအိတ်မှာ ဘောပင်တွေ၊ ပေတံတွေ အပြည့်ထည့်ထားတတ်ကြတယ်—အခုလို ပုံစံဟောင်း T-shirts တွေ ဝတ်တာ၊ ဖိနပ်အပါးတွေ စီးတာမျိုး မဟုတ်ဘူး—လက်ထဲမှာလည်း စကိတ်ဘုတ်တွေအစား slide rules (တွက်ချက်ရေး ပေတံများ) ကို ကိုင်ထားကြတာလေ။

နည်းလမ်းတော်တော်များများမှာတော့ ဒီလူတွေဟာ ကျွန်တော့်ရဲ့ စရိုက်နဲ့ လုံးဝ ဆန့်ကျင်ဘက်တွေပါပဲ။ ကျွန်တော်ကတော့ လုပ်ငန်းစဉ် (processes) တွေကို ဖန်တီးပြီး လိုက်နာတဲ့သူတွေကို ကြိုက်တယ်၊ စည်းကမ်းနဲ့ စနစ်တကျ ရှိမှုကို တန်ဖိုးထားတဲ့ လူတွေကို ကြိုက်တာပေါ့။ ဒီလူတွေကတော့ ကိုယ့်စိတ်အလိုအတိုင်း လုပ်ရတာကိုပဲ သဘောကျပြီး လုပ်ငန်းစဉ်တွေကို ရှောင်ကြဉ်ချင်ကြတာ။

ဒါပေမဲ့ သူတို့ ဒီမှာ ရှိနေပေးတာကိုပဲ ကျေးဇူးတင်ရမှာပါ။

လူတစ်စုကို သူတို့ရဲ့ အလုပ်အကိုင်ပေါ် မူတည်ပြီး တစ်ပုံစံတည်း သတ်မှတ်လိုက်တာဟာ မတရားဘူးဆိုတာ ကျွန်တော် သိပါတယ်။ ငါတို့ အောင်မြင်ချင်တယ် ဆိုရင်တော့ ဒီလို မတူကွဲပြားတဲ့ အရည်အချင်းတွေ အားလုံးက အရေးကြီးတယ်ဆိုတာလည်း သိတာပေါ့။ စိန်ခေါ်မှုကတော့ ငါတို့ အားလုံးကို တစ်သားတည်း ဖြစ်အောင် ဘယ်လို စုစည်းမလဲ၊ ဒါမှပဲ ပန်းတိုင်တစ်ခုတည်းဆီကို တူတူ လျှောက်လှမ်းနိုင်မှာ မဟုတ်လား။

ပထမဆုံး စိန်ခေါ်မှုကတော့ SWAT team ပရောဂျက်ကို နာမည်ပေးဖို့ပါပဲ။ အဲဒါကို “Phoenix အငယ်စား” လို့ပဲ ဆက်ခေါ်နေလို့ မရဘူးလေ၊ အဲဒါနဲ့ နာမည်ပေးဖို့အတွက် တစ်နာရီလောက် အချိန်ယူပြီး ငြင်းခုံခဲ့ကြရသေးတယ်။

ကျွန်တော့် လူတွေကတော့ “Cujo” ဒါမှမဟုတ် “Stiletto” လို့ နာမည်ပေးချင်ကြတာ။ ဒါပေမဲ့ developers တွေကတော့ “Unicorn” (ယူနီကွန်း) လို့ ခေါ်ချင်ကြတယ်တဲ့။

ယူနီကွန်း ဟုတ်လား။ သက်တံတွေနဲ့ Care Bears (ဝက်ဝံရုပ်လေးတွေ) လိုမျိုးလား။

ကျွန်တော် ထင်ထားတာနဲ့ လုံးဝ တခြားစီပဲ၊ “Unicorn” ဆိုတဲ့ နာမည်က မဲအများဆုံးနဲ့ နိုင်သွားခဲ့ပါတယ်။

developers တွေ… သူတို့ကို ကျွန်တော် ဘယ်တော့မှ နားလည်နိုင်မှာ မဟုတ်ဘူး။

နာရီကို ကြည့်ပြီး ကျွန်တော် စိတ်ပျက်မိပေမဲ့ Project Unicorn ဟာ တကယ့်ကို အံ့သြစရာကောင်းလောက်အောင် ပုံပေါ်လာခဲ့ပါပြီ။ ဖောက်သည်တွေအတွက် ထိရောက်တဲ့ အကြံပြုချက်တွေနဲ့ အရောင်းမြှင့်တင်ရေးတွေကို ပေးနိုင်ဖို့ဆိုတဲ့ ရည်မှန်းချက်နဲ့အတူ Phoenix ဆိုတဲ့ နာမည်ကြီး စနစ်ကြီးနဲ့ လုံးဝ မသက်ဆိုင်တဲ့ သန့်ရှင်းတဲ့ code base (ကုဒ်အခြေခံ) တစ်ခုကနေ စတင်ခဲ့ကြပါတယ်။

ဒီအဖွဲ့က အတားအဆီးတွေကို ဘယ်လိုများ ကျော်လွှားသလဲဆိုတာ မြင်ရတာ အံ့သြစရာပါပဲ။ ပထမဆုံး စိန်ခေါ်မှုကတော့ ဖောက်သည်တွေရဲ့ ဝယ်ယူမှု ဒေတာတွေကို စတင် သုတေသန လုပ်ဖို့ပါပဲ၊ အဲဒါကတော့ ပထမဆုံး ရင်ဆိုင်လိုက်ရတဲ့ နံရံကြီးပေါ့။ production databases တွေကို ထိတွေ့ဖို့ ဆိုတာကတင် အဲဒီထဲက libraries တွေနဲ့ ချိတ်ဆက်ရဦးမှာဖြစ်သလို ဘယ်လို ပြောင်းလဲမှုမျိုးမဆို လုပ်ဖို့ဆိုရင်လည်း architecture team (ဗိသုကာ အဖွဲ့) ဆီကနေ အတည်ပြုချက် ရဖို့ လိုအပ်တာလေ။

အဲဒီ အတည်ပြုချက် ရတဲ့အချိန်ဆိုရင် ကုမ္ပဏီတစ်ခုလုံးတောင် ပြုတ်သွားနိုင်တာမို့ developers တွေနဲ့ Brent တို့က open source tools (အခမဲ့ ရနိုင်သော ကိရိယာများ) ကို သုံးပြီး ဒေတာဘေ့စ် အသစ်တစ်ခုကို လုံးဝ သီးသန့် တည်ဆောက်လိုက်ကြပါတော့တယ်၊ အဲဒီထဲမှာ Phoenix က ဒေတာတွေတင် မကဘဲ order entry နဲ့ inventory management systems တွေက ဒေတာတွေကိုပါ ကော်ပီ ကူးထည့်လိုက်တာပေါ့။

ဒါမျိုး လုပ်လိုက်ခြင်းအားဖြင့် Phoenix ရဲ့ အခြား အရေးကြီးတဲ့ application တွေကို မထိခိုက်စေဘဲ ငါတို့တွေက လုပ်ချင်တာကို စိတ်ကြိုက် တိုးတက်အောင် လုပ်နိုင်မယ်၊ စမ်းသပ်နိုင်မယ်၊ လည်ပတ်နိုင်မယ်လေ။ တခြား ပရောဂျက်တွေဆီကနေ ခွဲထွက်လိုက်ခြင်းအားဖြင့် သူတို့ကို အန္တရာယ် မဖြစ်စေဘဲ ငါတို့ လိုအပ်သမျှ အပြောင်းအလဲ အကုန်လုံးကို လုပ်နိုင်သွားတာပေါ့။ တစ်ချိန်တည်းမှာပဲ ငါတို့နဲ့ မဆိုင်တဲ့ လုပ်ငန်းစဉ်တွေကြားမှာလည်း ပိတ်မိမနေတော့ဘူးလေ။

ဒီ နည်းလမ်းကို ကျွန်တော် စိတ်ရင်းနဲ့ ထောက်ခံခဲ့ပါတယ်။ ဒါပေမဲ့ ကျွန်တော့်ရဲ့ စိတ်ထဲမှာတော့—ပရောဂျက် တစ်ခုစီတိုင်းက ကိုယ့်အလိုအရ ဒေတာဘေ့စ်အသစ်တွေ လျှောက်ဆောက်နေကြရင်တော့ နောက်ပိုင်းမှာ ဘယ်လိုများ စီမံကြမလဲလို့ စိုးရိမ်မိတာတော့ ရှိသားပဲ။ ဒါပေမဲ့ ကိုယ့်ကိုယ်ကိုယ် ပြန်သတိပေးလိုက်ပါတယ်—production ထဲမှာ သုံးမယ့် ဒေတာဘေ့စ် အမျိုးအစားတွေကို standardise (စံသတ်မှတ်) လုပ်ထားဖို့နဲ့ အဲဒါတွေကို ရေရှည်မှာ အထောက်အပံ့ ပေးနိုင်မယ့် အရည်အချင်းရှိတဲ့သူတွေ ရှိနေဖို့ အာရုံစိုက်ရမယ်လို့ပေါ့။

အဲဒီ အချိန်အတွင်းမှာပဲ Brent ဟာ William ရဲ့ အဖွဲ့နဲ့ ပူးပေါင်းပြီး Dev, QA နဲ့ Production environments တွေကို တပြိုင်တည်း ဖန်တီးနိုင်မယ့် build procedures (တည်ဆောက်မှု လမ်းညွှန်ချက်များ) နဲ့ automated mechanisms (အလိုအလျောက် စနစ်များ) တွေကို တည်ဆောက်ခဲ့ပါတယ်။ ၃ ပတ်ကြာ sprint (အလုပ်ကာလ) အတွင်းမှာတင်—မှတ်မိသလောက်ဆိုရင်တော့ ပထမဆုံး အကြိမ်ပဲ ဖြစ်မှာပါ—developers အားလုံးဟာ တူညီတဲ့ operating system၊ library versions၊ databases၊ database settings (သတ်မှတ်ချက်များ) တွေကို သုံးနိုင်ခဲ့ကြတာကို အားလုံး အံ့သြခဲ့ကြရပါတယ်။

“ဒါကတော့ တကယ့်ကို မယုံနိုင်စရာပဲ” လို့ developer တစ်ယောက်က sprint တစ်ခုစီရဲ့ အဆုံးမှာ လုပ်လေ့ရှိတဲ့ sprint retrospective (ပြန်လည် သုံးသပ်ခြင်း) မှာ ပြောခဲ့ပါတယ်။ “Phoenix အတွက်ဆိုရင် developer အသစ်တစ်ယောက်က သူ့စက်မှာ စနစ်တွေ အလုပ်လုပ်အောင် လုပ်ဖို့ဆိုတာ ၃ ပတ် ၄ ပတ်လောက် ကြာတာဗျ၊ ဘာလို့လဲဆိုတော့ program တစ်ခု compile လုပ်ပြီး run နိုင်ဖို့အတွက် လိုအပ်တဲ့ ပစ္စည်းပေါင်း ကုဋေကုဋာရဲ့ စာရင်းကို ငါတို့လည်း အကုန် မသိကြလို့ပဲ။ အခုတော့ Brent တို့ အဖွဲ့ တည်ဆောက်ထားတဲ့ virtual machine (ကွန်ပျူတာ အတု) ကို ယူသုံးလိုက်ရုံနဲ့ အကုန်လုံး အဆင်သင့် ဖြစ်သွားပြီလေ” တဲ့။

အလားတူပါပဲ၊ ပရောဂျက် စတဲ့ အစောပိုင်း အဆင့်မှာတင် Dev နဲ့ တူညီတဲ့ QA environment တစ်ခု ရနေတာကိုလည်း ငါတို့ အားလုံး အံ့သြခဲ့ကြရပါတယ်။ ဒါဟာလည်း အရင်က တစ်ခါမှ မရှိခဲ့ဖူးတဲ့ ကိစ္စပဲလေ။ Dev စနစ်တွေက QA ထက် memory နဲ့ storage (သိမ်းဆည်းမှု ပမာဏ) နည်းတာရယ်၊ QA က Production ထက် နည်းတာရယ်ကို ချိန်ညှိဖို့အတွက် အပြောင်းအလဲတချို့ လုပ်ဖို့တော့ လိုအပ်ပါသေးတယ်။ ဒါပေမဲ့ environments အများစုကတော့ တစ်ပုံစံတည်း ဖြစ်နေသလို မိနစ်ပိုင်းအတွင်းမှာတင် ပြောင်းလဲနိုင်၊ ဖန်တီးနိုင်တာကိုး။

အလိုအလျောက် ကုဒ်ဖြန့်ကျက်မှု (Automated code deployments) တွေကတော့ အကုန်လုံး အလုပ်မလုပ်သေးသလို environments ကြားထဲမှာ ကုဒ်တွေ ကူးပြောင်းတာကလည်း လိုနေပါသေးတယ်၊ ဒါပေမဲ့ William ရဲ့ အဖွဲ့က အဲဒီ အရည်အချင်းတွေကို လက်တွေ့ ပြသနိုင်ခဲ့တာမို့ မကြာခင်မှာ အောင်အောင်မြင်မြင် လုပ်နိုင်တော့မယ်လို့ ငါတို့ အားလုံး ယုံကြည်နေကြပါတယ်။

ဒါ့အပြင် developers တွေဟာ သူတို့ရဲ့ feature sprint goals (အင်္ဂါရပ် တိုးတက်မှု ပန်းတိုင်များ) ကို သတ်မှတ်ထားတဲ့ အချိန်ထက် စောပြီး ပြီးမြောက်ခဲ့ကြပါသေးတယ်။ သူတို့က “ဒီပစ္စည်းကို ဝယ်တဲ့သူတွေဟာ အခြား ဒီပစ္စည်းတွေကိုလည်း ဝယ်ကြတယ်” ဆိုတဲ့ reports (အစီရင်ခံစာများ) တွေကို ထုတ်ပေးနိုင်ခဲ့ကြတာပေါ့။ အဲဒီ အစီရင်ခံစာတွေက ထင်ထားတာထက် အဆပေါင်း ရာနဲ့ချီပြီး ကြာနေသေးတယ် ဆိုပေမဲ့ performance (စွမ်းဆောင်ရည်) ကို တိုးတက်အောင် လုပ်ပေးပါ့မယ်လို့ သူတို့က ကတိပေးခဲ့ပါတယ်။

ငါတို့ရဲ့ တိုးတက်မှုက မြန်ဆန်နေတာကြောင့် sprint interval ကို ၂ ပတ်အထိ လျှော့ချဖို့ ဆုံးဖြတ်လိုက်ပါတယ်။ ဒါမျိုး လုပ်လိုက်ခြင်းအားဖြင့် planning horizon (စီမံကိန်း သတ်မှတ်ချိန်) ကို လျှော့ချနိုင်ပြီး တစ်လလောက်က ချထားတဲ့ အစီအစဉ်အတိုင်း တောင့်ခံနေမယ့်အစား ဆုံးဖြတ်ချက်တွေကို ပိုပြီး ခဏခဏ ချနိုင်၊ အကောင်အထည်ဖော်နိုင်မှာ ဖြစ်ပါတယ်။

Phoenix ကတော့ လွန်ခဲ့တဲ့ ၃ နှစ်ကျော်က ရေးထားတဲ့ အစီအစဉ်အတိုင်းပဲ ဆက်သွားနေတုန်းပေါ့။

ဒါကိုတော့ ကျွန်တော် သိပ်ပြီး မစဉ်းစားမိအောင် ကြိုးစားနေရပါတယ်။

ငါတို့ရဲ့ တိုးတက်မှုနှုန်းကတော့ တကယ့်ကို အဆမတန် မြန်လာခဲ့ပါပြီ။ အရင်ကထက် ပိုမြန်မြန် စီစဉ်နိုင်၊ အကောင်အထည်ဖော်နိုင်လာသလို Unicorn နဲ့ Phoenix ကြားက velocity gap (အရှိန်အဟုန် ကွာဟချက်) ကလည်း ပိုပြီး ကြီးမားလာခဲ့ပါတယ်။ Phoenix အဖွဲ့တွေကလည်း ဒါကို သတိထားမိလာသလို ငါတို့ရဲ့ နည်းလမ်းတွေကို လိုက်ပြီး အတုယူလာကြတဲ့အတွက် မဖြစ်နိုင်ဘူးလို့ ထင်ထားတဲ့ ရလဒ်တွေ ထွက်လာတာ တွေ့ရပါတယ်။

Unicorn ရဲ့ အရှိန်အဟုန် (momentum) ကတော့ ဘယ်သူမှ တားလို့ မရတော့ဘဲ သူ့အလိုလို လည်ပတ်နေပါပြီ။ ငါတို့ ချင်ရင်တောင်မှ သူတို့ကို ရပ်ခိုင်းပြီး နည်းလမ်းဟောင်းအတိုင်း ပြန်လုပ်ခိုင်းဖို့ဆိုတာ မဖြစ်နိုင်သလောက်ပါပဲ။

ဘတ်ဂျက် အစည်းအဝေးတစ်ခု လုပ်နေတုန်းမှာ Wes ဆီက ဖုန်းလာပါတယ်— “ငါတို့ ပြဿနာ အကြီးကြီး တက်နေပြီဗျ။”

ကျွန်တော် အခန်းပြင် ထွက်လိုက်ပြီး— “ဘာဖြစ်လို့လဲ” လို့ မေးလိုက်တာပေါ့။

“Brent ကို ရှာမတွေ့တာ ပြီးခဲ့တဲ့ နှစ်ရက်တောင် ရှိနေပြီဗျ။ မင်းရော သူ ဘယ်မှာလဲ သိလား” တဲ့။

“မသိဘူးလေ… နေဦး… ရှာမတွေ့ဘူး ဆိုတာ ဘာကို ဆိုလိုတာလဲ။ သူ နေကောင်းရဲ့လား။ သူ့ဖုန်းဆီ ဆက်ကြည့်ပြီလား” ဆိုတော့—

Wes က သူ့ရဲ့ စိတ်ပျက်မှုကို မဖုံးကွယ်နိုင်တော့ပါဘူး— “ဆက်တာပေါ့ဗျာ! နာရီတိုင်း သူ့ဆီ voicemail တွေ ထားခဲ့တာလေ။ လူတိုင်း သူ့ကို လိုက်ရှာနေတာ။ အလုပ်တွေကလည်း ပုံနေပြီ၊ သူ့အဖွဲ့သားတွေလည်း တော်တော်လေး စိတ်ပူနေကြပြီ—ဟာ… ဘုရားရေ… Brent ဆီက ဖုန်းလာနေပြီ… ခဏနော်…”

သူက စားပွဲတင်ဖုန်းကို ကိုင်လိုက်တာကို ကျွန်တော် ကြားနေရတယ်— “မင်း ဘယ်တွေ ရောက်နေတာလဲကွ။ လူတိုင်း မင်းကို လိုက်ရှာနေတာ! ဘာ… ဘာ… Des Moines (ဒီမွိုင်း) ဟုတ်လား။ မင်း အဲဒီမှာ ဘာလုပ်နေတာလဲ။ ဘယ်သူမှ ငါ့ကို မပြောပါလား… Dick နဲ့ Sarah တို့ရဲ့ လျှို့ဝှက် တာဝန် ဟုတ်လား? ဘာတွေလဲ—”

Wes တစ်ယောက် Brent ဆီကနေ အခြေအနေကို သိအောင် မနည်း မေးနေတာကို ကျွန်တော် ခဏလောက် ပြုံးပြီး နားထောင်နေမိပါတယ်။ နောက်ဆုံးတော့ သူက— “ခဏလေး… Bill ဘာလုပ်ချင်သလဲဆိုတာ ငါ မေးကြည့်လိုက်ဦးမယ်…” လို့ ပြောပြီး သူ့ဖုန်းကို ပြန်ကိုင်လိုက်ပါတယ်။

“အိုကေ… မင်းလည်း အကုန် ကြားလိုက်တယ် မဟုတ်လား” လို့ သူက ကျွန်တော့်ကို ပြောပါတယ်။

“ငါ အခုချက်ချင်း သူ့ဆီ ဖုန်းဆက်လိုက်မယ်လို့ ပြောလိုက်ပါ” လို့ ကျွန်တော်က ဆိုပါတယ်။

ဖုန်းချလိုက်ပြီး Brent ဆီ ဆက်လိုက်ရင်း Sarah ကတော့ နောက်တစ်ခါ ဘာတွေ လျှောက်လုပ်ပြန်ပြီလဲလို့ တွေးမိတာပေါ့။

“ဟယ်လို Bill” လို့ သူက နှုတ်ဆက်ပါတယ်။

“မင်း ဘာတွေ ဖြစ်နေတာလဲဆိုတာနဲ့ ဘာဖြစ်လို့ Des Moines ကို ရောက်နေရတာလဲဆိုတာ ပြောပြလို့ ရမလား” လို့ ကျွန်တော်က ယဉ်ယဉ်ကျေးကျေး မေးလိုက်ပါတယ်။

“Dick ရဲ့ ရုံးခန်းက ဘယ်သူမှ ဆရာ့ကို မပြောဘူးလား” လို့ သူက မေးပါတယ်။ ကျွန်တော် ဘာမှ ပြန်မပြောတော့ သူက ဆက်ပြောတယ်— “Dick နဲ့ ဘဏ္ဍာရေး အဖွဲ့က ကျွန်တော့်ကို မနေ့မနက်ကပဲ ကုမ္ပဏီကို ခွဲထုတ်ပစ်ဖို့ (splitting up the company) အစီအစဉ်ဆွဲတဲ့ task force (အထူးလုပ်ငန်းအဖွဲ့) မှာ ပါဖို့ အတင်း ခေါ်ထုတ်သွားတာလေ။ ကြည့်ရတာ ဒါဟာ အရေးကြီးဆုံး ပရောဂျက် (top priority project) ဖြစ်ပုံရတယ်၊ အဲဒီမှာ IT စနစ်တွေ အားလုံးက ဘယ်လို ထိခိုက်မလဲဆိုတာ သူတို့ သိချင်နေကြတာဗျ” တဲ့။

“ဒါနဲ့ Dick က ဘာလို့ မင်းကိုမှ အဖွဲ့ထဲ ထည့်တာလဲ” လို့ ကျွန်တော် မေးလိုက်ပါတယ်။

“ကျွန်တော်လည်း မသိဘူးလေ” လို့ သူက ပြန်ဖြေတယ်— “ယုံပါ ဆရာ… ကျွန်တော်လည်း ဒီမှာ မနေချင်ပါဘူး။ လေယာဉ် ပျံရတာကို ကျွန်တော် မုန်းတယ်။ သူတို့ရဲ့ business analysts (စီးပွားရေး သုတေသီ) တစ်ယောက်ယောက်ကိုပဲ ခိုင်းသင့်တာပေါ့၊ ဒါပေမဲ့ စနစ်ကြီးတွေ တစ်ခုနဲ့တစ်ခု ဘယ်လို ချိတ်ဆက်နေလဲ၊ ဘယ်နေရာမှာ ရှိသလဲ၊ ဘယ် ဝန်ဆောင်မှုတွေအပေါ် မှီခိုနေသလဲ ဆိုတာကို ကျွန်တော်က အသိဆုံး ဖြစ်နေလို့ ထင်တာပဲ… စကားမစပ် ဆရာ့ကို အခုပဲ ပြောနိုင်တာ တစ်ခုကတော့—ဒီကုမ္ပဏီကို ခွဲထုတ်ပစ်ဖို့ ဆိုတာဟာ တကယ့်ကို အိပ်မက်ဆိုးကြီး ဖြစ်မှာပါပဲ” တဲ့။

ကျွန်တော်တို့ လက်လီ ကုမ္ပဏီကြီးကို ဝယ်တုန်းက ကုမ္ပဏီ ပေါင်းစည်းရေး အဖွဲ့ကို ကျွန်တော် ဦးဆောင်ခဲ့တာ သတိရသွားမိတယ်။ အဲဒါက တကယ့်ကို ပရောဂျက် အကြီးကြီးလေ။ ကုမ္ပဏီကို ပြန်ပြီး ခွဲထုတ်ဖို့ ဆိုတာကတော့ ဒီထက်တောင် ပိုခက်ခဲနိုင်ပါတယ်။

အကယ်၍ ဒါဟာ ငါတို့ ထောက်ပံ့နေတဲ့ အပလီကေးရှင်း ရာနဲ့ချီကို ထိခိုက်မယ် ဆိုရင်တော့ Brent ပြောတာ မှန်မှာပါ။ ဒါက နှစ်နဲ့ချီပြီး ကြာဦးမှာလေ။

IT ဆိုတာ နေရာတိုင်းမှာ ရှိနေတာမို့လို့ ဒါဟာ ကိုယ်လက်အင်္ဂါ တစ်ခုကို ဖြတ်ထုတ်ရသလို မဟုတ်ဘူး။ ကုမ္ပဏီရဲ့ အာရုံကြော စနစ်ကြီး တစ်ခုလုံးကို ခွဲထုတ်ပစ်ရသလိုမျိုး ဖြစ်နေတာကိုး။

Dick နဲ့ Sarah တို့ဟာ ကျွန်တော့်ရဲ့ အဓိက အဖွဲ့ဝင် တစ်ယောက်ကို ကျွန်တော့်ကို တစ်ချက်တောင် မမေးဘဲ အတင်း ဆွဲထုတ်သွားကြတာကို သတိရလိုက်တော့ ကျွန်တော်က ပြတ်ပြတ်သားသားပဲ ပြောလိုက်ပါတော့တယ်— “Brent… သေသေချာချာ နားထောင်နော်—မင်းရဲ့ အရေးကြီးဆုံး အလုပ်ကတော့ Unicorn အဖွဲ့သားတွေ ဘာလိုအပ်သလဲဆိုတာကို ရှာဖွေပြီး ပေးဖို့ပဲ။ လိုအပ်ရင် မင်းရဲ့ လေယာဉ်ကိုတောင် လွတ်ခိုင်းလိုက်။ ငါ ဖုန်းဆက်လိုက်မယ်၊ ငါ့အတွင်းရေးမှူး Ellen က မင်းအတွက် ဒီည အိမ်ပြန်မယ့် လေယာဉ်ကို ဘွတ်ကင်လုပ်ပေးဖို့ အခွင့်အရေး ရှိတယ်နော်။ နားလည်လား” ဆိုတော့—

“ဆရာက ကျွန်တော့်ကို တမင်တကာ လေယာဉ် လွတ်ခိုင်းစေချင်တာလား” လို့ သူက မေးပါတယ်။

“ဟုတ်တယ်” တဲ့။

“ဒါဆိုရင် Dick နဲ့ Sarah ကို ကျွန်တော် ဘာပြောရမလဲ” လို့ သူက ဇဝေဇဝါနဲ့ မေးပါတယ်။

ကျွန်တော် ခဏစဉ်းစားပြီး— “Bill က အရေးပေါ် ဖုန်းခေါ်ဆိုမှု တစ်ခုအတွက် လိုအပ်နေလို့ပါ၊ ပြီးမှ ပြန်လာတွေ့ပါ့မယ်လို့ပဲ ပြောလိုက်” လို့ ဆိုပါတယ်။

“အိုကေလေ… ဒါနဲ့ ဘာတွေ ဖြစ်နေတာလဲ” ဆိုတော့—

“ရှင်းပါတယ် Brent ရာ” လို့ ကျွန်တော် ရှင်းပြလိုက်တယ်— “Unicorn ဟာ ငါတို့ရဲ့ သုံးလပတ် ကိန်းဂဏန်းတွေ ပြည့်မီဖို့ နောက်ဆုံး မျှော်လင့်ချက်ပဲလေ။ နောက်ထပ် သုံးလပတ်တစ်ခုသာ ထပ်ပြီး လွဲချော်သွားမယ် ဆိုရင်တော့ ဘုတ်အဖွဲ့က ကုမ္ပဏီကို တကယ်ပဲ ခွဲထုတ်ပစ်တော့မှာ၊ အဲဒီကျရင်တော့ မင်း အဲဒီ အဖွဲ့ကို ကူညီလို့ ရတာပေါ့။ ဒါပေမဲ့ အခု ငါတို့ ကိန်းဂဏန်းတွေ ပြည့်မီခဲ့မယ် ဆိုရင်တော့ ကုမ္ပဏီကို မကွဲအောင် ထိန်းထားနိုင်မယ့် အခွင့်အရေး ရှိသေးတယ်။ အဲဒါကြောင့် Unicorn ဟာ ငါတို့ရဲ့ အမြင်ဆုံး ဦးစားပေး (highest priority) ဖြစ်နေတာလေ။ Steve ကလည်း ဒါကို ရှင်းရှင်းလင်းလင်း ပြောထားပြီးသား” တဲ့။

Brent ကတော့ မယုံကြည်သလိုနဲ့— “ဟုတ်ကဲ့ပါ… ဆရာ ဘယ်သွားခိုင်းခိုင်း ကျွန်တော် သွားပါ့မယ်။ အဲဒီ လူကြီးပိုင်းတွေနဲ့ ငြင်းခုံရမှာကတော့ ဆရာ့ တာဝန်ပေါ့” ဆိုပါတယ်။ သူကတော့ ခေါင်းဆောင်တွေဆီက ရနေတဲ့ ရှုပ်ထွေးနေတဲ့ လမ်းညွှန်ချက်တွေကြောင့် တော်တော်လေး စိတ်ညစ်နေပုံ ရပါတယ်။

ဒါပေမဲ့ ကျွန်တော်ကတော့ သူ့ထက်တောင် ပိုပြီး စိတ်တိုနေမိတာပါ။

ကျွန်တော် Steve ရဲ့ အတွင်းရေးမှူး Stacy ဆီ ဖုန်းဆက်ပြီး ကျွန်တော် လာနေတဲ့အကြောင်း ပြောခိုင်းလိုက်ပါတယ်။

Steve ကို ရှာဖို့ Building 2 ဆီ လျှောက်သွားရင်း Wes ဆီ ဖုန်းဆက်လိုက်ပါတယ်။

“မင်း ဘာလုပ်လိုက်တယ် ဟုတ်လား” လို့ သူက တဟားဟား ရယ်ရင်း ပြောတယ်- “ကောင်းတယ်ဗျာ။ အခုတော့ မင်းဟာ တစ်ဖက်မှာ Steve၊ နောက်တစ်ဖက်မှာ Dick နဲ့ Sarah တို့ ရှိနေတဲ့ နိုင်ငံရေး စစ်တလင်း အလယ်ကောင်ကို ရောက်သွားပြီ။ ပြောရရင်တော့ မင်း ရွေးလိုက်တဲ့ ဘက်က အနိုင်ရမယ့် ဘက်လားဆိုတာ ငါလည်း မသေချာဘူးဗျ” တဲ့။

ခဏနေတော့ သူက— “Steve က ငါတို့ကို တကယ်ပဲ ထောက်ခံပေးမယ်လို့ မင်း ထင်လား” လို့ မေးပါတယ်။

ကျွန်တော် သက်ပြင်းတစ်ချက် ချလိုက်မိတယ်- “မျှော်လင့်တာပဲလေ။ အကယ်၍ Brent သာ အချိန်ပြည့် ပြန်မလာနိုင်ဘူး ဆိုရင်တော့ Unicorn ကတော့ သွားပြီပဲ။ အဲဒီလို ဖြစ်သွားရင်တော့ ငါတို့မှာ CEO အသစ် ရလာမယ်၊ အလုပ်တွေ အကုန် အပြင်ကို အပ်ခံရမယ် (outsourced)၊ ပြီးတော့ ကုမ္ပဏီကို ဘယ်လို ခွဲမလဲဆိုတာကိုလည်း စဉ်းစားကြရတော့မှာပေါ့။ အဲဒါက မင်းအတွက်တော့ ပျော်စရာကောင်းတဲ့ အလုပ်တစ်ခု ဖြစ်နေမလား” ဆိုတော့—

ဖုန်းချလိုက်ပြီး Steve ရဲ့ ရုံးခန်းထဲကို လှမ်းဝင်လိုက်ပါတယ်။ သူက ဖျော့တော့တော့ အပြုံးနဲ့— “မင်္ဂလာပါ Bill။ Stacy ပြောတာတော့ မင်းမှာ ငါ့အတွက် သတင်းဆိုးတွေ ရှိတယ်ဆို…” တဲ့။

Brent နဲ့ ဖုန်းပြောတုန်းက သိခဲ့ရတာတွေကို ကျွန်တော် ပြောပြလိုက်တဲ့ အခါမှာတော့ သူ့မျက်နှာ တစ်ခုလုံး နီရဲသွားတာကို တွေ့လိုက်ရလို့ ကျွန်တော် အံ့သြသွားမိတယ်။ သူက CEO တစ်ယောက်ဆိုတော့ ဒါတွေကို သိပြီးသား ဖြစ်မယ်လို့ ကျွန်တော် ထင်ခဲ့တာလေ။

ဒါပေမဲ့ သူလည်း မသိခဲ့ပါဘူး။

ခဏလောက် တိတ်သွားပြီးမှ သူက— “ဒီ သုံးလပတ်ရဲ့ ရလဒ် ထွက်လာတဲ့အထိ ကုမ္ပဏီကို ခွဲထုတ်မယ့် ကိစ္စကို ရှေ့မဆက်ပါဘူးလို့ ဘုတ်အဖွဲ့က ငါ့ကို ကတိပေးခဲ့တာပဲ။ ကြည့်ရတာ သူတို့ စိတ်မရှည်နိုင်တော့ဘူး ထင်တယ်” လို့ ဆိုပါတယ်။

သူက ဆက်ပြောတယ်— “ဒါဆို… Brent သာ တခြား အလုပ် ရောက်သွားမယ် ဆိုရင် Unicorn အပေါ်မှာ ဘယ်လို သက်ရောက်မှု ရှိမလဲဆိုတာ ပြောပြပါဦး” တဲ့။

“ကျွန်တော် Chris, Wes နဲ့ Patty တို့နဲ့ စကားပြောပြီးပါပြီ” လို့ ပြန်ဖြေလိုက်တယ်— “Project Unicorn ကတော့ လုံးဝ သွားပြီပဲဗျ။ ကျွန်တော်ကတော့ ပင်ကိုယ်ကတည်းက သံသယ ကြီးတဲ့သူပါ၊ ဒါပေမဲ့ Unicorn ကတော့ တကယ်ကို အလုပ်ဖြစ်လိမ့်မယ်လို့ ကျွန်တော် ယုံကြည်တယ်။ နောက်ထပ် နှစ်ပတ်ဆိုရင် Thanksgiving (ကျေးဇူးတော်နေ့) ရောက်တော့မှာ၊ Unicorn အတွက် လိုအပ်တဲ့ စွမ်းဆောင်ရည်တွေကို တည်ဆောက်တဲ့ နေရာမှာ Brent က အဓိက တာဝန်ယူထားတာလေ။ စကားမစပ်… ငါတို့ ရနေတဲ့ ထိုးဖောက်မှု အသစ်တွေကို Phoenix အဖွဲ့ကလည်း လိုက်ပြီး အတုယူနေကြပြီ၊ ဒါဟာ တကယ့်ကို ကောင်းတဲ့ ကိစ္စပဲလေ” တဲ့။

ကျွန်တော့်ရဲ့ အချက်ကို ပိုပြီး လေးနက်သွားအောင် ကျွန်တော်က ပြတ်ပြတ်သားသားပဲ— “Brent မရှိဘဲနဲ့တော့ Unicorn နဲ့ ပတ်သက်ပြီး ကျွန်တော်တို့ ချမှတ်ထားတဲ့ အရောင်းနဲ့ အမြတ် ရည်မှန်းချက်တွေကို ဘယ်လိုမှ ပြည့်မီမှာ မဟုတ်ပါဘူး။ လုံးဝ မဖြစ်နိုင်ဘူးဗျ” ဆိုတော့—

Steve က နှုတ်ခမ်းကို စေ့လိုက်ပြီး မေးတယ်— “အကယ်၍ Brent ရဲ့ နေရာမှာ မင်းရဲ့ နောက်ထပ် အတော်ဆုံး လူတစ်ယောက်နဲ့ အစားထိုးလိုက်မယ် ဆိုရင်ကော (backfill) ဘာဖြစ်မလဲ” တဲ့။

Wes ပြောခဲ့တာကိုပဲ ကျွန်တော် ပြန်ပြောပြလိုက်ပါတယ်။ “Brent ကတော့ တကယ့်ကို ထူးခြားတဲ့သူ တစ်ယောက်ပါဗျ။ Unicorn မှာဆိုရင် developers တွေရဲ့ လေးစားမှုကို ရထားတဲ့သူ၊ ငါတို့မှာ ရှိတဲ့ IT infrastructure တွေ အကုန်လုံးနီးပါးကို နှံ့နှံ့စပ်စပ် သိတဲ့သူ၊ ပြီးတော့ developers တွေ ဘာတွေ တည်ဆောက်ဖို့ လိုသလဲဆိုတာကို production ထဲမှာ လည်ပတ်နိုင်အောင် ရှင်းပြနိုင်တဲ့သူ လိုအပ်နေတာပါ။ အဲဒီ အရည်အချင်းတွေက ရှားပါးသလို အခု ချက်ချင်း အစားထိုးနိုင်မယ့်သူလည်း ငါတို့ဆီမှာ မရှိသေးဘူးဗျ” တဲ့။

“ဒါဆိုရင် Dick ရဲ့ အဖွဲ့ထဲကို မင်းရဲ့ နောက်ထပ် အတော်ဆုံး လူတစ်ယောက်ကို တာဝန်ပေးလိုက်ရင်ကော” လို့ သူက မေးပါတယ်။

“ကျွန်တော် ထင်တာကတော့—ကုမ္ပဏီ ခွဲထုတ်မယ့် အစီအစဉ်က သိပ်ပြီး မတိကျတာမျိုးတော့ ဖြစ်နိုင်ပေမဲ့ ပြီးသွားမှာတော့ သေချာပါတယ်” လို့ ပြန်ဖြေလိုက်ပါတယ်။

Steve က ကုလားထိုင်ပေါ်ကို မှီလိုက်ရင်း ဘာမှ မပြောဘဲ ခေတ္တ တိတ်နေပါတယ်။

နောက်ဆုံးမှာတော့ သူက— “Brent ကို ဒီကို ပြန်ခေါ်လိုက်ပါ။ ကျန်တာတွေကို ငါ ကိုင်တွယ်လိုက်မယ်” လို့ ပြတ်ပြတ်သားသား ဆုံးဖြတ်လိုက်ပါတော့တယ်။

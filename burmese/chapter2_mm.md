အခန်း ၂ (CHAPTER 2)

• အင်္ဂါနေ့၊ စက်တင်ဘာ ၂ ရက်

“အထဲမှာ ဘယ်လိုနေလဲ” Stacy က သူ့ကီးဘုတ်ကနေ မော့ကြည့်ရင်း နွေးနွေးထွေးထွေး မေးလိုက်တယ်။

ကျွန်တော် ခေါင်းယမ်းလိုက်မိတယ်။ “မယုံနိုင်စရာပဲ။ ကျွန်တော် မလိုချင်တဲ့ အလုပ်သစ်ကို လက်ခံဖို့ သူ ကျွန်တော့်ကို ပြောဆို စည်းရုံးသွားတယ်။ အဲဒါ ဘယ်လိုဖြစ်သွားတာလဲ။”

“သူက တကယ်ကို စည်းရုံးနိုင်စွမ်း ကောင်းတာလေ” လို့ သူက ပြောတယ်။ “သူကတော့ တစ်ယောက်ထဲ ရှိတဲ့လူမျိုးပါ။ ကျွန်မ သူနဲ့ အလုပ်လုပ်လာတာ ဆယ်နှစ်နီးပါး ရှိပြီ၊ သူ ဘယ်ကိုသွားသွား ကျွန်မ လိုက်မှာပဲ။ ရှင့်အလုပ် ပိုအဆင်ပြေအောင် ကျွန်မ ဘာများ ကူညီပေးရမလဲ။”

ခဏလောက် စဉ်းစားပြီးမှ ကျွန်တော်က “အရေးကြီးတဲ့ payroll (လစာပေးခြင်း) ပြဿနာတစ်ခုကို အမြန်ဖြေရှင်းဖို့ လိုနေတယ်။ Dick Landry က တတိယထပ်မှာ ရှိတာ မဟုတ်လား” လို့ မေးလိုက်တယ်။

ကျွန်တော် မေးလို့တောင် မဆုံးသေးဘူး၊ သူက Dick ရဲ့ ဆက်သွယ်ရမယ့် အချက်အလက်တွေ အားလုံးပါတဲ့ Post-it note လေးတစ်ခုကို ကမ်းပေးလိုက်တယ်။ ရုံးခန်းတည်နေရာ၊ ဖုန်းနံပါတ်တွေ အားလုံး ပါတာပေါ့။

ကျေးဇူးတင်လွန်းလို့ ကျွန်တော် သူ့ကို ပြုံးပြလိုက်တယ်။ “တကယ် ကျေးဇူးတင်ပါတယ်—ရှင်က တကယ့်ကို အံ့သြစရာကောင်းတဲ့လူပဲ။”

ဓာတ်လှေကားဆီ သွားတဲ့လမ်းမှာ Dick ရဲ့ ဖုန်းကို ကျွန်တော် ခေါ်လိုက်တယ်။ “Dick ပြောနေတယ်” လို့ သူက ခပ်ပြတ်ပြတ်ပဲ နှုတ်ဆက်ပြီး နောက်ကွယ်မှာတော့ ကီးဘုတ်ရိုက်သံတွေ ကြားနေရတယ်။

“ကျွန်တော် Bill Palmer ပါ။ Steve က ကျွန်တော့်ကို VP of IT Operations အသစ်အဖြစ် ခုလေးတင် ခန့်လိုက်လို့ပါ၊ သူက ကျွန်တော့်ကို—”

“ဂုဏ်ယူပါတယ်” လို့ သူက ကြားဖြတ်ပြောလိုက်တယ်။ “အခု ကြည့်ဦး၊ ငါ့လူတွေက payroll မှာ ကြီးမားတဲ့ ပုံမှန်မဟုတ်တဲ့ အခြေအနေတစ်ခု ရှာတွေ့ထားတယ်။ မင်း ငါ့ရုံးခန်းကို ဘယ်အချိန် ရောက်နိုင်မလဲ။”

“အခုချက်ချင်းပဲ လာခဲ့ပါ့မယ်” လို့ ကျွန်တော် ပြန်ဖြေလိုက်တယ်။ သူ ဖုန်းချသွားတဲ့ အသံကို ကြားလိုက်ရတယ်။ ဒီထက် ပိုနွေးထွေးတဲ့ ကြိုဆိုမှုကိုတော့ ကျွန်တော် ရခဲ့ဖူးပါတယ်။

တတိယထပ်မှာ Finance (ဘဏ္ဍာရေး) နဲ့ Accounting (စာရင်းကိုင်) ဌာနတွေကြားထဲ ကျွန်တော် လမ်းလျှောက်လာရင်း စနစ်တကျ ဝတ်စားထားတဲ့ လူတွေကြားထဲ ရောက်နေမိတယ်။ Dick က သူ့စားပွဲမှာ ဖုန်းပြောနေတုန်းပါပဲ။ ကျွန်တော့်ကို မြင်တဲ့အခါမှာတော့ သူက ဖုန်းကို လက်နဲ့အုပ်လိုက်ပြီး—

“မင်း IT ကလား” လို့ ခပ်ပြတ်ပြတ် မေးတယ်။

ကျွန်တော် ခေါင်းညိတ်ပြလိုက်တဲ့အခါ သူက ဖုန်းထဲကို “ခဏနေဦး၊ ငါ သွားရတော့မယ်။ ကူညီပေးမယ့်လူ ရောက်လာပြီ။ နောက်မှ ပြန်ဆက်မယ်” လို့ ပြောပြီး တစ်ဖက်က ပြန်ပြောတာကိုတောင် မစောင့်ဘဲ ဖုန်းချလိုက်တယ်။

လူတွေကို အဲဒီလို ခဏခဏ ဖုန်းချတတ်တဲ့လူမျိုး ကျွန်တော် တစ်ခါမှ မတွေ့ဖူးဘူး။ “တစ်ယောက်နဲ့တစ်ယောက် ရင်းနှီးအောင် လုပ်ကြရအောင်” ဆိုတဲ့ စကားမျိုး မပါဘဲ တိုတိုတုတ်တုတ်ပဲ ဖြစ်မယ့် စကားပြောဆိုမှုအတွက် ကိုယ့်ကိုယ်ကိုယ် ပြင်ဆင်လိုက်ရတယ်။

ဓားစာခံ အခြေအနေတစ်ခုလိုမျိုးပဲ ကျွန်တော် လက်နှစ်ဖက်ကို ဖြည်းဖြည်းချင်း မြှောက်ပြပြီး ပရင့်ထုတ်ထားတဲ့ အီးမေးလ်ကို Dick ကို ပြလိုက်တယ်။ “Steve က ကျွန်တော့်ကို payroll outage (စနစ်ပြတ်တောက်မှု) အကြောင်း ပြောလိုက်တယ်။ ဒီအခြေအနေကို အကောင်းဆုံး သိရအောင် ကျွန်တော် ဘယ်က စရမလဲ။”

“ငါတို့ တကယ် ဒုက္ခရောက်နေပြီ” လို့ Dick က ပြန်ဖြေတယ်။ “မနေ့က payroll run တဲ့အခါမှာ hourly employees (နာရီအလိုက် အလုပ်လုပ်တဲ့ ဝန်ထမ်းတွေ) ရဲ့ မှတ်တမ်းတွေ အားလုံး ပျောက်သွားတယ်။ ဒါ IT ပြဿနာပဲလို့ ငါတို့ သေချာပေါက် ပြောနိုင်တယ်။ ဒီအမှားကြောင့် ဝန်ထမ်းတွေကို လစာ မပေးနိုင်တော့ဘူး၊ ဒါက အလုပ်သမား ဥပဒေတွေကို ချိုးဖောက်သလို ဖြစ်နေပြီး သမဂ္ဂကလည်း တကယ်ကို အော်ဟစ်ကြတော့မှာပဲ။”

သူက ခဏလောက် တိုးတိုးလေး ညည်းတွားနေတယ်။ “လာ၊ ငါ့ရဲ့ Operations Manager ဖြစ်တဲ့ Ann ဆီ သွားရအောင်။ သူက မနေ့က ညနေကတည်းက တကယ်ကို ခေါင်းစားနေတာ။”

သူ့နောက်ကို မီအောင် အမြန်လိုက်သွားရင်း သူ ရုတ်တရက် ရပ်လိုက်ပြီး အစည်းအဝေးခန်း ပြတင်းပေါက်ကနေ ချောင်းကြည့်တဲ့အခါ ကျွန်တော် သူ့ကို ဝင်တိုက်မိမလို ဖြစ်သွားတယ်။ သူက တံခါးကို ဖွင့်လိုက်ပြီး— “Ann၊ အခြေအနေ ဘယ်လိုလဲ” လို့ မေးတယ်။

အစည်းအဝေးခန်းထဲမှာ သေသေသပ်သပ် ဝတ်စားထားတဲ့ အမျိုးသမီး နှစ်ယောက် ရှိနေပါတယ်။ တစ်ယောက်က အသက် ၄၅ နှစ် ဝန်းကျင်လောက်ရှိပြီး whiteboard ပေါ်က flowcharts (လုပ်ငန်းစဉ်ပြ ဇယား) တွေနဲ့ ကိန်းဂဏန်းတွေကို လေ့လာနေတယ်၊ နောက်တစ်ယောက်ကတော့ အသက် ၃၀ ဝန်းကျင်လောက်ရှိပြီး laptop ပေါ်မှာ စာရိုက်နေပါတယ်။

စားပွဲကြီးပေါ်မှာ spreadsheets (ဇယားကွက်) တွေ ပြန့်ကျဲနေပြီး အသက်ကြီးတဲ့ အမျိုးသမီးက ဖြစ်နိုင်ခြေရှိတဲ့ အမှားအယွင်း အကြောင်းအရင်းတွေကို အမှတ်အသားနဲ့ ညွှန်ပြနေပါတယ်။

သူတို့ ဝတ်စားထားပုံနဲ့ သူတို့ရဲ့ စိုးရိမ်တကြီး ဖြစ်နေတဲ့ မျက်နှာအမူအရာတွေက ဒေသခံ စာရင်းကိုင် လုပ်ငန်းတစ်ခုကနေ ခေါ်ယူထားတဲ့သူတွေနဲ့ တူနေပါတယ်။ စာရင်းစစ်ဟောင်းတွေပေါ့။ သူတို့ကို ကျွန်တော်တို့ဘက်မှာ ရှိနေတာတော့ ကောင်းပါတယ်လို့ ကျွန်တော် တွေးမိတယ်။

Ann က အရမ်းကို ပင်ပန်းနွမ်းနယ်နေတဲ့ ပုံစံနဲ့ ခေါင်းယမ်းပြတယ်။ “သိပ်ပြီး တိုးတက်မှု မရှိသေးဘူး။ ဒါက upstream timekeeping systems (အချိန်မှတ်တဲ့ စနစ်) ထဲက တစ်ခုမှာ ဖြစ်တဲ့ IT system fail ဖြစ်တာလို့ ကျွန်မတို့ သေချာသလောက် ရှိနေပါပြီ။ နောက်ဆုံး upload လုပ်တဲ့ အချိန်မှာ factory worker records တွေ အားလုံး မှားယွင်းသွားတာ—”

Dick က သူ့စကားကို ကြားဖြတ်ပြောလိုက်တယ်။ “ဒါ IT က Bill ပဲ။ သူက ဒီပြဿနာကို ပြီးအောင် ဖြေရှင်းဖို့ ဒါမှမဟုတ် ကျရှုံးဖို့ တာဝန်ယူလိုက်တဲ့လူလို့ သူကိုယ်တိုင် ပြောခဲ့တယ်လေ။”

ကျွန်တော်က “မင်္ဂလာပါခင်ဗျာ။ ကျွန်တော် အခု IT Operations ရဲ့ အကြီးအကဲသစ် ဖြစ်လာတဲ့ Bill ပါ။ ဒီပြဿနာနဲ့ ပတ်သက်ပြီး သိသမျှကို အစကနေ ရှင်းပြပေးလို့ ရမလဲ” လို့ မေးလိုက်တယ်။

Ann က whiteboard ပေါ်က flowchart ဆီကို လမ်းလျှောက်သွားတယ်။ “သတင်းအချက်အလက် စီးဆင်းပုံကနေ စရအောင်။ ကျွန်မတို့ရဲ့ financial system (ဘဏ္ဍာရေး စနစ်) က ဌာနအသီးသီးက payroll data (လစာ အချက်အလက်) တွေကို နည်းလမ်းအမျိုးမျိုးနဲ့ ရယူပါတယ်။ ကျွန်မတို့က လစာစား ဝန်ထမ်းတွေနဲ့ နာရီစား ဝန်ထမ်းတွေရဲ့ လစာ၊ အခွန် စတာတွေကို ပေါင်းစည်းရတာပါ။ လွယ်တယ်လို့ ထင်ရပေမဲ့ တကယ်တော့ အရမ်း ရှုပ်ထွေးပါတယ်၊ ဘာလို့လဲဆိုတော့ ပြည်နယ်တစ်ခုချင်းစီရဲ့ အခွန်ဇယားတွေ၊ အလုပ်သမား ဥပဒေတွေက မတူကြလို့ပါ။

“ဘာမှ မမှားယွင်းအောင်လို့” သူက ဆက်ပြောတယ်၊ “ကျွန်မတို့က စုစုပေါင်း ကိန်းဂဏန်းတွေဟာ ဌာနတစ်ခုချင်းစီက အသေးစိတ် ကိန်းဂဏန်းတွေနဲ့ ကိုက်ညီမှု ရှိမရှိ စစ်ဆေးရပါတယ်။”

ကျွန်တော် အမြန်ပဲ မှတ်သားထားလိုက်တယ်၊ သူက ဆက်ပြောပြန်တယ်၊ “ဒါက တကယ်တော့ လက်နဲ့လုပ်ရတဲ့ manual process (လက်ဖြင့် လုပ်ဆောင်ရသော လုပ်ငန်းစဉ်) ဖြစ်လို့ အတော်လေးကို အဆင်မပြေလှပါဘူး။ အချိန်အတော်များများမှာတော့ အဆင်ပြေပေမဲ့ မနေ့ကတော့ နာရီစား ဝန်ထမ်းတွေအတွက် general ledger upload (စာရင်းချုပ် တင်သွင်းခြင်း) က မရောက်လာတာကို တွေ့လိုက်ရတယ်။ နာရီစား အားလုံးရဲ့ အလုပ်လုပ်တဲ့ နာရီနေရာမှာ သုညတွေပဲ ပြနေတယ်။

“ဒီ upload ကိစ္စနဲ့ ပတ်သက်ပြီး ကျွန်မတို့ ပြဿနာတွေ ခဏခဏ ဖြစ်ဖူးတယ်” လို့ သူက တကယ်ကို စိတ်ပျက်လက်ပျက်နဲ့ ပြောတယ်၊ “အဲဒါကြောင့် IT က ကျွန်မတို့ကို manual corrections (လက်ဖြင့် ပြုပြင်ခြင်း) လုပ်လို့ရတဲ့ program တစ်ခု ပေးထားတယ်၊ အဲဒါကြောင့် သူတို့ကို ထပ်ပြီး မနှောင့်ယှက်တော့တာပေါ့။”

ကျွန်တော် မျက်မှောင်ကြုတ်လိုက်မိတယ်။ ဘဏ္ဍာရေး ဝန်ထမ်းတွေက payroll application (လစာပေးဆော့ဖ်ဝဲ) ရဲ့ အပြင်ဘက်ကနေ payroll data တွေကို manual နဲ့ လိုက်ပြင်နေတာကို ကျွန်တော် လုံးဝ သဘောမကျပါဘူး။ ဒါက မှားယွင်းဖို့ အရမ်း လွယ်ကူသလို အန္တရာယ်လည်း များပါတယ်။ တစ်ယောက်ယောက်က အဲဒီ data တွေကို USB drive ထဲ ကူးယူသွားတာမျိုး ဒါမှမဟုတ် အပြင်ကို အီးမေးလ်နဲ့ ပို့လိုက်တာမျိုး ဖြစ်သွားနိုင်ပြီး အဲဒီလိုပဲ sensitive data (ထိခိုက်လွယ်သော အချက်အလက်) တွေ ပေါက်ကြားသွားတာမျိုး ဖြစ်နိုင်လို့ပါ။

“လစာစား ဝန်ထမ်းတွေရဲ့ ကိန်းဂဏန်းတွေကတော့ အားလုံး အိုကေတယ်လို့ ပြောတာလား” လို့ ကျွန်တော် မေးလိုက်တယ်။ “ဟုတ်ပါတယ်” လို့ သူက ပြန်ဖြေတယ်။

“ဒါပေမဲ့ နာရီစား ဝန်ထမ်းတွေရဲ့ နေရာမှာတော့ သုညတွေပဲ ဖြစ်နေတာပေါ့” လို့ ကျွန်တော် အတည်ပြုလိုက်တယ်။

“ဟုတ်ကဲ့” လို့ သူက ထပ်ပြောတယ်။

စိတ်ဝင်စားစရာပဲ။ ကျွန်တော်က “အရင်က အဆင်ပြေနေတဲ့ payroll run က ဘာလို့ အခုမှ fail ဖြစ်သွားတာလဲလို့ ထင်လဲ။ အရင်ကရော ဒီလို ပြဿနာမျိုး ကြုံဖူးသလား” လို့ မေးလိုက်တယ်။

သူက ခပ်ပေါ့ပေါ့ပဲ ပုခုံးတွန့်ပြတယ်။ “ဒီလိုမျိုး တစ်ခါမှ မဖြစ်ဖူးပါဘူး။ ဘာကြောင့် ဖြစ်ရလဲဆိုတာ ကျွန်မ လုံးဝ မသိဘူး—ဒီလစာပေးတဲ့ ကာလမှာ ဘာ အပြောင်းအလဲကြီးကြီးမားမားမှလည်း မရှိခဲ့ပါဘူး။ ကျွန်မလည်း ဒီမေးခွန်းတွေကိုပဲ ပြန်မေးနေမိတာ၊ ဒါပေမဲ့ IT ကလူတွေဆီက သတင်းမရမချင်းတော့ ကျွန်မတို့ ဘာမှ လုပ်လို့မရဘူး ဖြစ်နေတယ်။”

“အကယ်၍ နာရီစား ဝန်ထမ်းတွေရဲ့ data ကို အချိန်မီ မရခဲ့ရင် ကျွန်တော်တို့ရဲ့ backup plan (အရန် အစီအစဉ်) က ဘာလဲ” လို့ ကျွန်တော် မေးလိုက်တယ်။

“အဲဒါ မင်းကိုင်ထားတဲ့ အီးမေးလ်ထဲမှာ ပါတယ်လေ” လို့ Dick က ပြောတယ်။

“electronic payments (အီလက်ထရွန်နစ် ငွေပေးချေမှု) အတွက် သတ်မှတ်ထားတဲ့ အချိန်က ဒီနေ့ ညနေ ၅ နာရီပဲ။ အဲဒီအချိန်မီအောင် မလုပ်နိုင်ရင် ဝန်ထမ်းတွေကို လစာတွေ ပေးဖို့အတွက် စက်ရုံတစ်ခုချင်းစီဆီကို ချက်လက်မှတ်တွေ (paper checks) အထုပ်လိုက် FedEx နဲ့ ပို့ပေးရလိမ့်မယ်။”

အဲဒီ အခြေအနေကို သိလိုက်ရတဲ့အခါ ကျွန်တော်ရော ဘဏ္ဍာရေး အဖွဲ့ပါ မျက်မှောင်ကြုတ်သွားကြတယ်။

“အဲဒါကတော့ အလုပ်မဖြစ်ပါဘူး” လို့ Ann က အမှတ်အသားတံကို သွားနဲ့ ကိုက်ရင်း ပြောတယ်။ “ကျွန်မတို့ရဲ့ payroll processing ကို အပြင်က ကုမ္ပဏီတစ်ခုကို အပ်ထားတာပါ။ လစာပေးတဲ့ ကာလတိုင်းမှာ ကျွန်မတို့က payroll data တွေကို သူတို့ဆီ upload လုပ်ပေးရတယ်၊ ပြီးမှ သူတို့က လုပ်ဆောင်ပေးတာပေါ့။ အဆိုးဆုံး အခြေအနေမှာတော့ အရင်ခေါက်က payroll run ကို ဒေါင်းလုဒ်ဆွဲပြီး spreadsheet ထဲမှာ လိုက်ပြင်ပြီးမှ ပြန်ပြီး re-upload လုပ်ရမှာလား။

“ဒါပေမဲ့ ဝန်ထမ်းတစ်ယောက်ချင်းစီက ဘယ်နှစ်နာရီ အလုပ်လုပ်ခဲ့လဲဆိုတာ ကျွန်မတို့ မသိရတော့ သူတို့ကို ဘယ်လောက် ပေးရမလဲဆိုတာ မသိတော့ဘူးလေ” လို့ သူက ဆက်ပြောတယ်။ “ဘယ်သူ့ကိုမှ ပိုမပေးချင်ပေမဲ့ လျှော့ပြီး ပေးမိတာထက်စာရင်တော့ ပိုပေးမိတာက ပိုကောင်းပါလိမ့်မယ်။”

အစီအစဉ် B ကတော့ ပြဿနာတွေ အများကြီး ရှိနေတာ သိသာပါတယ်။ ကျွန်တော်တို့က ဝန်ထမ်းတွေရဲ့ လစာတွေကို ခန့်မှန်းပြီး ပေးရသလို ဖြစ်နေမှာဖြစ်သလို၊ အလုပ်ထွက်သွားတဲ့ လူတွေကို ပေးမိတာ၊ အသစ်ဝင်တဲ့ လူတွေကို မပေးမိတာမျိုးတွေလည်း ဖြစ်နိုင်ပါတယ်။

Finance က လိုအပ်တဲ့ data တွေကို ရဖို့အတွက် ကျွန်တော်တို့က custom reports (စိတ်ကြိုက် အစီရင်ခံစာ) တွေ ဖန်တီးပေးရပါလိမ့်မယ်၊ အဲဒါက application developers (ဆော့ဖ်ဝဲ ရေးသားသူ) တွေ ဒါမှမဟုတ် database သမားတွေကို ခေါ်လာဖို့ လိုပါလိမ့်မယ်။

ဒါပေမဲ့ အဲဒါက မီးလောင်ရာ လေပင့်သလိုပါပဲ။ Developers တွေက networking သမားတွေထက်တောင် ပိုဆိုးပါသေးတယ်။ Production စနစ်တွေကို မဖျက်ဆီးတဲ့ developer တစ်ယောက်ကို ကျွန်တော့်ကို ပြပါ၊ ကျွန်တော်က မင်းကို မှန်မှာ အငွေ့မထွက်နိုင်တဲ့ (အသက်မရှူနိုင်တဲ့) လူတစ်ယောက်ကို ပြပါ့မယ်။ ဒါမှမဟုတ် သူတို့က ခွင့်ယူနေတာလည်း ဖြစ်နိုင်ပါတယ်။

Dick က “ဒါတွေက အသုံးမကျတဲ့ ရွေးချယ်စရာ နှစ်ခုပဲ။ ဒေတာအမှန်တွေ ရတဲ့အထိ payroll run ကို ရွှေ့ဆိုင်းထားလို့ ရတယ်။ ဒါပေမဲ့ ငါတို့ အဲဒီလို မလုပ်နိုင်ဘူး—တစ်ရက်လေး နောက်ကျရင်တောင် သမဂ္ဂက ဝင်ပါလာလိမ့်မယ်။ ဒါကြောင့် Ann ပြောသလိုပဲ ပမာဏ မမှန်ရင်တောင် တစ်ခုခုတော့ ပေးလိုက်ရမှာပဲ။ နောက်လစာပေးတဲ့ ကာလကျမှ ပြန်ပြီး ညှိနှိုင်းရမှာပေါ့။ ဒါပေမဲ့ အခုတော့ ငါတို့ဆီမှာ financial reporting error (ဘဏ္ဍာရေး အစီရင်ခံစာ အမှား) တစ်ခု ရှိသွားပြီး ပြန်ပြင်ရတော့မှာပေါ့။”

သူက နှာခေါင်းကို ခြေရင်းကနေ ညှစ်လိုက်ပြီး ဆက်ပြောတယ်၊ “ငါတို့ရဲ့ general ledger မှာ ပုံမှန်မဟုတ်တဲ့ စာရင်းတွေ အများကြီး ဖြစ်သွားမယ်၊ အဲဒီအချိန်မှာပဲ စာရင်းစစ်တွေက SOX-404 စစ်ဆေးဖို့ ရောက်လာကြတော့မှာ။ ဒါကို မြင်ရင် သူတို့ ဘယ်တော့မှ ပြန်ကြတော့မှာ မဟုတ်ဘူး။

“ဘုရားရေ၊ financial reporting error လား” လို့ Dick က ညည်းပြတယ်။ “Steve ရဲ့ ခွင့်ပြုချက် လိုလိမ့်မယ်။ စာရင်းစစ်တွေက နွားတွေ တောထဲက ပြန်လာတဲ့အထိ (အချိန်အတော်ကြာတဲ့အထိ) ဒီမှာ တပ်စွဲနေကြတော့မှာပဲ။ ဘယ်သူမှ တကယ့် အလုပ်တွေကို လုပ်နိုင်ကြတော့မှာ မဟုတ်ဘူး။”

SOX-404 ဆိုတာ ၂၀၀၂ ခုနှစ်မှာ Enron၊ WorldCom နဲ့ Tyco တို့ရဲ့ စာရင်းကိုင် အမှားတွေအတွက် ကွန်ဂရက်က ပြဋ္ဌာန်းလိုက်တဲ့ Sarbanes-Oxley Act ကို ခေါ်တာပါ။ CEO နဲ့ CFO တို့က ကုမ္ပဏီရဲ့ ဘဏ္ဍာရေး အစီရင်ခံစာတွေ မှန်ကန်ပါတယ်လို့ သူတို့ကိုယ်တိုင် လက်မှတ်ရေးထိုးရတာ ဖြစ်ပါတယ်။

စာရင်းစစ်တွေနဲ့ စကားပြောရင်း၊ အသစ်ထွက်လာတဲ့ စည်းမျဉ်းတွေကို လိုက်နာရင်းနဲ့ အချိန်တွေ မကုန်ဆုံးခဲ့ရတဲ့ နေ့ရက်တွေကို လူတိုင်း တမ်းတနေကြပါတယ်။

ကျွန်တော့် မှတ်စုတွေကို ကြည့်လိုက်ပြီး နာရီကို ကြည့်လိုက်တယ်။ အချိန်တွေ ကုန်ဆုံးနေပြီ။

“Dick၊ ကျွန်တော် ကြားလိုက်ရသမျှအရဆိုရင်တော့ အဆိုးဆုံး အခြေအနေအတွက် ဆက်ပြီး ပြင်ဆင်ထားဖို့နဲ့ အစီအစဉ် B ကို သေသေချာချာ မှတ်တမ်းတင်ထားဖို့ အကြံပြုချင်ပါတယ်။ ဒါမှပဲ နောက်ထပ် ပြဿနာ မရှိဘဲ တစ်ခုခု လုပ်နိုင်မှာပါ။ ဒါ့အပြင် ဆုံးဖြတ်ချက် မချခင် ညနေ ၃ နာရီအထိ စောင့်ကြည့်ဖို့ ကျွန်တော် တောင်းဆိုချင်ပါတယ်။ စနစ်တွေနဲ့ data တွေကို ပြန်ရနိုင်မလားဆိုတာ ကျွန်တော်တို့ ကြိုးစားကြည့်လို့ ရပါသေးတယ်။”

Ann က ခေါင်းညိတ်ပြတဲ့အခါ Dick က “အိုကေ၊ မင်းမှာ လေးနာရီ အချိန်ရှိတယ်” လို့ ပြောတယ်။

ကျွန်တော်က “ဒီအခြေအနေရဲ့ အရေးကြီးမှုကို ကျွန်တော်တို့ နားလည်ပါတယ်၊ ကျွန်တော် ကိုယ်တိုင် သတင်းရတာနဲ့ ချက်ချင်း အသိပေးပါ့မယ်” လို့ ပြောလိုက်တယ်။

“ကျေးဇူးပါ Bill” လို့ Ann က ပြောတယ်။ ကျွန်တော် လှည့်ထွက်သွားတဲ့အချိန်မှာ Dick ကတော့ ဘာမှ မပြောတော့ပါဘူး။

လုပ်ငန်းခွင်ရဲ့ ရှုထောင့်ကနေ ပြဿနာကို မြင်လိုက်ရတဲ့အတွက် ကျွန်တော် စိတ်သက်သာရာ ရသွားတယ်။ အခုတော့ payroll စနစ်ကြီးကို ဘာက ဖျက်ဆီးလိုက်လဲဆိုတာကို ရှာဖွေဖို့ အချိန်ရောက်ပါပြီ။

လှေကားကနေ ဆင်းလာရင်းနဲ့ ကျွန်တော့်ဖုန်းကို ထုတ်ပြီး အီးမေးလ်တွေကို စစ်ဆေးလိုက်တယ်။ Steve က ကျွန်တော့်ရဲ့ ရာထူးတိုးတဲ့ သတင်းကို ကြေညာချက် မထုတ်ပြန်သေးတာကို တွေ့လိုက်ရတဲ့အခါ ကျွန်တော့်ရဲ့ တည်ငြိမ်မှုတွေ ပျောက်ဆုံးသွားပြန်တယ်။ ဒီနေ့အထိ ကျွန်တော့်ရဲ့ လုပ်ဖော်ကိုင်ဖက်တွေ ဖြစ်ခဲ့တဲ့ Wes Davis နဲ့ Patty McKee တို့က ကျွန်တော်ဟာ သူတို့ရဲ့ အထက်လူကြီးသစ် ဖြစ်လာတယ်ဆိုတာကို အခုထိ မသိကြသေးဘူး။

ကျေးဇူးပဲ Steve ရေ။

Building 7 ထဲကို ဝင်လိုက်တဲ့အခါ ကျွန်တော့်ကို ချက်ချင်း ထိမှန်သွားတယ်။ ကျွန်တော်တို့ အဆောက်အဦးက Parts Unlimited တစ်ဝင်းလုံးမှာ အစုတ်ဆုံး နေရာပါပဲ။

ဒါကို ၁၉၅၀ ပြည့်လွန်နှစ်တွေက တည်ဆောက်ခဲ့တာဖြစ်ပြီး ၁၉၇၀ ပြည့်လွန်နှစ်တွေမှာ နောက်ဆုံး ပြုပြင်ခဲ့တာပါ၊ အလှအပထက် အသုံးဝင်မှုကိုပဲ ဦးစားပေးပြီး ဆောက်ထားတာ သိသာပါတယ်။ Building 7 က အရင်က ဘရိတ်ပြား (brake-pad) ထုတ်တဲ့ စက်ရုံကြီးပါ၊ နောက်မှ data center နဲ့ ရုံးခန်းတွေအဖြစ် ပြောင်းလဲလိုက်တာပါ။ အိုဟောင်းပြီး ပစ်ပယ်ခံထားရသလိုပဲ။

လုံခြုံရေးဝန်ထမ်းက “မင်္ဂလာပါ မစ္စတာ Palmer။ ဒီနေ့ မနက်ခင်း အဆင်ပြေရဲ့လား” လို့ ရွှင်ရွှင်ပျပျပဲ နှုတ်ဆက်ပါတယ်။

ဒီအပတ်မှာ သူ လစာ အမှန်အတိုင်း ရဖို့အတွက် ကျွန်တော့်ကို ဆုတောင်းပေးဖို့ ပြောဖို့တောင် ခဏလောက် စဉ်းစားလိုက်မိတယ်။ ကျွန်တော်ကတော့ အလိုက်သင့်ပဲ ပြန်ပြီး နှုတ်ဆက်လိုက်ပါတယ်။

Wes နဲ့ Patty တို့ ရှိနိုင်မယ့် Network Operations Center (သို့မဟုတ်) ကျွန်တော်တို့ ခေါ်တဲ့ NOC ဆီကို ကျွန်တော် သွားနေပါတယ်။ သူတို့က အခု ကျွန်တော့်ရဲ့ အဓိက စီမံခန့်ခွဲသူ နှစ်ယောက် ဖြစ်လာကြပြီ။

Wes က Director of Distributed Technology Operations ပါ။ သူက Windows servers တစ်ထောင်ကျော်နဲ့အတူ database နဲ့ networking အဖွဲ့တွေကို နည်းပညာအရ တာဝန်ယူထားသူပါ။ Patty ကတော့ Director of IT Service Support ပါ။ သူက ဖုန်းတွေကို အချိန်ပြည့် ဖြေကြားပေးနေတဲ့ level 1 နဲ့ 2 input help desk technicians တွေကို တာဝန်ယူထားပြီး အလုပ်အကိုင်တွေနဲ့ ပတ်သက်တဲ့ ပြဿနာတွေကို ကိုင်တွယ်ပေးသူပါ။ တစ်ဖွဲ့လုံး အားကိုးနေရတဲ့ trouble ticketing system (ပြဿနာ မှတ်တမ်းတင်စနစ်)၊ စောင့်ကြည့်စစ်ဆေးခြင်းနဲ့ change management meeting တွေကိုလည်း သူကပဲ ဦးစီးတာပါ။

ကျွန်တော် cubicles (အကန့်လေးများ) တန်းစီနေတဲ့ နေရာကို ဖြတ်လျှောက်လာခဲ့တယ်၊ ဒါက တခြား အဆောက်အဦးတွေနဲ့ အတူတူပါပဲ။ ဒါပေမဲ့ Buildings 2 နဲ့ 5 တို့နဲ့ မတူတာကတော့ ကွာကျနေတဲ့ ဆေးတွေနဲ့ ကော်ဇော်ပေါ်က အမည်းစက်တွေကို တွေ့နေရတာပါပဲ။

ဒီအပိုင်းကို အရင်က အဓိက တပ်ဆင်ရေး စက်ရုံဟောင်းပေါ်မှာ တည်ဆောက်ထားတာပါ။ ရုံးခန်းအဖြစ် ပြောင်းလဲလိုက်တဲ့အခါမှာ စက်ဆီတွေကို စင်အောင် မသန့်ရှင်းနိုင်ခဲ့ကြဘူး။ ကြမ်းပြင်ကို ဘယ်လောက်ပဲ ဆေးတွေနဲ့ ဖုံးအုပ်ထားပါစေ၊ ဆီတွေကတော့ ကော်ဇော်ပေါ်ကို ပြန်စိမ့်ထွက်လာတတ်ပါတယ်။

ကော်ဇော်တွေ လဲဖို့နဲ့ နံရံတွေကို ဆေးပြန်သုတ်ဖို့ ဘတ်ဂျက် တောင်းရမယ်လို့ ကျွန်တော် မှတ်သားထားလိုက်တယ်။ မရိန်းတပ်မှာတုန်းကတော့ စစ်တန်းလျားကို သေသေသပ်သပ် ထားတာက အလှအပအတွက်တင် မဟုတ်ဘဲ လုံခြုံရေးအတွက်လည်း ပါဝင်တာပါ။

အကျင့်ဟောင်းကတော့ ဖျောက်ရခက်ပါတယ်လေ။

NOC ကို မမြင်ရခင်ကတည်းက အသံကို ကြားနေရပြီ။ ဒါက အခန်းကျယ်ကြီးတစ်ခုပါ၊ နံရံတစ်လျှောက်မှာ စားပွဲရှည်တွေ ရှိပြီး IT ဝန်ဆောင်မှုတွေရဲ့ အခြေအနေကို မော်နီတာကြီးတွေနဲ့ ပြသထားပါတယ်။ Level 1 နဲ့ 2 help desk သမားတွေက တန်းစီထားတဲ့ နေရာတွေမှာ ထိုင်နေကြတယ်။

ဒါက *Apollo 13* ထဲက mission control နဲ့တော့ လုံးဝ မတူပါဘူး၊ ဒါပေမဲ့ ကျွန်တော့် အမျိုးတွေကိုတော့ အဲဒီလိုပဲ ရှင်းပြလေ့ရှိတယ်။

ပြဿနာတစ်ခုခု တက်လာပြီဆိုရင်တော့ တာဝန်ရှိသူတွေနဲ့ နည်းပညာ မန်နေဂျာတွေ အားလုံးက ပြဿနာ မပြေလည်မချင်း ပူးပေါင်းဆောင်ရွက်ဖို့ လိုပါတယ်။ အခုလို အချိန်မျိုးပေါ့။ အစည်းအဝေးစားပွဲမှာ လူ ၁၅ ယောက်လောက်က အကျိတ်အနယ် ဆွေးနွေးနေကြပြီး UFO နဲ့ တူတဲ့ မီးခိုးရောင် စပီကာဖုန်းလေးကို ဝိုင်းနေကြပါတယ်။

Wes နဲ့ Patty တို့က အစည်းအဝေးစားပွဲမှာ ဘေးချင်းယှဉ်ထိုင်နေကြလို့ ကျွန်တော် သူတို့နောက်ကနေ သွားပြီး နားထောင်လိုက်တယ်။ Wes က ကုလားထိုင်ပေါ်မှာ မှီထိုင်ရင်း လက်နှစ်ဖက်ကို ဗိုက်ပေါ်မှာ ပိုက်ထားတယ်။ လက်ကတော့ ဗိုက်ကို အကုန် မပတ်မိပါဘူး။ အရပ် ၆ ပေ ၃ လက်မ ရှိပြီး အလေးချိန် ၂၅၀ ကျော်ရှိတဲ့ သူက လူအများစုအပေါ် အရိပ်မိုးထားနိုင်သူပါ။ သူက အမြဲတမ်း လှုပ်ရှားနေတတ်ပြီး သူ့စိတ်ထဲ ရှိတာတွေကို တဲ့တိုး ပြောတတ်သူအဖြစ်လည်း နာမည်ကြီးပါတယ်။

Patty ကတော့ လုံးဝ ဆန့်ကျင်ဘက်ပါ။ Wes က အသံကျယ်ကျယ်နဲ့ ပွင့်လင်းစွာ ပြောတတ်ပြီး လက်တည့်စမ်းတတ်သူဖြစ်ပေမဲ့ Patty ကတော့ စဉ်းစားဆင်ခြင်တတ်သူ၊ စနစ်ကျသူနဲ့ လုပ်ငန်းစဉ်တွေကို တိကျစွာ လိုက်နာသူပါ။ Wes က ခန္ဓာကိုယ်ကြီးမားပြီး တိုက်ခိုက်လိုစိတ်ရှိကာ တစ်ခါတလေ ရန်လိုတတ်ပေမဲ့ Patty ကတော့ သေးသေးသွယ်သွယ်နဲ့ ယုတ္တိရှိကာ တည်ငြိမ်သူပါ။ သူက လူတွေထက် လုပ်ငန်းစဉ်တွေကို ပိုချစ်မြတ်နိုးသူအဖြစ် နာမည်ကြီးပြီး IT ထဲက ရှုပ်ထွေးမှုတွေကို စနစ်တကျ ဖြစ်အောင် လုပ်ပေးရသူလည်း ဖြစ်ပါတယ်။

သူက IT အဖွဲ့တစ်ခုလုံးရဲ့ မျက်နှာစာပါပဲ။ IT မှာ တစ်ခုခု မှားယွင်းသွားရင် လူတွေက Patty ကိုပဲ ဖုန်းခေါ်ကြတယ်။ သူက ကျွန်တော်တို့ရဲ့ တရားဝင် တောင်းပန်မည့်သူဖြစ်ပြီး ဝန်ဆောင်မှုတွေ ပျက်စီးတာပဲဖြစ်ဖြစ်၊ ဝက်ဘ်စာမျက်နှာတွေ တက်လာတာ နောက်ကျတာပဲဖြစ်ဖြစ်၊ ဒါမှမဟုတ် ဒီနေ့လိုမျိုး data တွေ ပျောက်ဆုံးတာ ဒါမှမဟုတ် ပျက်စီးတာမျိုးတွေ ဖြစ်လာရင် သူပဲ ဖြေရှင်းရတာပါ။

ကွန်ပျူတာ အဆင့်မြှင့်ချင်တာ၊ ဖုန်းနံပါတ် ပြောင်းချင်တာ ဒါမှမဟုတ် application အသစ် တင်ချင်တာမျိုးတွေအတွက်လည်း လူတွေက Patty ကိုပဲ ခေါ်ကြပါတယ်။ သူက အချိန်ဇယားဆွဲပေးရသူဖြစ်လို့ လူတွေက သူတို့အလုပ်ကို အရင်ဆုံး လုပ်ပေးဖို့ သူ့ကိုပဲ အမြဲ ဖိအားပေးကြပါတယ်။ ပြီးမှ သူက အလုပ်လုပ်မယ့်လူတွေကို လွှဲပေးတာပါ။ အဲဒီလူတွေကတော့ ကျွန်တော့် အဖွဲ့ဟောင်း ဒါမှမဟုတ် Wes ရဲ့ အဖွဲ့ထဲမှာ ရှိကြတာပါ။

Wes က စားပွဲကို ထုလိုက်ပြီး “vendor (ပစ္စည်းရောင်းချသူ) ဆီကို ဖုန်းဆက်ပြီး အခုချက်ချင်း နည်းပညာရှင် တစ်ယောက် လွှတ်မပေးရင် ပြိုင်ဘက်ဆီ ပြောင်းမယ်လို့ ပြောလိုက်။ ငါတို့က သူတို့ရဲ့ အကြီးဆုံး customer တွေလေ! စဉ်းစားကြည့်ရင် အဲဒီ အမှိုက်ပုံကြီးကို ငါတို့ အခုလောက်ဆို စွန့်ပစ်ထားသင့်တာ” လို့ ပြောလိုက်တယ်။

သူက ပတ်ပတ်လည်ကို ကြည့်ပြီး ဟာသလုပ်လိုက်သေးတယ်၊ “မင်းတို့ သိကြတယ်မဟုတ်လား။ vendor တစ်ယောက် လိမ်နေတယ်ဆိုတာကို သူတို့ နှုတ်ခမ်းတွေ လှုပ်နေတာကို ကြည့်ပြီး သိနိုင်တယ်လေ။”

Wes ရဲ့ တစ်ဖက်က အင်ဂျင်နီယာတစ်ယောက်က “သူတို့နဲ့ ဖုန်းပြောနေပါတယ်။ သူတို့ရဲ့ SAN field engineer (နယ်ပယ် အင်ဂျင်နီယာ) ဒီကို ရောက်ဖို့ အနည်းဆုံး လေးနာရီလောက် ကြာဦးမှာတဲ့” လို့ ပြောတယ်။

ကျွန်တော် မျက်မှောင်ကြုတ်လိုက်မိတယ်။ ဘာဖြစ်လို့ သူတို့က SAN အကြောင်း ပြောနေကြတာလဲ။ SAN (Storage Area Networks) ဆိုတာ ကျွန်တော်တို့ရဲ့ အရေးကြီးဆုံး စနစ်တွေအတွက် ဗဟိုထိန်းချုပ် သိမ်းဆည်းတဲ့ နေရာဖြစ်လို့ အဲဒါ ပျက်စီးရင် တစ်ခုလုံးကို ထိခိုက်စေတတ်ပါတယ်—server တစ်လုံးတည်း ပျက်မှာ မဟုတ်ဘဲ ရာနဲ့ချီတဲ့ server တွေ တစ်ပြိုင်တည်း ပျက်သွားမှာပါ။

Wes က အင်ဂျင်နီယာတွေနဲ့ ငြင်းခုံနေတုန်း ကျွန်တော် စဉ်းစားလိုက်မိတယ်။ ဒီ payroll run failure က SAN ပြဿနာနဲ့တော့ မတူပါဘူး။ Ann ပြောတာကတော့ စက်ရုံတစ်ခုချင်းစီရဲ့ timekeeping applications တွေမှာ ပြဿနာရှိနိုင်တယ်လို့ ပြောခဲ့တာပါ။

“ဒါပေမဲ့ ကျွန်တော်တို့ SAN ကို rollback လုပ်ဖို့ ကြိုးစားပြီးတဲ့နောက်မှာတော့ သူက data တွေကို လုံးဝ မပေးတော့ဘူး” လို့ တခြား အင်ဂျင်နီယာတစ်ယောက်က ပြောတယ်။ “ပြီးတော့ screen ပေါ်မှာ kanji (ဂျပန်စာ) တွေပဲ ပေါ်လာတော့တာပဲ! ဟုတ်ပါတယ်၊ kanji လို့ ထင်တာပါပဲ။ ဘာစာတွေမှန်းလည်း မသိဘူး၊ နားမလည်တော့ဘူး။ အဲဒီတော့မှပဲ vendor ကို ခေါ်ဖို့ ဆုံးဖြတ်လိုက်ကြတာ။”

ကျွန်တော် နောက်ကျမှ ပါဝင်လာခဲ့တာဖြစ်ပေမဲ့ ကျွန်တော်တို့တွေ လမ်းကြောင်း မှားနေပြီဆိုတာ ကျွန်တော် သေချာပေါက် ပြောနိုင်တယ်။

ကျွန်တော် Wes နဲ့ Patty တို့ဆီ တိုးတိုးလေး ကပ်ပြောလိုက်တယ်၊ “ခဏလောက် သီးသန့် စကားပြောလို့ ရမလား။”

Wes က လှည့်ကြည့်တယ်၊ ဒါပေမဲ့ စိတ်မဝင်စားတဲ့ပုံနဲ့ “အခု စောင့်လို့ မရဘူးလား။ မင်းမမြင်ဘူးလား၊ ငါတို့ ပြဿနာ အကြီးအကျယ် တက်နေတာလေ” လို့ အသံကျယ်ကျယ်နဲ့ ပြောလိုက်တယ်။

ကျွန်တော် သူ့ပုခုံးကို ခိုင်ခိုင်မာမာ ကိုင်လိုက်တယ်၊ “Wes၊ ဒါ တကယ် အရေးကြီးတယ်။ payroll failure ကိစ္စဖြစ်ပြီး Steve Masters နဲ့ Dick Landry တို့နဲ့ ခုလေးတင် ပြောခဲ့တဲ့ ကိစ္စပါ။”

သူ အံ့သြသွားတယ်။ Patty ကတော့ သူ့ကုလားထိုင်ကနေ ထပြီးသားပါပဲ။ “ကျွန်မ ရုံးခန်းမှာပဲ ပြောရအောင်” လို့ သူက ဦးဆောင်ခေါ်သွားတယ်။

Patty ရဲ့ ရုံးခန်းထဲကို လိုက်သွားတဲ့အခါ နံရံမှာ သူ့သမီးရဲ့ ဓာတ်ပုံကို တွေ့လိုက်ရတယ်။ အသက် ၁၁ နှစ်လောက် ရှိမယ် ထင်ရတူပါ။ Patty နဲ့ တကယ့်ကို တူတာပဲ—မကြောက်မရွံ့တတ်တဲ့ပုံ၊ ဉာဏ်ကောင်းတဲ့ပုံနဲ့ တကယ်ကို စွမ်းဆောင်ရည်ရှိတဲ့ ပုံမျိုးပါပဲ။ ကလေးမလေး တစ်ယောက်အတွက်တော့ အဲဒီလို ဖြစ်နေတာက နည်းနည်းတော့ ကြောက်ဖို့ ကောင်းပါတယ်။

Wes က ခပ်ပြတ်ပြတ်ပဲ “ကဲ Bill၊ Sev 1 outage (အဆင့် ၁ ပြင်းထန်သော ပြတ်တောက်မှု) ဖြစ်နေတာကို ကြားဖြတ် ပြောရလောက်အောင် ဘာက ဒီလောက် အရေးကြီးနေလို့လဲ” လို့ မေးတယ်။

ဒါက မေးသင့်တဲ့ မေးခွန်းပါပဲ။ Severity 1 outages ဆိုတာ လုပ်ငန်းကို အကြီးအကျယ် ထိခိုက်စေတဲ့ ပြဿနာမျိုးဖြစ်လို့ အရာအားလုံးကို ထားခဲ့ပြီး ဒါကိုပဲ ဖြေရှင်းကြရတာပါ။

ကျွန်တော် အသက်ပြင်းပြင်း ရှူလိုက်တယ်။ “မင်းတို့ သိပြီးကြပြီလားတော့ မသိဘူး၊ Luke နဲ့ Damon တို့ ကုမ္ပဏီမှာ မရှိတော့ဘူး။ တရားဝင် သတင်းကတော့ သူတို့ နားချင်လို့ ထွက်သွားတာတဲ့။ ကျန်တာတော့ ငါလည်း မသိဘူး။”

သူတို့ရဲ့ အံ့သြသွားတဲ့ မျက်နှာတွေကို ကြည့်ရုံနဲ့ သူတို့ မသိခဲ့ဘူးဆိုတာ သိသာပါတယ်။ ကျွန်တော် ဒီမနက် ဖြစ်ခဲ့တာတွေကို အကျဉ်းချုပ် ပြောပြလိုက်တယ်။ Patty က ခေါင်းယမ်းပြီး မကျေနပ်တဲ့အမူအရာနဲ့ ညည်းပြတယ်။

Wes ကတော့ တကယ် စိတ်တိုသွားပုံရတယ်။ သူက Damon နဲ့ အလုပ်လုပ်လာတာ နှစ်ပေါင်းများစွာ ရှိပြီလေ။ သူ့မျက်နှာ နီမြန်းလာတယ်။ “ဒါဆို အခု ငါတို့က မင်းရဲ့ အမိန့်ကို နားထောင်ရမှာပေါ့။ ဟုတ်လား။ သူငယ်ချင်း၊ ငါ ပြောတာကို စိတ်မဆိုးနဲ့ဦး၊ မင်းက ဒီရာထူးနဲ့ နည်းနည်း များ ဝေးမနေဘူးလား။ မင်းက midrange systems တွေကိုပဲ တာဝန်ယူတာ နှစ်ပေါင်းများစွာ ရှိနေပြီလေ၊ အဲဒါက တကယ်တော့ ရှေးဟောင်း ပစ္စည်းတွေလိုပဲ။ အဲဒီမှာ မင်းက ကိုယ့်အတွက် အေးအေးဆေးဆေး အလုပ်တစ်ခု ဖန်တီးထားတာပေါ့။ မင်း သိရဲ့လား၊ မင်းမှာ ခေတ်မီတဲ့ distributed systems (ဖြန့်ကြက်စနစ်များ) ကို ဘယ်လို စီမံရမလဲဆိုတာ လုံးဝ အသိမရှိဘူး—မင်းအတွက်ကတော့ ၁၉၉၀ ပြည့်လွန်နှစ်တွေကတောင် အနာဂတ် ဖြစ်နေတုန်းပဲ!

“တကယ်တော့” သူက ဆက်ပြောတယ်၊ “ငါ နေ့တိုင်း ရင်ဆိုင်နေရတဲ့ ဒီလို ရှုပ်ထွေးပြီး လျင်မြန်တဲ့ အခြေအနေမျိုးထဲမှာ မင်းသာ နေရမယ်ဆိုရင် မင်းရဲ့ ခေါင်းက ပေါက်ကွဲသွားလိမ့်မယ်။”

ကျွန်တော် သုံးအထိ စိတ်ထဲက ရေတွက်ပြီး အသက်ရှူထုတ်လိုက်တယ်။ “မင်း ငါ့အလုပ်ကို လိုချင်တယ်ဆိုရင် Steve ကို သွားပြောလို့ ရတယ်။ ဒါပေမဲ့ အခုတော့ လုပ်ငန်းအတွက် လိုအပ်တာကို အရင်လုပ်ပြီး အားလုံး လစာ အချိန်မီ ရအောင် အရင် လုပ်ကြရအောင်။”

Patty က ချက်ချင်း ဝင်ပြောတယ်၊ “ရှင် ကျွန်မကို မမေးခဲ့ပေမဲ့ payroll incident (လစာပေးခြင်းဆိုင်ရာ ပြဿနာ) ကို အာရုံစိုက်ရမယ်ဆိုတာ ကျွန်မ သဘောတူပါတယ်။ Steve က ရွေးချယ်မှု မှန်ကန်တယ်လို့ ကျွန်မ ထင်ပါတယ်။ ဂုဏ်ယူပါတယ် Bill။ ကျွန်မတို့ ဘတ်ဂျက် တိုးဖို့ကိစ္စကို ဘယ်တော့ ဆွေးနွေးကြမလဲ။”

ကျွန်တော် သူ့ကို ပြုံးပြလိုက်ပြီး Wes ဘက်ကို ပြန်လှည့်လိုက်တယ်။ ခဏလောက် ကြာတဲ့အခါမှာတော့ Wes က “အေးပါ၊ ထားလိုက်ပါတော့။ Steve နဲ့လည်း ငါ သွားပြောမှာပါ။ သူ ရှင်းပြစရာတွေ အများကြီး ရှိလိမ့်မယ်” လို့ ပြောလာတယ်။

ကျွန်တော် ခေါင်းညိတ်လိုက်တယ်။ Steve နဲ့ ကျွန်တော့်ရဲ့ အတွေ့အကြုံအရဆိုရင် Wes သာ သူ့ကို တကယ် စိန်ခေါ်မယ်ဆိုရင်တော့ သူ့အတွက် ကံကောင်းဖို့ပဲ ဆုတောင်းပေးရမှာပါ။

“အခုလို ကူညီပေးတဲ့အတွက် ကျေးဇူးတင်ပါတယ် သူငယ်ချင်းတို့။ ကဲ၊ အခု ပျက်စီးသွားတဲ့အရာတွေ၊ ဒါမှမဟုတ် အမှားတွေအကြောင်း ငါတို့ ဘာသိထားလဲ။ မနေ့က SAN upgrade အကြောင်းကရော ဘာလဲ။ အဲဒါတွေက ဆက်စပ်နေသလား။”

“ငါတို့လည်း မသိဘူး” လို့ Wes က ခေါင်းယမ်းရင်း ပြောတယ်။ “မင်း မဝင်လာခင်က အဲဒါကိုပဲ ရှာနေကြတာ။ မနေ့က SAN firmware upgrade လုပ်နေတုန်းမှာ payroll run က fail ဖြစ်သွားတာ။ Brent ကတော့ SAN က data တွေကို ဖျက်ဆီးနေတာလို့ ထင်လို့ အပြောင်းအလဲတွေကို ပြန်ဖျက်ဖို့ ပြောခဲ့တာပေါ့။ ငါလည်း အဲဒါက ယုတ္တိရှိတယ်လို့ ထင်ခဲ့ပေမဲ့ မင်းသိတဲ့အတိုင်း အခုတော့ အဲဒီ စက်ကြီးက bricking (ပြန်သုံးမရအောင် ပျက်စီးခြင်း) ဖြစ်သွားပြီ။”

အခုထိတော့ ဘာပဲဖြစ်ဖြစ် bricking ဖြစ်တယ်ဆိုတာ ဖုန်းတစ်လုံးကို update လုပ်ရင်း ပျက်သွားတာမျိုး သေးသေးမွှားမွှား ကိစ္စတွေမှာပဲ သုံးတာ ကြားဖူးတာပါ။

အခုတော့ ဒေါ်လာ သန်းချီတန်တဲ့၊ ကုမ္ပဏီရဲ့ အစားထိုးမရတဲ့ data တွေ အားလုံးကို သိမ်းထားတဲ့ စက်ကြီးတစ်ခုအတွက် အဲဒီစကား သုံးလိုက်ရတာကို ကြားတော့ ကျွန်တော် တကယ်ကို နေမကောင်းချင်သလို ဖြစ်သွားမိတယ်။

Brent က Wes လက်အောက်မှာ အလုပ်လုပ်တာပါ။ IT ရဲ့ အရေးကြီးတဲ့ ပရောဂျက်တိုင်းမှာ သူ အမြဲ ပါဝင်လေ့ရှိတယ်။ ကျွန်တော် သူနဲ့ အလုပ်လုပ်တာ အကြိမ်ပေါင်းများစွာ ရှိပါပြီ။ သူက တကယ် ဉာဏ်ကောင်းတဲ့လူပါ၊ ဒါပေမဲ့ သူ သိထားတာတွေ များလွန်းလို့ တခြားလူတွေအပေါ် လွှမ်းမိုးတတ်တာတော့ ရှိတယ်။ ပိုဆိုးတာက သူပြောတာတွေက အချိန်အတော်များများမှာ မှန်နေတတ်တာပါပဲ။

“မင်းလည်း ကြားသားပဲ” လို့ Wes က အစည်းအဝေးစားပွဲဘက်ကို ညွှန်ပြရင်း ပြောတယ်။ “SAN က boot တက်မလာဘူး၊ data တွေလည်း မပေးဘူး၊ ငါတို့လူတွေက error message တွေကိုတောင် ဖတ်လို့ မရတော့ဘူး၊ ဘာစာတွေမှန်း မသိလို့လေ။ အခု payroll အပါအဝင် database တွေ အများကြီး ကျနေပြီ။”

“SAN ပြဿနာကို ဖြေရှင်းဖို့အတွက် Sarah ကို လုပ်ပေးမယ်လို့ ကတိပေးထားတဲ့ Phoenix ရဲ့ အလုပ်ကနေ Brent ကို ဆွဲထုတ်လိုက်ရတယ်” လို့ Patty က စိုးရိမ်တကြီး ပြောတယ်။ “ဒါကတော့ ပြဿနာ တက်တော့မှာပဲ။”

“အလို…။ သူ့ကို ဘာတွေ ကတိပေးထားလို့လဲ” လို့ ကျွန်တော် ထိတ်လန့်တကြား မေးလိုက်တယ်။

Sarah က Retail Operations ရဲ့ ဒုတိယဥက္ကဋ္ဌ (SVP) ဖြစ်ပြီး Steve ဆီမှာလည်း အလုပ်လုပ်ပါတယ်။ သူ့မှာ ထူးခြားတဲ့ စွမ်းရည်တစ်ခု ရှိတာကတော့ သူ့ရဲ့ အမှားတွေကို တခြားလူတွေအပေါ် လွှဲချတတ်တာပါပဲ၊ အထူးသဖြင့် IT သမားတွေအပေါ်မှာပေါ့။ နှစ်ပေါင်းများစွာ သူ အဲဒီလိုပဲ လွတ်မြောက်လာခဲ့တာပါ။

Steve က သူ့ကို သူ့နေရာမှာ ဆက်ခံမယ့်သူအဖြစ် လေ့ကျင့်ပေးနေတယ်လို့ ကောလာဟလတွေ ကြားဖူးပေမဲ့ ကျွန်တော်ကတော့ အဲဒါ မဖြစ်နိုင်ဘူးလို့ပဲ ထင်ခဲ့တာပါ။ Steve ဟာ သူ့ရဲ့ လုပ်ရပ်တွေကို မသိဘဲတော့ မနေနိုင်ပါဘူး။

“Chris ဆီကို virtual machines (အတုယောင် စက်များ) တွေ ပို့ပေးဖို့ ငါတို့ နောက်ကျနေတယ်လို့ တစ်ယောက်ယောက်ဆီက Sarah ကြားသွားတာ” လို့ သူက ပြန်ဖြေတယ်။ “ငါတို့ အဲဒါကို အရင်ဆုံး လုပ်ပေးဖို့ အရာအားလုံးကို ထားခဲ့လိုက်ရတယ်။ အခုတော့ SAN ကို ပြင်ဖို့အတွက် အဲဒါကိုပါ ထပ်ပြီး ထားခဲ့လိုက်ရတာပေါ့။”

Chris Allers က Application Development ရဲ့ VP ဖြစ်ပြီး လုပ်ငန်းအတွက် လိုအပ်တဲ့ application တွေနဲ့ code တွေကို ရေးသားရသူပါ။ အခုတော့ သူ့ဘဝက Phoenix နဲ့ပဲ အချိန်ကုန်နေရတာပါ။

ကျွန်တော် ခေါင်းကုတ်လိုက်မိတယ်။ ကုမ္ပဏီတစ်ခုအနေနဲ့ virtualization (အတုယောင် စနစ်ဖန်တီးခြင်း) မှာ ငါတို့ အကြီးအကျယ် ရင်းနှီးမြှုပ်နှံထားကြတယ်။ ဒါက ၁၉၆၀ ပြည့်လွန်နှစ်တွေက mainframe ပတ်ဝန်းကျင်နဲ့ တူနေပေမဲ့ virtualization က Wes ရဲ့ ကမ္ဘာကို အကြီးအကျယ် ပြောင်းလဲပစ်ခဲ့တာပါ။ ရုတ်တရက်ဆိုရင် server အလုံးပေါင်း ထောင်ချီကို ကိုယ်တိုင် လိုက်ပြီး စီမံစရာ မလိုတော့ဘူး။ အဲဒါတွေက အခုဆိုရင် server ကြီးတစ်လုံးတည်းရဲ့ အတွင်းထဲမှာ ဒါမှမဟုတ် cloud တစ်နေရာရာမှာ ရှိနေကြတာလေ။

Server အသစ်တစ်ခု ဆောက်ဖို့ဆိုတာ application တစ်ခုထဲမှာ right-click တစ်ချက် နှိပ်လိုက်ရုံပါပဲ။

Cabling (ကြိုးသွယ်တန်းခြင်း) လား။ ဒါက အခုဆိုရင် configuration ပြင်လိုက်ရုံပါပဲ။ ဒါပေမဲ့ virtualization က ပြဿနာတွေ အားလုံးကို ဖြေရှင်းပေးမယ်လို့ ကတိပေးခဲ့ပေမဲ့ အခုတော့ Chris ဆီကို virtual machine တစ်ခု ပို့ဖို့တောင် ငါတို့ နောက်ကျနေရတုန်းပါပဲ။

“SAN ကိစ္စအတွက် Brent လိုအပ်တယ်ဆိုရင် သူ့ကို အဲဒီမှာပဲ ထားလိုက်ပါ။ Sarah ကိစ္စကို ငါ ကိုင်တွယ်မယ်” လို့ ကျွန်တော် ပြောလိုက်တယ်။ “ဒါပေမဲ့ payroll failure က SAN ကြောင့်ဖြစ်တာဆိုရင် ဘာဖြစ်လို့ တခြား outages တွေ အများကြီး မဖြစ်ရတာလဲ။”

“Sarah ကတော့ တကယ်ကို စိတ်တိုတော့မှာပဲ။ အခုတော့ မင်းရဲ့ အလုပ်ကို ငါ မလိုချင်တော့ဘူး” လို့ Wes က အကျယ်ကြီး ရယ်ပြီး ပြောတယ်။ “ပထမဆုံးနေ့မှာတင် အလုပ်မပြုတ်စေနဲ့ဦး။ နောက်ဆိုရင် သူတို့ ငါ့ဆီ လာကြဦးမှာ!”

Wes က ခေတ္တရပ်ပြီး စဉ်းစားလိုက်တယ်။ “မင်းပြောတဲ့ SAN ကိစ္စကတော့ သတိထားစရာပဲ။ Brent က အခု အဲဒီကိစ္စကို လုပ်နေတုန်းပဲ။ သူ့ဆီ သွားပြီး သူ ဘာထင်လဲဆိုတာ သွားမေးကြည့်ရအောင်။”

Patty နဲ့ ကျွန်တော် ခေါင်းညိတ်လိုက်ကြတယ်။ ဒါက အကြံကောင်းပါပဲ။ ကျွန်တော်တို့မှာ ဖြစ်ပျက်ခဲ့တာတွေကို တိကျတဲ့ timeline တစ်ခု ရှိဖို့ လိုပါတယ်။ အခုထိတော့ ကျွန်တော်တို့က တစ်ဆင့်စကားတွေအပေါ်မှာပဲ အခြေခံနေရတာကိုး။ အဲဒါက ရာဇဝတ်မှုတွေကို ဖြေရှင်းတဲ့နေရာမှာလည်း အသုံးမဝင်သလို outage တွေကို ဖြေရှင်းတဲ့နေရာမှာလည်း လုံးဝ အသုံးမဝင်ပါဘူး။

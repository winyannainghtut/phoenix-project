အခန်း ၃၃ (CHAPTER 33)

• အင်္ဂါနေ့၊ နိုဝင်ဘာ ၁၁ ရက်

နောက်တစ်နေ့မှာတော့ Brent ဟာ Unicorn အဖွဲ့ဆီ ပြန်ရောက်လာခဲ့ပြီး level 3 engineers (အဆင့် ၃ အင်ဂျင်နီယာ) တစ်ယောက်ကတော့ နှင်းတွေ ကျနေတဲ့ အနောက်အလယ်ပိုင်းက Dick ရဲ့ အဖွဲ့ဆီကို သွားရောက် ပူးပေါင်းခဲ့ပါတယ်။ နာရီပိုင်းအတွင်းမှာပဲ Sarah ဆီက ပေးပို့တဲ့ အီးမေးလ်တစ်စောင်မှာ ကျွန်တော့်ကိုပါ copy (မိတ္တူ) ထည့်ထားတာကို တွေ့လိုက်ရတယ်—

ပေးပို့သူ- Sarah Moulton
လက်ခံသူ- Bob Strauss
မိတ္တူ- Dick Landry, Steve Masters, Bill Palmer
ရက်စွဲ- နိုဝင်ဘာ ၁၁၊ နံနက် ၇:၂၄ နာရီ
အကြောင်းအရာ- Project Talon ကို တစ်ယောက်ယောက်က နှောင့်ယှက်နေပါတယ်

Bob ရှင့်… IT Operations ရဲ့ ဒု-ဥက္ကဋ္ဌ (VP of IT Operations) အဖြစ် တာဝန်ယူထားတဲ့ Bill Palmer ဟာ Project Talon အတွက် အရေးအကြီးဆုံး လူသားအရင်းအမြစ် (critical resource) ကို ခိုးယူသွားတယ်ဆိုတာ ကျွန်မ သိလိုက်ရပါတယ်။

Bill… ရှင့်ရဲ့ မကြာသေးခင်က လုပ်ရပ်တွေအတွက် ကျွန်မ တကယ့်ကို စိုးရိမ်မိပါတယ်။ ဘာဖြစ်လို့ Brent ကို အိမ်ပြန်ဖို့ ခိုင်းခဲ့ရတာလဲဆိုတာ ကျွန်မတို့ကို ရှင်းပြပေးပါ။ ဒါဟာ လုံးဝ လက်ခံနိုင်စရာ မရှိတဲ့ ကိစ္စပဲ။

ဘုတ်အဖွဲ့က ကျွန်မတို့ကို မဟာဗျူဟာမြောက် ရွေးချယ်စရာတွေကို ရှာဖွေဖို့ ညွှန်ကြားထားပြီးသားပါ။

Brent ကို Talon အဖွဲ့ဆီ အမြန်ဆုံး ပြန်လည် ပူးပေါင်းစေဖို့ ကျွန်မ တောင်းဆိုပါတယ်။ ဒီ မက်ဆေ့ခ်ျကို နားလည်ကြောင်း အတည်ပြုပေးပါ။

Sarah

ကုမ္ပဏီ ဥက္ကဋ္ဌဆီ ပို့တဲ့ အီးမေးလ်မှာ ကျွန်တော့်ကို နာမည်တပ်ပြီး တိုင်ကြားခံရတာမို့ တကယ့်ကို ထိတ်လန့်သွားမိပါတယ်။ ကျွန်တော် Steve ဆီ ဖုန်းဆက်လိုက်တော့ သူက Sarah ရဲ့ ဒီလိုမျိုး သစ္စာဖောက်သလို လုပ်ရပ်အပေါ် တော်တော်လေး ဒေါသထွက်နေတာပေါ့။ သူက တိုးတိုးလေး ကျိန်ဆဲလိုက်ပြီး ကျန်တာတွေကို သူ ကိုင်တွယ်လိုက်မယ်၊ ကျွန်တော့်ဘက်ကတော့ စီစဉ်ထားတဲ့အတိုင်း ဆက်လုပ်ဖို့ အာမခံလိုက်ပါတယ်။

နေ့စဉ်လုပ်လေ့ရှိတဲ့ Unicorn stand-up meeting (အဖွဲ့လိုက် အမြန်ညှိနှိုင်းမှု အစည်းအဝေး) မှာတော့ William ရဲ့ မျက်နှာက သိပ်မကောင်းဘူးဗျ။

“သတင်းကောင်းကတော့ မနေ့ညက ငါတို့ရဲ့ ပထမဆုံး ဖောက်သည် အရောင်းမြှင့်တင်ရေး အစီရင်ခံစာ (customer promotion report) ကို ထုတ်ပေးနိုင်ခဲ့ပြီး အကုန်လုံး မှန်မှန်ကန်ကန် အလုပ်လုပ်နေပုံ ရတာပါပဲ။ ဒါပေမဲ့ ကုဒ်တွေက မျှော်လင့်ထားတာထက် အဆ ၅၀ လောက် ပိုနှေးနေတယ်ဗျ။ clustering algorithms (အုပ်စုခွဲ တွက်ချက်မှုများ) ထဲက တစ်ခုက ငါတို့ ထင်သလို parallelizing (တပြိုင်တည်း တွက်ချက်ခြင်း) မဖြစ်နေတော့ ဖောက်သည် ဒေတာ အနည်းငယ်နဲ့ စမ်းသပ်တာတောင်မှ ခန့်မှန်းချက်တွေ ထွက်လာဖို့ ၂၄ နာရီကျော် ကြာနေပြီလေ” တဲ့။

အခန်းထဲမှာ ငြီးတွားသံတွေ ထွက်လာပါတော့တယ်။

developer တစ်ယောက်က— “hardware အင်အားနဲ့ပဲ ဖြေရှင်းလို့ မရဘူးလား (brute force)။ ပစ္စည်းတွေ ပိုထည့်လိုက်ပေါ့။ compute servers (တွက်ချက်ရေး ဆာဗာများ) အများကြီး ထပ်သုံးလိုက်ရင် အချိန်ကို လျှော့ချနိုင်မှာပဲ” လို့ အကြံပေးပါတယ်။

“မင်း စနောက်နေတာလား” လို့ Wes က စိတ်ပျက်လက်ပျက်နဲ့ ဆိုတယ်- “ငါတို့က အမြန်ဆုံး ဆာဗာ ၂၀ အတွက်ပဲ ဘတ်ဂျက် လျာထားတာလေ။ မင်းပြောသလို အချိန်မီ ပြီးဖို့ဆိုရင် ဆာဗာ အလုံးပေါင်း တစ်ထောင်ကျော် လိုလိမ့်မယ်။ အဲဒါက ဘတ်ဂျက်ထဲမှာ မပါတဲ့ အရင်းအနှီး ဒေါ်လာ ၁ သန်းကျော် ကုန်မှာနော်!” တဲ့။

ကျွန်တော် နှုတ်ခမ်းကို စေ့လိုက်မိတယ်။ Wes ပြောတာ မှန်ပါတယ်။ Phoenix က ဘတ်ဂျက်တွေ အများကြီး ကျော်နေပြီဖြစ်သလို အခု ပြောနေတဲ့ ငွေပမာဏကလည်း ငါတို့ရဲ့ လက်ရှိ ဘဏ္ဍာရေး အခြေအနေအရ အတည်ပြုချက် ရဖို့ဆိုတာ မဖြစ်နိုင်ဘူးလေ။

“hardware အသစ်တွေ မလိုပါဘူး” လို့ developer က ပြန်ဖြေပါတယ်- “ငါတို့တွေ deploy (ဖြန့်ကျက်) လုပ်လို့ရမယ့် compute images (တွက်ချက်မှု ပုံရိပ်များ) တွေကို ဖန်တီးဖို့ တော်တော်လေး ကြိုးစားထားကြတာပဲ။ အဲဒါတွေကို cloud (ကလောက်ဒ် စနစ်) ဆီ ပို့လိုက်ရင်ကော။ လိုအပ်သလို compute instances (တွက်ချက်မှု ယူနစ်များ) ရာနဲ့ချီ၊ ထောင်နဲ့ချီပြီး ဖန်တီးလိုက်မယ်၊ အလုပ်ပြီးရင် ပြန်ဖျက်လိုက်မယ်၊ ပြီးတော့ ငါတို့ သုံးလိုက်တဲ့ အချိန်အတွက်ပဲ ပိုက်ဆံ ပေးရမှာလေ” တဲ့။

Wes က Brent ကို လှည့်ကြည့်တော့ Brent က— “ဖြစ်နိုင်ပါတယ်ဗျ။ ငါတို့က environment အများစုအတွက် virtualization (ကွန်ပျူတာဖြင့် ဖန်တီးထားသော စနစ်များ) ကို သုံးနေကြတာပဲ။ အဲဒါကို cloud computing provider (ကလောက်ဒ် ဝန်ဆောင်မှု ပေးသူ) ဆီမှာ run လို့ရအောင် ပြောင်းလဲဖို့ဆိုတာ သိပ်မခက်ခဲပါဘူး” ဆိုပါတယ်။

ခဏနေတော့ သူက ထပ်ဖြည့်ပြောတယ်— “ဒါကတော့ ပျော်စရာ ကောင်းမှာပဲ။ ဒါမျိုးကို ငါ အမြဲ စမ်းကြည့်ချင်နေတာ” တဲ့။

Brent ရဲ့ စိတ်လှုပ်ရှားမှုက လူတိုင်းဆီ ကူးစက်သွားပါတယ်။

ဒါကို တကယ် လုပ်လို့ရမလားဆိုတာ စူးစမ်းဖို့ အလုပ်တွေကို အဖွဲ့လိုက် ခွဲလိုက်ကြပါတယ်။ Brent က အဲဒီ စိတ်ကူးပေးတဲ့ developer နဲ့ ပူးပေါင်းပြီး အမှန်တကယ် ဖြစ်နိုင်ခြေ ရှိမရှိ သိရအောင် prototype (နမူနာ ပုံစံငယ်) တစ်ခုကို အမြန် တည်ဆောက်ပါတော့တယ်။

Maggie ကလည်း Unicorn အပေါ် စိတ်ဝင်စားလွန်းလို့ နေ့စဉ် stand-up အစည်းအဝေးတွေကိုတောင် ပုံမှန် တက်နေတာလေ၊ သူမက ဈေးနှုန်းတွေကို လေ့လာပေးဖို့နဲ့ လုပ်ငန်းတူ တခြားသူတွေဆီ ဖုန်းဆက်ပြီး ဒါမျိုး လုပ်ဖူးသူ ရှိမရှိ၊ ဘယ် vendors (ဝန်ဆောင်မှု ပေးသူ) တွေက ကောင်းသလဲဆိုတာ မေးမြန်းပေးဖို့ တာဝန်ယူလိုက်ပါတယ်။

John ရဲ့ security engineers (လုံခြုံရေး အင်ဂျင်နီယာ) တစ်ယောက်က ကြားဖြတ်ပြောတယ်— “ငါတို့ ဖောက်သည် ဒေတာတွေကို cloud ဆီ ပို့တာက အန္တရာယ် ရှိနိုင်တယ်နော်၊ ဥပမာ- ဒေတာတွေ ပေါက်ကြားတာ (disclosure of private data) ဒါမှမဟုတ် ခွင့်ပြုချက်မရှိဘဲ တစ်ယောက်ယောက်က အဲဒီ compute servers တွေကို hacking (ဟက်ကာ လုပ်ခြင်း) လုပ်တာမျိုးပေါ့” ဆိုတော့—

“စဉ်းစားတာ ကောင်းတယ်” လို့ ကျွန်တော်က ဆိုလိုက်တယ်— “မင်းတို့ စိုးရိမ်ရတဲ့ အဓိက အချက်တွေကို စာရင်းလုပ်ပေးပါဦး၊ ပြီးတော့ အဲဒါတွေကို ကာကွယ်ဖို့အတွက် counter-measures (တန်ပြန် ကာကွယ်မှုများ) နဲ့ controls (ထိန်းချုပ်မှုများ) ဘယ်လို လုပ်လို့ရမလဲဆိုတာပါ ပြင်ဆင်ပေးပါ” လို့ ပြောလိုက်ပါတယ်။

သူက အားရပါးရ ပြုံးပြပါတယ်၊ တစ်ယောက်ယောက်က သူ့ကို အကူအညီ တောင်းတာကို သူ ပျော်နေတာပေါ့။ developer တစ်ယောက်ကလည်း သူ့ကို ကူညီပေးဖို့ တာဝန်ယူလိုက်ပါတယ်။

အစည်းအဝေး အပြီးမှာတော့ ကျွန်တော်တို့ရဲ့ deployment process ကို automating (အလိုအလျောက် စနစ်) ဖြစ်အောင် လုပ်လိုက်တာဟာ မထင်မှတ်ထားတဲ့ အကျိုးကျေးဇူးတွေ ပေးနေတာကို တွေ့လိုက်ရလို့ အံ့သြမိပါတယ်။ developers တွေဟာ application ကို ပိုပြီး မြန်မြန်ဆန်ဆန် scale (ချဲ့ထွင်) လုပ်နိုင်လာသလို ငါတို့ဘက်ကလည်း အပြောင်းအလဲ သိပ်မလုပ်ပေးရတော့ဘူးလေ။

ဒါတောင်မှ ကျွန်တော်ကတော့ ဒီ cloud computing ဆိုတဲ့ ဟိုဟိုဒီဒီ အသံတွေကို သံသယ ရှိနေတုန်းပါပဲ။ လူတွေကတော့ ဒါဟာ ကုန်ကျစရိတ်တွေကို ချက်ချင်း လျှော့ချပေးမယ့် မှော်ဆေးတစ်ခွက်လို ပြောနေကြတာ။ ကျွန်တော့်အမြင်မှာတော့ ဒါဟာ outsourcing (အပြင်လူကို ခိုင်းခြင်း) ရဲ့ နောက်ထပ် ပုံစံတစ်မျိုးပဲလေ။

ဒါပေမဲ့ ဒါဟာ ငါတို့ ကြုံနေရတဲ့ ပြဿနာကို ဖြေရှင်းပေးနိုင်မယ် ဆိုရင်တော့ စမ်းကြည့်ဖို့ ကျွန်တော် ဝန်မလေးပါဘူး။ Wes ကိုလည်း စိတ်ကို ဖွင့်ထားဖို့ သတိပေးလိုက်ပါတယ်။

တစ်ပတ်အကြာမှာတော့ demo (လက်တွေ့ ပြသခြင်း) အချိန် ရောက်ပြန်ပါပြီ။ ငါတို့ အားလုံး Unicorn အဖွဲ့ရဲ့ နေရာမှာ စုဝေးနေကြတာပေါ့။ ဒါဟာ sprint ရဲ့ အဆုံးသတ်ပဲဖြစ်ပြီး Development ခေါင်းဆောင်က အဖွဲ့သားတွေ ဘာတွေ လုပ်ထားနိုင်သလဲဆိုတာကို ဂုဏ်ယူစွာနဲ့ ပြသချင်နေတာပါ။

“ငါတို့တွေ အများကြီး လုပ်နိုင်ခဲ့တာကို ငါတောင် မယုံနိုင်ဘူးဗျ” လို့ သူက စပြောတယ်— “deployment automation (အလိုအလျောက် ဖြန့်ကျက်မှု) တွေ လုပ်ထားတဲ့အတွက် cloud ထဲမှာ compute instances တွေ run ဖို့က ထင်သလောက် မခက်ခဲဘူးဗျာ။ တကယ်တော့ ဒါက တကယ့်ကို အလုပ်ဖြစ်လွန်းလို့ Unicorn ရဲ့ production systems တွေ အကုန်လုံးကို cloud ပေါ် တင်လိုက်ပြီး လက်ရှိ ကုမ္ပဏီထဲက စနစ်တွေကိုတော့ စမ်းသပ်တဲ့ test systems တွေအဖြစ်ပဲ သုံးဖို့တောင် စဉ်းစားနေကြတာ။

“ညတိုင်း အကြံပြုချက်တွေ တွက်ချက်တဲ့အခါ compute instances ရာနဲ့ချီပြီး ဖန်တီးလိုက်တယ်၊ ပြီးသွားရင် ပြန်ပိတ်လိုက်တယ်။ ဒါကို ပြီးခဲ့တဲ့ လေးရက်လုံးလုံး လုပ်ခဲ့တာ၊ တကယ့်ကို အဆင်ပြေတယ်ဗျ” တဲ့။

Brent ရဲ့ မျက်နှာမှာ အပြုံးကြီး တစ်ခု ရှိနေသလို အဖွဲ့သားတွေ အကုန်လုံးလည်း ပျော်နေကြတာပေါ့။

နောက်ထပ် ပြသရမယ့်သူက ပုံမှန်ဆိုရင် product manager ဖြစ်ပေမဲ့ ဒီတစ်ခါတော့ Maggie က ကိုယ်တိုင် တင်ပြနေပါတယ်။ သူမဟာ ဒီပရောဂျက်အပေါ်မှာ ပုံမှန်ထက် ပိုပြီး စိတ်ဝင်စားနေတာ သိသာပါတယ်။ သူမက ပရိုဂျက်တာ (projector) မှာ PowerPoint (ပါဝါပွိုင့်) စလိုက်တစ်ခုကို ပြလိုက်တယ်- “ဒါကတော့ ကျွန်မရဲ့ ဖောက်သည် အကောင့်အတွက် Unicorn ကနေ ထုတ်ပေးတဲ့ အရောင်းမြှင့်တင်ရေးတွေပါ။ တွေ့တဲ့အတိုင်းပဲ… ကျွန်မရဲ့ ဝယ်ယူမှု မှတ်တမ်းကို ကြည့်ပြီး နှင်းတောထဲ မောင်းတဲ့ တာယာတွေ (snow tires) နဲ့ ဘက်ထရီတွေကို ၁၅ ရာခိုင်နှုန်း လျှော့ပေးထားတာပေါ့။ ကျွန်မ တကယ်လည်း လိုအပ်နေတာမို့ ဝက်ဘ်ဆိုက်ကနေ ချက်ချင်းပဲ ဝယ်လိုက်ပါတယ်။ ကုမ္ပဏီကတော့ ပိုက်ဆံ ရသွားတာပေါ့၊ ဘာလို့လဲဆိုတော့ အဲဒီ ပစ္စည်းတွေက ငါတို့ဆီမှာ အပိုတွေ ရှိနေသလို အမြတ်လည်း အများကြီး ရတဲ့ ပစ္စည်းတွေ (high profit margins) မဟုတ်လား” တဲ့။

ကျွန်တော် ပြုံးမိပါတယ်။ ဒါဟာ တကယ့်ကို တော်တဲ့ အလုပ်ပဲ။

“ကဲ… ဒါကတော့ Wes အတွက် Unicorn ကမ်းလှမ်းချက်တွေပေါ့” လို့ သူမက နောက်တစ်ချက်ကို ပြုံးပြီး ပြပြန်တယ်- “ကြည့်ရတာ ရှင့်အတွက် ပြိုင်ကားသုံး ဘရိတ်ပြားတွေ (racing brake pads) နဲ့ လောင်စာဆီ ဖြည့်စွက်စာတွေ (fuel additives) ကို လျှော့ဈေး ပေးထားတယ်နော်။ စိတ်ဝင်စားစရာပဲ မဟုတ်လား” တဲ့။

Wes က ပြုံးပြီး— “မဆိုးပါဘူး!” ဆိုပါတယ်။

Maggie က ရှင်းပြပါတယ်—ဒီ ကမ်းလှမ်းချက်တွေ အကုန်လုံးဟာ Phoenix စနစ်ထဲမှာ ရှိနေပြီးသားပါ၊ အဲဒါကို ဖောက်သည်တွေဆီ ပို့ပေးဖို့အတွက် အရောင်းမြှင့်တင်ရေး အင်္ဂါရပ် (promotion functionality) ကိုပဲ စောင့်နေခဲ့ရတာလေ။

သူမက ဆက်ပြောတယ်— “ကျွန်မရဲ့ အဆိုပြုချက်ကတော့—ငါတို့ရဲ့ ဖောက်သည် အားလုံးထဲက ၁ ရာခိုင်နှုန်းဆီကို အီးမေးလ် ပို့ပြီး ဘာဖြစ်မလဲဆိုတာ စမ်းကြည့်ချင်တယ်။ Thanksgiving (ကျေးဇူးတော်နေ့) က နောက်တစ်ပတ်မှာလေ။ အကယ်၍ တစ်ခါနှစ်ခါ စမ်းကြည့်လို့ အဆင်ပြေတယ် ဆိုရင်တော့—တစ်နှစ်ပတ်လုံးမှာ ဈေးဝယ်အများဆုံးနေ့ဖြစ်တဲ့ Black Friday (ဘလက် ဖရိုက်ဒေး) မှာ အစွမ်းကုန် လုပ်ကြတာပေါ့” တဲ့။

“ကောင်းတဲ့ အစီအစဉ်ပဲ” လို့ ကျွန်တော်က ဆိုလိုက်တယ်- “Wes… ဒါကို မလုပ်သင့်တဲ့ အကြောင်းရင်း တစ်ခုခု ရှိမလား” ဆိုတော့—

Wes က ခေါင်းခါပြတယ်— “Ops (လည်ပတ်မှု အဖွဲ့) ဘက်ကနေ ကြည့်ရင်တော့ ဘာမှ မရှိပါဘူး။ ခက်ခဲတဲ့ အလုပ်တွေ အကုန် ပြီးသွားပြီလေ။ Chris, William နဲ့ Marketing တို့က ကုဒ်က အလုပ်လုပ်တယ်လို့ ယုံကြည်ရင်တော့ လုပ်ကြတာပေါ့” တဲ့။

လူတိုင်း သဘောတူကြပါတယ်။ ပြဿနာတချို့ ပေါ်လာပေမဲ့ Maggie ကတော့ သူမရဲ့ အဖွဲ့ဟာ ဒါကို ဖြစ်မြောက်အောင် တညလုံး အလုပ်လုပ်ဖို့ အဆင်သင့်ပဲလို့ ဆိုပါတယ်။

ကျွန်တော် စိတ်ထဲကနေ ပြုံးမိပါတယ်။ တစ်ခုခု မှားယွင်းသွားလို့ တညလုံး မအိပ်ဘဲ နေရတာမျိုး မဟုတ်ဘဲ—တကယ်တော့ ဒါဟာ အဲဒါနဲ့ လုံးဝ ဆန့်ကျင်ဘက်လေ။ အရာရာ အဆင်ပြေနေလို့ လူတွေက မအိပ်ဘဲ အလုပ်လုပ်နေကြတာကိုး။

နောက်တစ်ပတ် တနင်္လာနေ့မှာတော့ အလုပ်သွားတုန်း လေက အရမ်း အေးနေပေမဲ့ နေကတော့ တောက်ပစွာ သာနေပါတယ်။ နောက်လာမယ့် Thanksgiving အားလပ်ရက်အတွက် အပတ်ကောင်းတစ်ခု ဖြစ်လိမ့်မယ်လို့ ထင်ရတာပဲ။ ပိတ်ရက်တစ်လျှောက်လုံး စန်တာကလော့စ် ပုံတွေနဲ့ ကြော်ငြာတွေကို တွေ့လိုက်ရလို့ တော်တော်လေး အံ့သြမိပါတယ်။

ရုံးခန်းရောက်တော့ အနွေးထည် အထူကြီးကို ကုလားထိုင်ပေါ် တင်လိုက်ပါတယ်။ Patty က ကျွန်တော့် ရုံးခန်းထဲ ပြုံးပြုံးကြီး ဝင်လာတာကို တွေ့လိုက်ရတယ်— “Marketing ဆီက အံ့သြစရာ သတင်းကို ကြားပြီးပြီလား” တဲ့။

ကျွန်တော် ခေါင်းခါပြလိုက်တော့ သူမက— “Maggie အခုလေးတင် ပို့လိုက်တဲ့ အီးမေးလ်ကို ဖတ်ကြည့်လိုက်ဦး” လို့ ဆိုပါတယ်။

ကျွန်တော် လက်ပ်တော့ကို ဖွင့်ပြီး ဖတ်ကြည့်လိုက်တယ်—

ရက်စွဲ- နိုဝင်ဘာ ၂၄၊ နံနက် ၇:၄၇ နာရီ
အကြောင်းအရာ- ပထမဆုံး Unicorn promotion campaign- မယုံနိုင်စရာပဲ!

Marketing အဖွဲ့ဟာ ပိတ်ရက်မှာ တညလုံး အပင်ပန်းခံ လုပ်ခဲ့ကြပြီး ငါတို့ရဲ့ ဖောက်သည် ၁ ရာခိုင်နှုန်းဆီကို စမ်းသပ်မှု လုပ်နိုင်ခဲ့ပါတယ်။

ရလဒ်တွေကတော့ တကယ့်ကို အံ့မခန်း (STELLAR) ပါပဲ! လက်ခံရရှိသူတွေထဲက ၂၀ ရာခိုင်နှုန်းကျော်က ငါတို့ ဝက်ဘ်ဆိုက်ကို လာကြည့်ကြပြီး ၆ ရာခိုင်နှုန်းကျော်ကတော့ ပစ္စည်း ဝယ်ကြပါတယ်။ ဒါဟာ တကယ့်ကို မြင့်မားတဲ့ conversion rates (ရောင်းအား ပြောင်းလဲမှုနှုန်း) ပါပဲ—အရင်က လုပ်ခဲ့သမျှ ပရောဂျက်တွေထက် အနည်းဆုံး ၅ ဆလောက် ပိုများနေတာပါ။

Thanksgiving နေ့မှာ ဖောက်သည် အားလုံးဆီကို Unicorn promotion တွေ ပို့ဖို့ ကျွန်မတို့ အကြံပြုချင်ပါတယ်။ Unicorn campaigns တွေရဲ့ ရလဒ်ကို အချိန်နဲ့တပြေးညီ (real-time) မြင်နိုင်ဖို့ dashboard (အချက်အလက်ပြ ဘုတ်) တစ်ခုကိုလည်း ကျွန်မ ပြင်ဆင်နေပါတယ်။

ဒါ့အပြင် အခု ရောင်းပေးနေတဲ့ ပစ္စည်းတွေက အမြတ်များတဲ့ ပစ္စည်းတွေမို့ ကုမ္ပဏီရဲ့ အမြတ်အစွန်း (bottom line) အတွက် တကယ့်ကို ကောင်းမွန်ပါလိမ့်မယ်။

PS: Bill ရေ… ရလဒ်တွေအရတော့ ဝက်ဘ်ဆိုက်ကို လာကြည့်သူတွေ (web traffic) အများကြီး တက်လာဖို့ ရှိပါတယ်။ ဝက်ဘ်ဆိုက်ကြီး ကျမသွားအောင် (fall over) ပြန်စစ်ပေးလို့ ရမလား။

အားလုံးပဲ တော်ကြပါတယ်!
Maggie

“သဘောကျတယ်ဗျာ” လို့ Patty ကို ပြောလိုက်တယ်— “Wes နဲ့ ပူးပေါင်းပြီး traffic တက်လာတာကို ကိုင်တွယ်ဖို့ ဘာလုပ်ရမလဲဆိုတာ ရှာဖွေလိုက်ပါ။ ငါတို့မှာ ၃ ရက်ပဲ အချိန်ရှိတော့တာဆိုတော့ သိပ်မကျန်တော့ဘူး။ ဒါကို အမှားမခံနိုင်ဘူး၊ ဝယ်ချင်တဲ့သူတွေကို စိတ်ပျက်အောင် အလုပ် မခံနိုင်ဘူးလေ” ဆိုတော့—

သူမက ခေါင်းညိတ်ပြီး ပြန်ဖြေမလို့ ရှိသေးတယ်၊ သူမရဲ့ ဖုန်း တုန်ခါလာပါတယ်။ ခဏနေတော့ ကျွန်တော့်ဖုန်းလည်း တုန်လာပြန်တာပေါ့။ သူမက ဖုန်းကို ကြည့်လိုက်ပြီး— “နဂါးမကြီးကတော့ နောက်တစ်ခါ လာပြန်ပြီ” လို့ ဆိုပါတယ်။

“သူ့ အီးမေးလ်တွေအတွက် ‘unsubscribe’ (စာရင်းဖျက်ခြင်း) ခလုတ်များ ရှိရင် ကောင်းမှာပဲ” လို့ Patty က ရေရွတ်ရင်း ထွက်သွားပါတယ်။

နာရီဝက်လောက် ကြာတော့ Steve က Unicorn အဖွဲ့သား အားလုံးကို ဂုဏ်ပြုတဲ့ စာတစ်စောင် ပို့ပါတယ်၊ အဲဒါကို လူတိုင်းက ဖတ်ပြီး ပျော်နေကြတာပေါ့။ ပိုပြီး အံ့သြစရာကောင်းတာက—သူက Sarah ဆီကိုလည်း လူသိရှင်ကြား ပြန်စာ ပို့လိုက်ပြီး “လူတွေကို လိုက်ပြီး ဆူအောင် မလုပ်ဖို့နဲ့ ပြဿနာ မရှာဖို့”၊ ပြီးတော့ “အားတဲ့ အချိန်မှာ ငါ့ဆီ လာတွေ့ဖို့” တောင်းဆိုလိုက်တာပါပဲ။

ဒါတောင်မှ Sarah, Steve နဲ့ Bob တို့ကြားမှာ လူသိရှင်ကြား အီးမေးလ်တွေ အပြန်အလှန် ပို့နေတာကတော့ မရပ်သေးပါဘူး။ Sarah တစ်ယောက် ငါတို့ရဲ့ ဥက္ကဋ္ဌသစ် Bob ဆီမှာ ဖားနေတာ (toadying up) ကို မြင်ရတာ တော်တော်လေးကို ကြည့်ရဆိုးပြီး အနေရခက်လှပါတယ်။ Sarah က သူမ ဘာတွေ လုပ်နေသလဲဆိုတာ လူတွေ အကုန်လုံး သိနေတာကိုတောင် ဂရုမစိုက်တော့သလို အရာရာကိုလည်း ဖျက်ဆီးပစ်နေတာပေါ့။

ကျွန်တော် SOX-404 နဲ့ Unicorn security issues (လုံခြုံရေး ပြဿနာများ) ဖြေရှင်းပြီးစီးတဲ့ ကိစ္စအတွက် John နဲ့ တွေ့ဖို့ အစည်းအဝေးခန်းထဲ ဝင်လာပါတယ်။ သူက အကွက်စိတ် Oxford အင်္ကျီနဲ့ လက်ပြတ် အင်္ကျီ (vest) ကို ဝတ်ထားပြီး လက်စွပ် (cufflinks) တွေနဲ့ တကယ့်ကို သေသပ်နေတာပေါ့။ ကြည့်ရတာ Vanity Fair (ဗနီတီး ဖယ်ရာ) မဂ္ဂဇင်းအတွက် ဓာတ်ပုံရိုက်ပြီး ထွက်လာတဲ့အတိုင်းပဲ၊ သူက ခေါင်းကိုလည်း နေ့တိုင်း တုံးနေတုန်းပေါ့။

“Unicorn ရဲ့ လုံခြုံရေး ပြင်ဆင်ချက်တွေ အဲဒီလောက် မြန်မြန် ပေါင်းစပ်နိုင်တာကို ကျွန်တော် တကယ်ပဲ အံ့သြမိတယ်ဗျာ” တဲ့။ “Phoenix နဲ့ နှိုင်းယှဉ်ရင် Unicorn ရဲ့ လုံခြုံရေး ပြဿနာတွေကို ပြင်ရတာဟာ လေညင်းခံရသလိုပဲ။ cycle time (သံသရာ ကြာမြင့်ချိန်) က အရမ်းတိုတော့ တစ်ခါတုန်းကဆိုရင် တစ်နာရီအတွင်းမှာတောင် အမှားပြင်နိုင်ခဲ့သေးတယ်။ ပုံမှန်ဆိုရင်လည်း တစ်ရက် နှစ်ရက်အတွင်း ပြီးတာပဲလေ။ ဒါနဲ့ နှိုင်းယှဉ်ရင် Phoenix ပြဿနာတွေကို ပြန်ပြင်ရတာ (remediating) ဟာ မေ့ဆေး (anesthesia) မပါဘဲ ကိုယ့်သွားကိုယ် ပြန်နှုတ်နေရသလိုပဲဗျာ။ ပုံမှန်ဆိုရင်တော့ အပြောင်းအလဲ တစ်ခုခု ဖြစ်ဖို့ဆိုတာ ၃ လလောက် စောင့်ရသလို အရေးပေါ် အပြောင်းအလဲ (emergency change order) တစ်ခု ရဖို့အတွက်ဆိုရင်လည်း အဆင့်ဆင့် ဖြတ်ကျော်နေရတာနဲ့တင် အလုပ်က လုပ်ရကျိုး မနပ်တော့သလိုပဲ။

“တကယ်တော့” လို့ သူက ဆက်ပြောတယ်— “patching (အမှားဖာထေးခြင်း) လုပ်ရတာ အရမ်း လွယ်ကူသွားတယ်။ ဘာလို့လဲဆိုတော့ production ထဲမှာ ရှိသမျှ အရာရာတိုင်းကို ခလုတ်တစ်ချက် နှိပ်ရုံနဲ့ ပြန်တည်ဆောက် (rebuilding) လို့ ရနေပြီလေ။ အကယ်၍ ပျက်သွားပြီ ဆိုရင်လည်း အသစ်ကနေ ပြန်ပြီး တည်ဆောက်လိုက်ရုံပဲ” တဲ့။

ကျွန်တော် ခေါင်းညိတ်လိုက်တယ်- “Unicorn ရဲ့ မြန်ဆန်တဲ့ cycle times တွေနဲ့ ငါတို့ ဘာတွေ လုပ်နိုင်သလဲဆိုတာ ကျွန်တော်လည်း အံ့သြနေမိတာပဲဗျ။ Phoenix တုန်းကဆိုရင် deployment လုပ်ဖို့အတွက် ၃ လမှ တစ်ခါပဲ လေ့ကျင့်နိုင်ခဲ့ကြတာ။ အခု ၅ ပတ်အတွင်းမှာတင် Unicorn ကုဒ်နဲ့ environment deployments ပေါင်း ၂၀ ကျော် လုပ်ခဲ့ပြီးပြီ။ ဒါဟာ အခုဆို ပုံမှန် အလုပ်တစ်ခုလို ဖြစ်နေပြီလေ။ ခင်ဗျား ပြောသလိုပဲ… Phoenix နဲ့ လုံးဝ ဆန့်ကျင်ဘက်ပေါ့” တဲ့။

John က— “Unicorn အပေါ်မှာ ကျွန်တော် ထားခဲ့တဲ့ စိုးရိမ်မှုတွေက အခုတော့ အချည်းနှီး ဖြစ်သွားပါပြီဗျာ။ production ကို နေ့တိုင်း ဝင်နိုင်တဲ့ developers တွေဟာ read-only (ဖတ်ခွင့်သက်သက်) ပဲ ရှိနေသလားဆိုတာကို ပုံမှန် ပြန်စစ်နေသလို ငါတို့ရဲ့ security tests (လုံခြုံရေး စမ်းသပ်ချက်များ) တွေကိုလည်း build procedures တွေထဲမှာ ပေါင်းစပ်ထည့်သွင်းဖို့ တိုးတက်နေပြီလေ။ ဒေတာ လုံခြုံရေး ဒါမှမဟုတ် authentication modules (စနစ်ဝင်ရောက်ခွင့် စိစစ်မှု) တွေကို ထိခိုက်စေမယ့် ဘယ်လို အပြောင်းအလဲမျိုးမဆို ချက်ချင်း ဖမ်းမိနိုင်မယ်လို့ ကျွန်တော် ယုံကြည်တယ်” လို့ ဆိုပါတယ်။

သူက ကုလားထိုင်ပေါ် မှီလိုက်ရင်း လက်နှစ်ဖက်ကို ခေါင်းနောက်မှာ ထားလိုက်တယ်- “အရင်ကတော့ Unicorn ရဲ့ လုံခြုံရေးကို ဘယ်လိုမျိုး အာမခံချက် (assurance) ရအောင် လုပ်မလဲဆိုတာ ကျွန်တော် တကယ်ကို ထိတ်လန့်ခဲ့တာဗျ။ ဘာလို့လဲဆိုတော့ application security review (အပလီကေးရှင်း လုံခြုံရေး စစ်ဆေးမှု) တစ်ခု လုပ်ဖို့ တစ်လလောက် အချိန်ယူနေကျ ဖြစ်နေလို့ပေါ့။ အရေးပေါ် အနေနဲ့ audit (စာရင်းစစ်) တွေ ရှိလာခဲ့ရင်တောင်မှ အမြန်ဆုံး တစ်ပတ်လောက် ကြာတာလေ။

“ဒါပေမဲ့ တစ်နေ့ကို deploys ၁၀ ကြိမ် လုပ်မယ့် နှုန်းကို လိုက်မှီဖို့ ဆိုတာကတော့…” သူက ဆက်ပြောတယ်— “အရူးအလုပ်လို့ပဲ ထင်ခဲ့တာ! ဒါပေမဲ့ ငါတို့ရဲ့ လုံခြုံရေး စမ်းသပ်မှုတွေကို အလိုအလျောက် ဖြစ်အောင် (automate) လုပ်ဖို့ အတင်း အကျပ် ခိုင်းခံရပြီး William ရဲ့ အလိုအလျောက် QA testing တွေထဲမှာ ပေါင်းစပ်လိုက်ပြီးတဲ့ နောက်မှာတော့—developer တစ်ယောက် ကုဒ် commit လုပ်တိုင်း ငါတို့တွေ စမ်းသပ်နိုင်သွားပြီဗျ။ နည်းလမ်းတော်တော်များများမှာတော့ အခုဆိုရင် တခြား application တွေထက်စာရင် ကုဒ်တွေကို ပိုပြီး မြင်နေရသလို ပိုပြီးတော့လည်း လုံခြုံရေး စစ်ဆေးနိုင်စွမ်း (code coverage) ရှိနေပြီလေ!” တဲ့။

သူက ထပ်ဖြည့်ပြောတယ်— “ဆရာ သိထားရမှာက ငါတို့ရဲ့ SOX-404 ပြဿနာတွေ အကုန်လုံးကို အခုပဲ ပိတ်လိုက်နိုင်ပြီဗျ။ ဆရာ ချမှတ်ပေးခဲ့တဲ့ change control processes (အပြောင်းအလဲ ထိန်းချုပ်မှု လုပ်ငန်းစဉ်) အသစ်တွေကြောင့်—လက်ရှိ ထိန်းချုပ်မှုတွေဟာ လုံလောက်ပါတယ်ဆိုတာကို စာရင်းစစ် (auditors) တွေကို သက်သေပြနိုင်ခဲ့တယ်၊ ဒါကြောင့် ၃ နှစ်တာ ရင်ဆိုင်ခဲ့ရတဲ့ audit finding (စာရင်းစစ် ကန့်ကွက်ချက်) ကို အခုပဲ အဆုံးသတ်လိုက်နိုင်ပါပြီ” တဲ့။

သူက ပြုံးပြီး— “ဂုဏ်ယူပါတယ် Bill။ အရင်က ဘယ်သူမှ မလုပ်နိုင်ခဲ့တာကို ဆရာ လုပ်နိုင်ခဲ့ပါပြီ၊ အဲဒါကတော့ စာရင်းစစ်တွေကို ငါတို့ နောက်ကနေ အပြီးအပိုင် ဖယ်ရှားလိုက်တာပါပဲ!” တဲ့။

ကျွန်တော် အံ့သြသွားတာကတော့ ဒီ အပတ်တိုလေးဟာ တကယ့်ကို အဆင်ပြေပြေ ကုန်ဆုံးသွားခဲ့တာပါပဲ။ ဗုဒ္ဓဟူးနေ့မှာ အားလုံး Thanksgiving အားလပ်ရက်အတွက် မပြန်ခင်မှာ Unicorn campaign အကြီးကြီးကတော့ အဆင်သင့် ဖြစ်နေပါပြီ။ ကုဒ်ရဲ့ စွမ်းဆောင်ရည်က လိုအပ်တာထက် ၁၀ ဆလောက် နှေးနေတုန်းပဲ ဆိုပေမဲ့—cloud ပေါ်မှာ compute instances ရာနဲ့ချီပြီး ဖန်တီးနိုင်တဲ့အတွက် အခုလောလောဆယ်တော့ အဆင်ပြေနေတာပေါ့။

QA ဘက်ကနေ ပစ္စည်းပြတ်လပ်နေတာတွေကိုပါ (out of stock) လျှော့ဈေးနဲ့ အကြံပေးနေမိတာကို ရှာတွေ့ခဲ့တုန်းကတော့ တကယ့်ကို ရပ်ဆိုင်းလုနီးပါး ဖြစ်သွားခဲ့သေးတယ်။ အဲဒါသာ တကယ် ဖြစ်သွားရင်တော့—ဖောက်သည်တွေက ဝမ်းသာအားရနဲ့ နှိပ်လိုက်ပေမဲ့ “backordered” (ပစ္စည်း ပြန်မှာထားသည်) လို့ ရေးထားတာကို တွေ့ရရင် တကယ့်ကို ကပ်ဘေးကြီး ဖြစ်မှာလေ။ အံ့သြစရာကောင်းတာက Development က တစ်ရက်အတွင်းမှာတင် အမှားပြင်နိုင်ခဲ့သလို တစ်နာရီအတွင်းမှာပဲ deploy လုပ်နိုင်ခဲ့ပါတယ်။

ညနေ ၆ နာရီမှာတော့ ကျွန်တော့် ပစ္စည်းတွေကို သိမ်းလိုက်ပြီး ရှည်လျားတဲ့ ပိတ်ရက်ကြီးကို မျှော်လင့်နေမိပါတော့တယ်။ ငါတို့ အားလုံးက ဒါနဲ့ ထိုက်တန်ကြတာကိုး။

အခန်း ၂၃ (CHAPTER 23)

• အင်္ဂါနေ့၊ အောက်တိုဘာ ၇ ရက်

အောက်တိုဘာလရဲ့ ဒုတိယအပတ် အင်္ဂါနေ့ မနက်မှာ ရုံးကို ကားမောင်းလာတုန်း Kirsten ဆီက အရေးပေါ် ဖုန်းခေါ်ဆိုမှုတစ်ခု ရပါတယ်။ Brent ဟာ နောက်ထပ် Phoenix task (ဖီးနစ် အလုပ်) တစ်ခုကို အချိန်မီ မပြီးဘဲ တစ်ပတ်လောက် နောက်ကျနေပြီတဲ့။ အဲဒီအလုပ်ဟာ Brent ပြောပုံအရဆိုရင် တစ်နာရီလောက်ပဲ ကြာမယ့်အလုပ်ပါ၊ အခုတော့ Phoenix testing schedule (စမ်းသပ်မှု အချိန်ဇယား) တစ်ခုလုံး အန္တရာယ် ဖြစ်နေပြန်ပါပြီ။

ဒါ့အပြင် ကျွန်တော့်အဖွဲ့ရဲ့ တခြား အရေးကြီးတဲ့ အလုပ်တချို့လည်း နောက်ကျနေလို့ သတ်မှတ်ရက်အတွက် ဖိအားတွေ ပိုများလာပါတယ်။ ဒါဟာ တကယ့်ကို စိတ်ပျက်စရာပါပဲ။ ပြီးခဲ့တဲ့ အောင်မြင်မှုတွေက ဒီလို သတ်မှတ်ရက် မမီတဲ့ ပြဿနာတွေကို ဖြေရှင်းပေးနိုင်လိမ့်မယ်လို့ ကျွန်တော် ထင်ခဲ့တာ။ အခု အလုပ်တွေကိုတောင် အချိန်မီ မလုပ်နိုင်ဘဲနဲ့ နောက်ထပ် အလုပ်တွေကို ဘယ်လိုများ ပြန်ဖွင့်ပေးနိုင်ပါ့မလဲ။

ကျွန်တော် Patty ဆီကို အသံမက်ဆေ့ခ်ျ တစ်ခု ချန်ထားခဲ့ပါတယ်။ အံ့သြစရာကောင်းတာက သူမဆီက ဖုန်းပြန်လာဖို့ ၃ နာရီလောက် ကြာသွားခဲ့တာပါ။ သူမကတော့ ကျွန်တော်တို့ရဲ့ အချိန်ဇယား ခန့်မှန်းချက်တွေမှာ တစ်ခုခု အကြီးအကျယ် လွဲချော်နေပြီဖြစ်ကြောင်းနဲ့ ချက်ချင်း တွေ့ဖို့ လိုတယ်လို့ ပြောပါတယ်။

နောက်တစ်ကြိမ် အစည်းအဝေးခန်းထဲ ရောက်ပြန်ပါပြီ။ Patty က whiteboard ရှေ့မှာ ရှိနေပြီး Wes ကတော့ သူမ ကပ်ထားတဲ့ စာရွက်တွေကို သေသေချာချာ ကြည့်နေတာပေါ့။

“အခုထိ ကျွန်မ သိထားတာကတော့ ဒါပါပဲ” လို့ Patty က စာရွက်တစ်ရွက်ကို ညွှန်ပြပြီး ပြောတယ်— “Kirsten ဖုန်းဆက်ပြီး ပြောတဲ့ အလုပ်က QA (အရည်အသွေး စစ်ဆေးရေး အဖွဲ့) ဆီကို test environment (စမ်းသပ်မှု ပတ်ဝန်းကျင်) တစ်ခု ပေးပို့ဖို့ပါ။ သူမ ပြောသလိုပဲ Brent က ဒါဟာ ၄၅ မိနစ်ပဲ ကြာမယ်လို့ ခန့်မှန်းခဲ့တာ” တဲ့။

“အဲဒါကတော့ ဟုတ်ပါတယ်” လို့ Wes က ဆိုတယ်— “virtualized server အသစ်တစ်ခု ဆောက်မယ်၊ ပြီးရင် OS နဲ့ လိုအပ်တဲ့ packages တချို့ ထည့်လိုက်ရုံပဲလေ။ သူကတော့ စိတ်ချရအောင် အချိန်ကို နှစ်ဆလောက်တောင် ပိုပြီး ခန့်မှန်းထားတာ ဖြစ်မှာပါ” ဆိုတော့—

“ကျွန်မလည်း အစကတော့ အဲဒီလိုပဲ ထင်ခဲ့တာပါ” လို့ Patty က ခေါင်းယမ်းရင်း ပြောတယ်— “ဒါပေမဲ့ ဒါဟာ အလုပ်တစ်ခုတည်း မဟုတ်ဘူးဗျ။ Brent လက်ခံလိုက်တဲ့ အလုပ်ဟာ တကယ်တမ်းကျတော့ ပရောဂျက် သေးသေးလေး တစ်ခုလိုပဲ—အနည်းဆုံး အဖွဲ့ ၆ ဖွဲ့လောက် ပါဝင်နေတဲ့ အဆင့်ပေါင်း ၂၀ ကျော် ရှိတာလေ! OS နဲ့ ဆော့ဖ်ဝဲလ် packages တွေအပြင် license keys တွေ၊ သီးသန့် IP address တွေ၊ အထူး user အကောင့်တွေ၊ mount points တွေ ပြုလုပ်ရသလို IP addresses တွေကိုလည်း file server ပေါ်က ACL list ထဲမှာ သွားထည့်ရသေးတယ်။ ဒီ အလုပ်မှာဆိုရင် physical server တစ်ခု လိုအပ်တယ်လို့ သတ်မှတ်ထားတော့ router port တွေ၊ ကြိုးသွယ်တာတွေနဲ့ နေရာလွတ်ရှိတဲ့ server rack တစ်ခုပါ လိုအပ်နေတာပေါ့” တဲ့။

“အော်…” လို့ Wes က စိတ်ပျက်လက်ပျက်နဲ့ ညည်းတွားလိုက်မိတယ်— “physical servers တွေကတော့ တကယ့်ကို ဒုက္ခပေးတာပဲဗျာ။”

“ရှင်က အဓိက အချက်ကို လွဲနေပြီ။ ဒါဟာ virtualized ဖြစ်နေရင်တောင်မှ ကြုံရဦးမှာပဲ” လို့ Patty က ဆိုပါတယ်— “ပထမအချက်ကတော့ Brent ရဲ့ ‘task’ (အလုပ်) ဆိုတာဟာ အလုပ်တစ်ခုတည်း ထက် အများကြီး ပိုများနေတာပဲ။ ဒုတိယအချက်ကတော့ ဒါဟာ လူအများကြီး လွှဲပြောင်းလုပ်ကိုင်ရတဲ့ အလုပ်တွေ ဖြစ်နေပြီး အဲဒီလူတိုင်းမှာလည်း သူတို့ရဲ့ အရေးပေါ် အလုပ်တွေ ကိုယ်စီ ရှိနေတာ။ တစ်ယောက်ဆီကနေ တစ်ယောက်ဆီ အလုပ်လွှဲပြောင်းတဲ့အခါ (handoff) မှာ ရက်တွေ အများကြီး ကုန်ဆုံးသွားရတယ်။ ဒီအတိုင်းသာဆိုရင်တော့ QA ဘက်က လိုချင်တာ ရဖို့အတွက် ရက်သတ္တပတ်တွေ အများကြီး ကြာသွားလိမ့်မယ်” တဲ့။

“အနည်းဆုံးတော့ firewall (လုံခြုံရေးတံတိုင်း) အပြောင်းအလဲတော့ မလိုဘူးပေါ့” လို့ Wes က ခနဲ့တဲ့တဲ့ ပြောလိုက်တယ်— “ပြီးခဲ့တဲ့ တစ်ခါကဆိုရင် John ရဲ့ အဖွဲ့ဆီက firewall change တစ်ခု ရဖို့ တစ်လနီးပါး ကြာခဲ့တာ။ စက္ကန့် ၃၀ လောက်ပဲ ကြာမယ့် အလုပ်လေးတစ်ခုအတွက် ၄ ပတ်တောင် စောင့်ခဲ့ရတာလေ!” ဆိုတော့—

Wes ပြောတာကို ကျွန်တော် ခေါင်းညိတ် ထောက်ခံလိုက်မိတယ်။ firewall changes တွေအတွက် ကြာမြင့်ချိန်ကတော့ တကယ့်ကို နာမည်ကြီးလှတာကိုး။

နေပါဦး… Erik ဒါမျိုး ပြောခဲ့တာ မဟုတ်လား။ firewall change တစ်ခုအတွက် တကယ့်ကို အလုပ်လုပ်တဲ့ အချိန် (touch time) က စက္ကန့် ၃၀ ပဲ ရှိပေမဲ့ စောင့်ရတဲ့ အချိန် (clock time) က ၄ ပတ်တောင် ကြာတယ်ဆိုတာလေ။

ဒါဟာ Brent ဆီမှာ ဖြစ်နေတဲ့ အရာရဲ့ ပုံစံငယ်လေးပါပဲ။ ဒါပေမဲ့ အခု ငါတို့ ကြုံနေရတာက အများကြီး ပိုဆိုးပါတယ်။ ဘာလို့လဲဆိုတော့ အလုပ်လွှဲပြောင်းမှုတွေ (handoffs) ရှိနေလို့ပေါ့။

ကျွန်တော် စိတ်ပျက်စွာနဲ့ အစည်းအဝေး စားပွဲပေါ် ခေါင်းမှောက်ချလိုက်မိတယ်။

“ဆရာ… အဆင်ပြေရဲ့လား” လို့ Patty က မေးပါတယ်။

“ခဏလေး…” လို့ ပြောပြီး ကျွန်တော် whiteboard ဆီ လမ်းလျှောက်သွားကာ marker တစ်ခုနဲ့ ဂရပ်ဖ်တစ်ခု ဆွဲဖို့ ကြိုးစားလိုက်တယ်။ အကြိမ်ကြိမ် ကြိုးစားပြီးတဲ့နောက်မှာတော့ ဒီလိုပုံစံ ဂရပ်ဖ်တစ်ခု ထွက်လာတာပေါ့—

MRP-8 မှာ Erik ပြောပြခဲ့တဲ့ အရင်းအမြစ် အသုံးပြုမှု (resource utilization) ပေါ် မူတည်ပြီး စောင့်ဆိုင်းရတဲ့ အချိန် (wait times) တွေ ဘယ်လို ကွာခြားသွားသလဲဆိုတာကို သူတို့ကို ရှင်းပြလိုက်ပါတယ်။ “စောင့်ဆိုင်းရတဲ့ အချိန်ဆိုတာကတော့ ‘အလုပ်များနေတဲ့ အချိန် ရာခိုင်နှုန်း’ ကို ‘အားနေတဲ့ အချိန် ရာခိုင်နှုန်း’ နဲ့ စားတာပါပဲ။ တစ်နည်းအားဖြင့် ပြောရရင် အရင်းအမြစ် တစ်ခုက ၅၀ ရာခိုင်နှုန်း အလုပ်များနေရင် ၅၀ ရာခိုင်နှုန်း အားနေတာပေါ့။ အဲဒီတော့ စောင့်ဆိုင်းချိန်က ၅၀ ကို ၅၀ နဲ့ စားတော့ ၁ ယူနစ်ပေါ့။ ငါတို့ ဒါကို တစ်နာရီလို့ပဲ သတ်မှတ်လိုက်ကြရအောင်။

“အဲဒီတော့ ပျမ်းမျှအားဖြင့် အလုပ်တစ်ခုဟာ အလုပ်မစခင် တန်းစီဇယား (queue) ထဲမှာ တစ်နာရီလောက် စောင့်ရမယ်ပေါ့။

“ဒါပေမဲ့ အရင်းအမြစ်တစ်ခုက ၉၀ ရာခိုင်နှုန်းလောက် အလုပ်များနေရင်တော့ စောင့်ဆိုင်းချိန်က ‘၉၀ ကို ၁၀ နဲ့ စားတာ’ ဖြစ်တဲ့အတွက် ၉ နာရီ ကြာသွားမယ်။ တစ်နည်းအားဖြင့် ပြောရရင် အရင်းအမြစ်တစ်ခုက ၅၀ ရာခိုင်နှုန်း အားနေတုန်းကထက် ၉ ဆ ပိုပြီး စောင့်ရမယ်ပေါ့။”

ကျွန်တော် ဆက်ပြောလိုက်တယ်— “အဲဒီတော့ Phoenix အလုပ်အတွက် အလုပ်လွှဲပြောင်းမှု ၇ ကြိမ် ရှိမယ်၊ အဲဒီ ဝန်ထမ်းတွေတိုင်းက ၉၀ ရာခိုင်နှုန်းလောက် အလုပ်များနေကြမယ်ဆိုရင်တော့ ဒီအလုပ်ဟာ queue ထဲမှာတင် စုစုပေါင်း ၉ နာရီ နဲ့ ၇ ကြိမ် မြှောက်တော့…”

“ဘာဗျာ! queue ထဲမှာတင် ၆၃ နာရီတောင် ကြာမယ် ဟုတ်လား” လို့ Wes က မယုံနိုင်သလို အော်လိုက်တယ်— “ဒါကတော့ မဖြစ်နိုင်တာ!” တဲ့။

Patty က ခနဲ့တဲ့တဲ့ အပြုံးနဲ့— “အမှန်ပေါ့… ဘာလို့လဲဆိုတော့ တကယ်တမ်း စာရိုက်ရတာက စက္ကန့် ၃၀ ပဲ မဟုတ်လား” လို့ ဆိုပါတယ်။

“ဟာ… သွားပြီ” လို့ Wes က ဂရပ်ဖ်ကို စိုက်ကြည့်ရင်း ရေရွတ်လိုက်ပါတယ်။

ရုတ်တရက်ဆိုရင်ပဲ Sarah နဲ့ Chris တို့က Kirsten ရဲ့ အစည်းအဝေးမှာ Phoenix ကို စတင်ဖြန့်ကျက် (deploy) ဖို့ ဆုံးဖြတ်ခဲ့တုန်းက Wes ပြောခဲ့တာကို သတိရသွားမိတယ်။ Phoenix နဲ့ ပတ်သက်တဲ့ tickets တွေဟာ ရက်သတ္တပတ်တွေ အများကြီး ကြာအောင် ဟိုဘက် ဒီဘက် ရောက်နေတာကြောင့် deployment နှောင့်နှေးရတာပါလို့ သူ ညည်းခဲ့ဖူးတာလေ။

အဲဒီတုန်းကလည်း ဒါမျိုး ဖြစ်ခဲ့တာပါပဲ။ အဲဒါက IT Operations အဖွဲ့အချင်းချင်း လွှဲပြောင်းတာ မဟုတ်ဘဲ Development (ဆော့ဖ်ဝဲလ် အဖွဲ့) နဲ့ IT Operations အဖွဲ့ကြား လွှဲပြောင်းတာမို့လို့ အများကြီး ပိုပြီး ရှုပ်ထွေးခဲ့တာပေါ့။

ဌာနတစ်ခုအတွင်းမှာတင် အလုပ်တွေကို ဦးစားပေး သတ်မှတ်ဖို့ ခက်ခဲနေရင် ဌာနတွေကြားထဲမှာ အလုပ်တွေကို စီမံခန့်ခွဲဖို့ကတော့ ဆယ်ဆလောက် ပိုပြီး ခက်ခဲပါလိမ့်မယ်။

Patty က— “ဒီ ဂရပ်ဖ်က ပြောချင်တာကတော့ လူတိုင်းမှာ အားလပ်ချိန် (idle time ဒါမှမဟုတ် slack time) ရှိဖို့ လိုအပ်တယ်ဆိုတာပဲ။ ဘယ်သူမှ အားလပ်ချိန် မရှိဘူးဆိုရင်တော့ WIP တွေက စနစ်ထဲမှာ ပိတ်မိနေလိမ့်မယ်။ အတိအကျ ပြောရရင် queue ထဲမှာ စောင့်ရင်းနဲ့ ပိတ်မိနေကြတာပေါ့” တဲ့။

ဒါကို ငါတို့တွေ စဉ်းစားနေတုန်းမှာပဲ Patty က ဆက်ပြောပြန်တယ်— “whiteboard ပေါ်က စာရွက်တစ်ရွက်ချင်းစီဟာ ဒီ Phoenix ‘task’ (အလုပ်) တွေလိုမျိုးပဲလေ” လို့ သူမက လက်ဟန်ပြရင်း ပြောပါတယ်။ “ဒါဟာ လူတစ်ယောက်တည်း လုပ်ရမယ့် အလုပ်လို့ ထင်ရပေမဲ့ အမှန်တော့ အဲဒီလို မဟုတ်ဘူး။ လူအများကြီးကြားမှာ အလုပ်လွှဲပြောင်းမှုတွေ အများကြီး ပါဝင်နေတဲ့ အဆင့်တွေ အများကြီးပါပဲ။ Kirsten ရဲ့ ပရောဂျက် ခန့်မှန်းချက်တွေ လွဲနေတာလည်း မဆန်းပါဘူး။

“ငါတို့ ဒါကို Kirsten ရဲ့ အချိန်ဇယားနဲ့ သူမရဲ့ လုပ်ငန်းခွဲခြမ်းစိတ်ဖြာမှု ပုံစံ (WBS) ထဲမှာ ပြင်ဆင်ပေးဖို့ လိုတယ်။ ကျွန်မ တွေ့ထားသလောက်ဆိုရင် Kirsten ကို ပေးထားတဲ့ ကတိတွေရဲ့ သုံးပုံတစ်ပုံလောက်က ဒီလို အမျိုးအစားတွေ ဖြစ်နေတာ” တဲ့။

“ကောင်းလှချည်လား” လို့ Wes က ဆိုတယ်— “Gilligan’s Island ဇာတ်လမ်းလိုပါပဲ။ လူတွေကို ၃ နာရီကြာ ကမ္ဘာလှည့် ခရီးစဉ်တွေ ပို့ပေးလိုက်တယ်၊ လပေါင်းများစွာ ကြာတဲ့အထိ ဘယ်သူမှ ပြန်မလာကြတော့မှ ငါတို့တွေက ဘာဖြစ်လို့လဲဆိုပြီး အံ့သြနေကြတာမျိုးပေါ့” ဆိုတော့—

Patty က— “ဒီ ‘အလုပ်’ တွေ တစ်ခုချင်းစီအတွက် ငါတို့ kanban lane (ကန်ဘန် လမ်းကြောင်း) တွေ ဖန်တီးရင်ရော ဘယ်လိုလဲ” လို့ မေးပါတယ်။

“ဟုတ်တယ်… အဲဒါပဲ!” လို့ ကျွန်တော်က ထောက်ခံလိုက်တယ်— “Erik ပြောတာ မှန်နေတာပဲ။ မင်းက ထပ်ခါတလဲလဲ လုပ်နေရတဲ့ အလုပ်ပုံကြီး (recurring work) ကို ရှာတွေ့သွားတာပဲ! အကယ်၍ ငါတို့ ဒါတွေကို မှတ်တမ်းတင်ပြီး စံနှုန်း သတ်မှတ်နိုင်ခဲ့ရင်၊ ပြီးတော့ လက်တော့ပ် လဲလှယ်ရေးမှာ လုပ်ခဲ့သလိုမျိုး ဒါကို ကျွမ်းကျင်အောင် လုပ်နိုင်ခဲ့ရင်တော့ အလုပ်စီးဆင်းမှုက အများကြီး ပိုကောင်းလာမှာ သေချာတယ်!”

ကျွန်တော် ထပ်ပြောလိုက်တယ်— “အကယ်၍ ငါတို့တွေ ထပ်ခါတလဲလဲ လုပ်နေရတဲ့ deployment (စနစ်ဖြန့်ကျက်ခြင်း) အလုပ်တွေကို စံနှုန်း သတ်မှတ်နိုင်ခဲ့ရင်တော့ production (အလုပ်လုပ်နေတဲ့ စနစ်) ထဲမှာ ရှိတဲ့ အရာတွေ အကုန်လုံးဟာ တစ်ပြေးညီ (uniformity) ဖြစ်လာလိမ့်မယ်။ အခုကတော့ စနစ်တစ်ခုနဲ့တစ်ခု မတူညီတဲ့ infrastructure snowflake (နှင်းပွင့်တစ်ခုချင်းစီ မတူသလိုမျိုး စနစ်တွေ မတူညီကြတဲ့ ပြဿနာ) တွေ ဖြစ်နေတာလေ။ Brent က ဘာလို့ Brent ဖြစ်လာရသလဲဆိုတော့ သူတစ်ယောက်တည်းပဲ နားလည်နိုင်တဲ့ စနစ်တွေကို သူတစ်ယောက်တည်းပဲ တည်ဆောက်ခွင့် ပေးခဲ့လို့ပဲ။ ဒါမျိုး နောက်တစ်ခါ ထပ်ဖြစ်အောင် ခွင့်မပြုနိုင်တော့ဘူး” ဆိုတော့—

“အချက်ကောင်းပဲ” လို့ Wes က ခေါင်းညိတ် ထောက်ခံပါတယ်— “ဒါက ထူးဆန်းတယ်နော်။ ငါတို့ ကြုံတွေ့နေရတဲ့ ပြဿနာ တော်တော်များများဟာ ငါတို့ ချခဲ့တဲ့ ဆုံးဖြတ်ချက်တွေကြောင့်ပဲ ဖြစ်နေတာ။ ရန်သူဆိုတာ တခြားလူ မဟုတ်ဘူး… ငါတို့ ကိုယ်တိုင်ပဲ ဖြစ်နေတာကိုး” တဲ့။

Patty က— “စနစ်ဖြန့်ကျက်တာ (deployments) ဆိုတာ စက်ရုံတစ်ရုံရဲ့ နောက်ဆုံး တပ်ဆင်ရေး အဆင့် (final assembly) နဲ့ တူတူပါပဲ။ အလုပ်စီးဆင်းမှုတိုင်းက အဲဒီ အဆင့်ကို ဖြတ်ရတာလေ၊ အဲဒါ မရှိဘဲနဲ့ ပစ္စည်းကို အပြင် ထုတ်လို့ မရဘူး။ အခုတော့ kanban က ဘယ်လိုမျိုး ဖြစ်ရမယ်ဆိုတာ ကျွန်မ ကောင်းကောင်း သိသွားပြီ” လို့ ဆိုပါတယ်။

နောက်ထပ် မိနစ် ၄၀ လောက် ကြာတဲ့အထိ ငါတို့တွေ အစီအစဉ် ဆွဲကြပါတယ်။ Patty က Wes ရဲ့ အဖွဲ့နဲ့ ပူးပေါင်းပြီး အဖြစ်အများဆုံး အလုပ် ၂၀ ကို စုစည်းပါလိမ့်မယ်။ ပြီးတော့ queue ထဲ ရောက်နေတဲ့ အလုပ်တွေကိုလည်း ဘယ်လို စီမံခန့်ခွဲမလဲဆိုတာကို သူမ ရှာဖွေပါလိမ့်မယ်။ Patty က ‘water spider’ (ရေပင့်ကူ) လို့ ခေါ်တဲ့ အခန်းကဏ္ဍ အသစ်တစ်ခုကို အဆိုပြုပါတယ်၊ အဲဒါကတော့ ပရောဂျက် မန်နေဂျာနဲ့ အလုပ်တွန်းအားပေးသူ (expediter) တို့ကို ပေါင်းစပ်ထားတဲ့ အခန်းကဏ္ဍပေါ့။ နေ့စဉ် ကြီးကြပ်နေမယ့်အစား မိနစ်တိုင်း မိနစ်တိုင်းမှာ အလုပ်တွေကို အနီးကပ် ထိန်းချုပ်ပေးမှာပါ။ သူမက— “အလုပ်တစ်ခု ပြီးတာနဲ့ နောက်ထပ် အလုပ်စခန်း တစ်ခုဆီကို အမြန်ဆုံးနဲ့ ထိရောက်စွာ လွှဲပြောင်းပေးဖို့ လိုတယ်။ လိုအပ်ရင်တော့ ဒီလူဟာ အလုပ်စခန်း ရှေ့မှာပဲ အလုပ်ပြီးတဲ့အထိ စောင့်နေပြီး ပြီးတာနဲ့ နောက်တစ်ယောက်ဆီကို လက်နဲ့ တန်းပြီး သွားပို့ပေးရမယ်။ ဒါမှပဲ အရေးကြီးတဲ့ အလုပ်တွေက tickets ပုံကြီးထဲမှာ ပျောက်မသွားမှာပေါ့” လို့ ဆိုတော့—

“ဘာဗျာ! အလုပ်တွေကို တစ်ယောက်ဆီကနေ တစ်ယောက်ဆီကို စားပွဲထိုးသလိုမျိုး လိုက်ပို့ပေးရမယ့်သူ ဟုတ်လား” လို့ Wes က မယုံနိုင်သလို မေးပါတယ်။

“MRP-8 မှာတော့ ‘water spider’ လို့ ခေါ်တဲ့သူက အဲဒီအတိုင်းပဲ လုပ်တာလေ” လို့ သူမက ပြန်ပြောပါတယ်— “အခု ဖြစ်ခဲ့တဲ့ Phoenix နှောင့်နှေးမှု တော်တော်များများဟာ အလုပ်တွေ queue ထဲမှာ ရက်ပေါင်းများစွာ ကြာနေတာကြောင့် ဖြစ်ခဲ့တာလေ။ ဒါမျိုး နောက်တစ်ခါ ထပ်မဖြစ်အောင် ဒီနည်းလမ်းက ကူညီပေးလိမ့်မယ်။

“နောက်ပိုင်းမှာတော့…” သူမက ဆက်ပြောတယ်— “kanban စနစ်တွေ သေသေချာချာ ဖြစ်လာရင်တော့ ဒီလို လူနဲ့ လိုက်ပို့နေစရာတောင် မလိုတော့ပါဘူး။ စိတ်ချပါ… နောက်ထပ် နှစ်ရက်လောက်နေရင် ကျွန်မ ဒါကို အကောင်အထည်ဖော်နိုင်အောင် လုပ်လိုက်ပါ့မယ်” တဲ့။

Wes နဲ့ ကျွန်တော်ကတော့ သူမကို ဆက်ပြီး သံသယ မဝင်ရဲတော့ပါဘူး။

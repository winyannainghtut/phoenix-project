အခန်း ၁၂ (CHAPTER 12)

• သောကြာနေ့၊ စက်တင်ဘာ ၁၂ ရက်

သောကြာနေ့ ညနေ ၇:၃၀ နာရီ ရှိပါပြီ။ Phoenix deployment (စတင်ဖြန့်ချိမှု) စတင်ဖို့ သတ်မှတ်ထားတဲ့ အချိန်ထက် နှစ်နာရီ နောက်ကျနေတာပါ။ အခြေအနေတွေကတော့ သိပ်မကောင်းလှပါဘူး။ ခုချိန်မှာ ပီဇာနံ့ ရလာတာနဲ့တင် အရှုံးမှန်းသိရက်နဲ့ ဇွတ်တိုးနေရတဲ့ (death march) အလုပ်မျိုးကို သတိရလာစေပါတယ်။

IT Operations အဖွဲ့တစ်ခုလုံးဟာ deployment အတွက် ညနေ ၄ နာရီကတည်းက အဆင်သင့် ဖြစ်နေခဲ့တာပါ။ ဒါပေမဲ့ Chris ရဲ့ အဖွဲ့ဆီက ဘာမှ မရသေးတဲ့အတွက် ဘာမှ လုပ်စရာ မရှိသေးပါဘူး။ သူတို့က နောက်ဆုံးမိနစ် အပြောင်းအလဲတွေကို လုပ်နေကြတုန်းပါပဲ။

လွှတ်တင်မယ့် အချိန်ရောက်နေပြီဖြစ်တဲ့ အာကာသယာဉ်မှာ အစိတ်အပိုင်းတွေကို အခုမှ လိုက်တပ်နေကြတယ်ဆိုတာ ဘယ်လိုမှ မကောင်းတဲ့ လက္ခဏာပါပဲ။

ညနေ ၄:၃၀ မှာ William က ဒေါသတကြီးနဲ့ Phoenix war room (စစ်ဆင်ရေးခန်း) ထဲ ဝင်လာပါတယ်။ Phoenix code တွေကို test environment (စမ်းသပ်ပတ်ဝန်းကျင်) ထဲမှာ ဘယ်သူမှ လည်ပတ်အောင် မလုပ်နိုင်ကြတာကို သူ တကယ်ကို စိတ်ပျက်နေတာပါ။ ပိုဆိုးတာကတော့ Phoenix ရဲ့ လည်ပတ်နေတဲ့ အစိတ်အပိုင်း အနည်းငယ်ကတည်း အရေးကြီးတဲ့ test (စမ်းသပ်မှု) တွေမှာ ရှုံးနိမ့်နေတာပါပဲ။

William က developer တွေဆီကို အရေးကြီးတဲ့ bug reports (အမှားအယွင်း အစီရင်ခံစာများ) ကို ပြန်ပို့နေရတယ်။ အဲဒီ developer တွေထဲက အများစုက အိမ်ပြန်သွားကြပြီဖြစ်လို့ Chris က သူတို့ကို ပြန်ခေါ်ရပြီး William ရဲ့ အဖွဲ့ကတော့ version အသစ်တွေကို စောင့်နေရပါတယ်။

ကျွန်တော့်လူတွေလည်း ဒီအတိုင်း လက်ပိုက်ကြည့်နေကြတာ မဟုတ်ပါဘူး။ William ရဲ့ အဖွဲ့နဲ့ ပေါင်းပြီး Phoenix ကို test environment ထဲမှာ တက်လာအောင် အသည်းအသန် ကြိုးစားနေကြတာပါ။ ဘာလို့လဲဆိုတော့ တကယ်လို့ test environment မှာတောင် အလုပ်မလုပ်ဘူးဆိုရင် production (တကယ့် လုပ်ငန်းသုံး ပတ်ဝန်းကျင်) မှာ ဒါကို deploy လုပ်ပြီး လည်ပတ်ဖို့ဆိုတာ ဘယ်လိုမှ မဖြစ်နိုင်လို့ပါပဲ။

ကျွန်တော့် အကြည့်တွေက နာရီကနေ အစည်းအဝေး စားပွဲဆီကို ရောက်သွားတယ်။ Brent နဲ့ နောက်ထပ် အင်ဂျင်နီယာ သုံးယောက်ဟာ သူတို့ရဲ့ QA အဖွဲ့ဝင်တွေနဲ့ စုပြီး အလုပ်လုပ်နေကြတယ်။ ညနေ ၄ နာရီကတည်းက အသည်းအသန် လုပ်နေကြရတာဖြစ်လို့ သူတို့ မျက်နှာတွေက တော်တော်လေး နွမ်းနယ်နေကြပါပြီ။ တချို့လည်း laptop တွေဖွင့်ပြီး Google မှာ ရှာဖွေနေကြသလို တချို့ကလည်း server တွေ၊ operating systems တွေ၊ databases တွေနဲ့ Phoenix application ရဲ့ settings တွေကို စနစ်တကျ လိုက်ပြင်ရင်း အရာအားလုံး ပြန်တက်လာအောင် ကြိုးစားနေကြတယ်။

developer တစ်ယောက်က လွန်ခဲ့တဲ့ မိနစ်အနည်းငယ်ကမှ ဝင်လာပြီး— “ကြည့်ပါဦး… ကျွန်တော့် laptop မှာတော့ အလုပ်လုပ်နေတာပဲ။ အဲဒါက ဘယ်လောက်တောင် ခက်လို့လဲ” လို့ ပြောသွားသေးတယ်။

Wes က ဆဲဆိုနေပြီး ကျွန်တော့် အင်ဂျင်နီယာ နှစ်ယောက်နဲ့ William ရဲ့ အင်ဂျင်နီယာ သုံးယောက်တို့က အဲဒီ developer ရဲ့ laptop ကို ဝိုင်းကြည့်ပြီး test environment နဲ့ ဘာတွေ ကွာခြားနေလဲဆိုတာကို လိုက်ရှာနေကြပါတယ်။

အခန်းရဲ့ တခြား တစ်ဖက်မှာလည်း အင်ဂျင်နီယာ တစ်ယောက်က ဖုန်းထဲက တစ်ယောက်ကို ဒေါသတကြီး ပြောနေတယ်— “ဟုတ်ကဲ့… ခင်ဗျား ပေးတဲ့ file ကို ကျွန်တော်တို့ copy ကူးပြီးသား… ဟုတ်ကဲ့… version 1.0.13 ပါ… ဘာပြောတယ်၊ version မှားနေတာ ဟုတ်လား… ဘာ! ဘယ်တုန်းက ပြောင်းလိုက်တာလဲ?… အခု ချက်ချင်း copy ကူးပြီး ပြန်စမ်းကြည့်လိုက်မယ်… အိုကေ… ဒါပေမဲ့ ကျွန်တော် ပြောသားပဲ၊ ဒါဟာ အလုပ်ဖြစ်မှာ မဟုတ်ဘူး… ဒါဟာ networking problem (ကွန်ရက် ပြဿနာ) လို့ ကျွန်တော် ထင်တယ်… ဘာ! firewall port တစ်ခု ဖွင့်ဖို့ လိုတယ် ဟုတ်လား? အဲဒါကို လွန်ခဲ့တဲ့ နှစ်နာရီကတည်းက ဘာလို့ မပြောတာလဲဗျာ!”

သူ ဖုန်းကို စားပွဲပေါ် ပြင်းပြင်းထန်ထန် ချလိုက်ပြီး လက်သီးနဲ့ ထုလိုက်ရင်း— “ငတုံးတွေ!” လို့ အော်လိုက်ပါတယ်။

Brent က developer ရဲ့ laptop ဆီကနေ မျက်လုံးတွေကို ပွတ်ပြီး မော့ကြည့်လိုက်တယ်။ “မှန်းကြည့်ရရင်တော့… front-end က database server ကို ဆက်သွယ်လို့ မရတာဟာ firewall port တစ်ခု ဖွင့်ဖို့ လိုအပ်တယ်ဆိုတာ တစ်ယောက်ယောက်က ငါတို့ကို အသိမပေးခဲ့လို့ မဟုတ်လား” တဲ့။

အဲဒီ အင်ဂျင်နီယာက စိတ်ဆိုးမာန်ဆိုးနဲ့ ခေါင်းညိတ်ပြပြီး— “ဒါက တကယ်ကို မယုံနိုင်စရာပဲဗျာ။ အဲဒီ အရူးနဲ့ ဖုန်းပြောနေတာ မိနစ် ၂၀ ရှိပြီ၊ ဒါဟာ code ပြဿနာ မဟုတ်ဘူးဆိုတာကို သူ တစ်ခါမှ မစဉ်းစားမိဘူး။ ဒါဟာ FUBAR ပဲ” (FUBAR: လုံးဝကို ပျက်စီးနေတဲ့ အခြေအနေ) လို့ ပြောပါတယ်။

ကျွန်တော် တိတ်တိတ်လေးပဲ နားထောင်နေပေမဲ့ သူ့ရဲ့ သုံးသပ်ချက်ကိုတော့ သဘောတူမိပါတယ်။ မရိန်းတပ်မှာလည်း ငါတို့တွေ FUBAR ဆိုတဲ့ စကားလုံးကို သုံးခဲ့ကြတာပဲလေ။

လူတွေ စိတ်တိုနေကြတာကို ကြည့်ရင်း နာရီကို ကြည့်လိုက်တော့—ညနေ ၇:၃၇ နာရီ။

ကျွန်တော့်အဖွဲ့ဆီကနေ စီမံခန့်ခွဲမှုပိုင်းဆိုင်ရာ သုံးသပ်ချက်တစ်ခု ယူဖို့ အချိန်ရောက်ပါပြီ။ Wes နဲ့ Patty ကို ခေါ်လိုက်ပြီး William ကို လိုက်ရှာလိုက်တယ်။ သူက သူ့အင်ဂျင်နီယာတစ်ယောက်ရဲ့ ပုခုံးပေါ် ကျော်ကြည့်နေတာ တွေ့တာနဲ့ ကျွန်တော်တို့နဲ့ အတူတူ လိုက်ခဲ့ဖို့ ပြောလိုက်ပါတယ်။

သူက ငါတို့နဲ့ အတူတူ အလုပ်လုပ်လေ့ မရှိလို့ ခဏတော့ ကြောင်သွားပေမဲ့ ခေါင်းညိတ်ပြီး ကျွန်တော့်ရုံးခန်းဆီကို လိုက်ခဲ့ပါတယ်။

“ကဲ… အားလုံးပဲ၊ ဒီအခြေအနေအပေါ် ဘယ်လို ထင်ကြလဲ” လို့ ကျွန်တော် မေးလိုက်တယ်။

Wes က အရင်ဆုံး ပြောလိုက်တယ်— “သူတို့ ပြောတာ မှန်ပါတယ်။ ဒါဟာ FUBAR ပဲ။ developer တွေဆီကနေ အပြည့်အစုံ မပါတဲ့ releases တွေကိုပဲ ငါတို့ ရနေတုန်းပဲ။ ပြီးခဲ့တဲ့ နှစ်နာရီအတွင်းမှာတင် အရေးကြီးတဲ့ files တွေ ပေးဖို့ မေ့ခဲ့လို့ code က လုံးဝ အလုပ်မလုပ်တော့တဲ့ ဖြစ်ရပ် နှစ်ခုကို ငါ မြင်ခဲ့ရပြီးပြီ။ ပြီးတော့ ဆရာ မြင်တဲ့အတိုင်းပဲ၊ Phoenix လည်ပတ်နိုင်အောင် test environment ကို ဘယ်လို configure လုပ်ရမလဲဆိုတာကို ငါတို့ ခုထိ မသိကြသေးဘူး။”

သူ ခေါင်းယမ်းလိုက်ပြီး— “ပြီးခဲ့တဲ့ နာရီဝက်အတွင်း မြင်ခဲ့တာအရဆိုရင်တော့ ငါတို့တွေဟာ တိုးတက်လာမယ့်အစား နောက်ကို ပြန်ရောက်သွားကြတာပဲ” လို့ ပြောပါတယ်။

Patty ကတော့ စိတ်ပျက်လက်ပျက်နဲ့ ခေါင်းယမ်းပြပြီး ဘာမှ မပြောတော့ပါဘူး။

ကျွန်တော် William ကို— “ငါတို့ အတူတူ အလုပ်မလုပ်ဖြစ်ကြတာ ကြာပြီဆိုပေမဲ့ မင်းရဲ့ အမြင်ကိုလည်း သိချင်ပါတယ်။ မင်းဘက်က ကြည့်ရင် အခြေအနေ ဘယ်လို ရှိသလဲ” လို့ မေးလိုက်တယ်။

သူက ငုံ့ကြည့်ပြီး သက်ပြင်းချလိုက်တယ်— “အမှန်အတိုင်း ပြောရရင် ကျွန်တော် ဘာမှ မပြောတတ်တော့ဘူး။ code တွေက အရမ်း မြန်မြန် ပြောင်းလဲနေတော့ ကျွန်တော်တို့ လိုက်မမီတော့ဘူး။ အကယ်၍ ကျွန်တော်သာ လောင်းကြေးထပ်ရမယ်ဆိုရင်တော့ Phoenix ဟာ production မှာ သေချာပေါက် ပေါက်ကွဲသွားလိမ့်မယ်။ Chris ကို ဒီ release ကို ရပ်ထားဖို့ ကျွန်တော် အကြိမ်ကြိမ် ပြောခဲ့ပေမဲ့ သူနဲ့ Sarah က ကျွန်တော့်ကို လျစ်လျူရှုခဲ့ကြတယ်လေ” လို့ ပြောပါတယ်။

“ ‘လိုက်မမီတော့ဘူး’ ဆိုတာ ဘာကို ဆိုလိုတာလဲ” လို့ ကျွန်တော် မေးလိုက်တယ်။

“ကျွန်တော်တို့ testing လုပ်နေရင်း ပြဿနာတွေ တွေ့ရင် Development ဆီ ပြန်ပို့ရတယ်၊ သူတို့က ပြင်ပေးဖို့ပေါ့” လို့ သူက ရှင်းပြတယ်။ “အဲဒီနောက် သူတို့က release အသစ်တစ်ခု ပြန်ပို့ပေးတယ်။ ပြဿနာကတော့ အရာအားလုံးကို အဆင်သင့် ဖြစ်အောင် လုပ်ဖို့က နာရီဝက်လောက် ကြာသလို smoke test (အခြေခံ စမ်းသပ်မှု) လုပ်ဖို့ကလည်း နောက်ထပ် သုံးနာရီလောက် ကြာတယ်။ အဲဒီအချိန်အတွင်းမှာတင် Development ဆီကနေ release သုံးခုလောက် ထပ်ရောက်လာပြန်ပြီလေ။”

smoke test ဆိုတဲ့ စကားလုံးကြောင့် ကျွန်တော် ပြုံးမိသွားတယ်။ ဒါက လျှပ်စစ်ပတ်လမ်း ဒီဇိုင်းဆွဲသူတွေ သုံးတဲ့ စကားလုံးလေ။ “ပတ်လမ်းပြားကို ဖွင့်လိုက်တဲ့အခါ မီးခိုးတွေ ထွက်မလာဘူးဆိုရင်တော့ ဒါဟာ အလုပ်လုပ်ဖို့ များပါတယ်” ဆိုတဲ့ ဆိုရိုးစကားလည်း ရှိတယ် မဟုတ်လား။

သူ ခေါင်းယမ်းပြီး ဆက်ပြောတယ်— “ကျွန်တော်တို့ smoke test ကိုတောင် ခုထိ မအောင်မြင်သေးဘူး။ version control (ဗားရှင်း ထိန်းချုပ်မှု) လည်း ကောင်းကောင်း မရှိတော့တာကို စိုးရိမ်မိတယ်—တစ်ခုလုံးရဲ့ version numbers တွေကို လိုက်မှတ်ဖို့ကိုတောင် အရမ်း ပေါ့ဆနေကြပြီ။ တစ်ခုခုကို ပြင်လိုက်တိုင်း နောက်ထပ် တစ်ခုခု ပျက်သွားလေ့ ရှိတယ်။ ဒါကြောင့် သူတို့က package တစ်ခုလုံး ပို့မယ့်အစား file တစ်ခုချင်းစီပဲ လိုက်ပို့နေကြတာ။”

သူ ဆက်ပြောတယ်— “အခု အခြေအနေက တကယ့်ကို ပရမ်းပတာ ဖြစ်နေတာ။ တကယ်လို့ Phoenix က ကံကောင်းလို့ smoke test အောင်မြင်သွားမယ်ဆိုရင်တောင် အဲဒါကို နောက်တစ်ခါ ထပ်ပြီး ပြန်လုပ်နိုင်ဖို့ဆိုတာ မလွယ်တော့ဘူး၊ ဘာလို့လဲဆိုတော့ ပြောင်းလဲနေတဲ့ အစိတ်အပိုင်းတွေက အရမ်း များလွန်းလို့ပဲ။”

သူ့မျက်မှန်ကို ချွတ်လိုက်ပြီး ပြတ်ပြတ်သားသား ပြောလိုက်တယ်— “ဒါဟာ လူတိုင်းအတွက်တော့ တစ်ညလုံး မအိပ်ဘဲ လုပ်ရမယ့် အလုပ်ပဲ။ မနက်ဖြန် မနက် ၈ နာရီ စတိုးဆိုင်တွေ ဖွင့်တဲ့အချိန်မှာ ဘာမှ အလုပ်မလုပ်နိုင်မှာကို ကျွန်တော် တကယ်ကို စိုးရိမ်မိတယ်။ အဲဒါက တကယ့် ပြဿနာကြီးပဲ။”

အဲဒါကတော့ ပြဿနာကြီးဆိုတာထက်ကို ပိုပါတယ်။ အကယ်၍ release က မနက် ၈ နာရီမှာ မပြီးဘူးဆိုရင် စတိုးဆိုင်တွေမှာ သုံးတဲ့ POS (Point of Sale) စနစ်တွေက အလုပ်လုပ်မှာ မဟုတ်ဘူး။ အဲဒီတော့ ဖောက်သည်တွေနဲ့ အရောင်းအဝယ် လုပ်လို့ ရမှာ မဟုတ်တော့ပါဘူး။

Wes က ခေါင်းညိတ်ပြီး— “William ပြောတာ မှန်ပါတယ်။ ငါတို့ ဒီမှာ တစ်ညလုံး ရှိနေကြရမှာ သေချာတယ်။ ပြီးတော့ performance (စွမ်းဆောင်ရည်) ကလည်း ငါ ထင်ထားတာထက် ပိုဆိုးနေတယ်။ အလုပ်တွေကို ခွဲဝေဖို့အတွက် server အနည်းဆုံး နောက်ထပ် ၂၀ လောက် လိုဦးမယ်၊ အခုလို အချိန်တိုအတွင်းမှာ အဲဒီလောက် အများကြီး ဘယ်က ရှာရမလဲ မသိတော့ဘူး။ အပို hardware တွေ ရှိမလားလို့ ငါ့လူတွေကို လိုက်ရှာခိုင်းထားတယ်။ ဒါမှမဟုတ် production ထဲက server တွေကိုပဲ ဆွဲထုတ်သုံးရမလား မသိဘူး” လို့ ပြောပါတယ်။

“deployment ကို ရပ်လိုက်ဖို့ အချိန် နောက်ကျသွားပြီလား” လို့ ကျွန်တော် မေးလိုက်တယ်။ “ပြန်လှည့်လို့ မရတော့တဲ့ အချိန်က အတိအကျ ဘယ်အချိန်လဲ။”

“ကောင်းတဲ့ မေးခွန်းပဲ” လို့ Wes က ဖြည်းဖြည်းချင်း ဖြေတယ်— “Brent ကိုတော့ မေးကြည့်ရဦးမယ်၊ ဒါပေမဲ့ အခုချက်ချင်း ရပ်လိုက်မယ်ဆိုရင်တော့ ဘာပြဿနာမှ မရှိသေးဘူး ထင်တယ်။ ဒါပေမဲ့ POS စနစ်ရော Phoenix ကပါ အော်ဒါတွေ လက်ခံနိုင်အောင် database conversion (ဒေတာဘေ့စ် ပြောင်းလဲခြင်း) စလုပ်လိုက်ပြီဆိုရင်တော့ ငါတို့တွေဟာ ပြန်လှည့်လို့ မရတော့ဘူး။ အဲဒီအချိန် ရောက်ဖို့ကတော့ နောက်ထပ် နာရီအနည်းငယ်လောက် လိုသေးတယ် ထင်တာပဲ။”

ကျွန်တော် ခေါင်းညိတ်လိုက်တယ်။ ကျွန်တော် သိချင်တဲ့ အချက်အလက်တွေ ရပါပြီ။

“အားလုံးပဲ… ကျွန်တော် Steve၊ Chris နဲ့ Sarah တို့ဆီကို deployment ကို ရွှေ့ဆိုင်းဖို့ အီးမေးလ် ပို့လိုက်ပါ့မယ်။ ပြီးရင် Steve ကို လိုက်ရှာမယ်။ နောက်ထပ် တစ်ပတ်လောက် အချိန်ရအောင် ကြိုးစားကြည့်မယ်။ အင်း… တစ်ရက်လောက် ရရင်တောင်မှ အောင်မြင်မှုပဲပေါ့။ ဘယ်လို ထင်ကြလဲ” လို့ ကျွန်တော် မေးလိုက်တယ်။

Wes၊ Patty နဲ့ William တို့အားလုံး ခေါင်းယမ်းပြပြီး ဘာမှ မပြောကြတော့ပါဘူး။

ကျွန်တော် Patty ဘက် လှည့်ပြီး— “William နဲ့ ပေါင်းပြီး releases တွေမှာ traffic coordination (အသွားအလာ ညှိနှိုင်းမှု) ပိုကောင်းအောင် ဘယ်လို လုပ်မလဲဆိုတာ သွားစဉ်းစားလိုက်ပါ။ developer တွေ ရှိတဲ့နေရာကို သွားပြီး air traffic controller (လေကြောင်း ထိန်းချုပ်သူ) လိုမျိုး အရာရာတိုင်းကို သေသေချာချာ version သတ်မှတ်ခိုင်းပါ။ ပြီးရင် ဘာတွေ လာနေလဲဆိုတာကို Wes တို့ အဖွဲ့ကို အသိပေးပါ။ အရာအားလုံးကို ပွင့်လင်းမြင်သာမှု ရှိစေချင်သလို လူတိုင်းကိုလည်း လုပ်ငန်းစဉ်အတိုင်း လုပ်ခိုင်းစေချင်တယ်။ code Drops တွေအတွက် ဝင်ပေါက် တစ်ခုတည်း ထားမယ်၊ အချိန်အလိုက် releases တွေကို ထိန်းချုပ်မယ်၊ မှတ်တမ်းတင်မယ်… ငါ ဘာပြောချင်လဲဆိုတာ နားလည်တယ် မဟုတ်လား” လို့ မှာလိုက်တယ်။

သူက— “ဝမ်းမြောက်ဝမ်းသာပါပဲ။ Phoenix war room ကနေပဲ စလိုက်ပါ့မယ်။ တံခါးကို ကန်ဖွင့်ပြီး ‘ငါတို့ ကူညီဖို့ လာခဲ့တာ…’ လို့တောင် ပြောလိုက်ချင်သေးတယ်” လို့ ပြန်ဖြေပါတယ်။

ကျွန်တော် သူတို့ကို ကျေးဇူးတင်ကြောင်း ပြောလိုက်ပြီး အီးမေးလ် ရေးဖို့ laptop ဆီကို သွားလိုက်တယ်။

Steve ခင်ဗျား၊

ပထမဆုံးအနေနဲ့ ကျွန်တော်ဟာ Phoenix ကို production ထဲကို တခြားလူတွေလိုပဲ အမြန်ဆုံး ရောက်စေချင်ပါတယ်၊ ဒါဟာ ကုမ္ပဏီအတွက် ဘယ်လောက်အထိ အရေးကြီးလဲဆိုတာ ကျွန်တော် နားလည်ပါတယ်။

ဒါပေမဲ့ ကျွန်တော့်ရဲ့ လေ့လာချက်အရ မနက်ဖြန် မနက် ၈ နာရီ သတ်မှတ်ရက်မှာ Phoenix ဟာ အဆင်သင့် ဖြစ်မှာ မဟုတ်ဘူးလို့ ကျွန်တော် ယုံကြည်ပါတယ်။ ဒါဟာ စတိုးဆိုင်တွေက POS စနစ်တွေကိုပါ ထိခိုက်နိုင်တဲ့ အန္တရာယ် (SIGNIFICANT RISK) ရှိနေပါတယ်။

William နဲ့ ဆွေးနွေးပြီးတဲ့နောက်မှာတော့ Phoenix ဟာ သူ့ရဲ့ ရည်မှန်းချက်တွေကို အောင်မြင်အောင် လုပ်ဆောင်နိုင်ဖို့နဲ့ မလွဲမသွေ ဖြစ်လာတော့မယ့် ဘေးဒုက္ခကို ရှောင်လွှဲနိုင်ဖို့အတွက် Phoenix လွှတ်တင်မယ့်ရက်ကို တစ်ပတ်လောက် ရွှေ့ဆိုင်းဖို့ ကျွန်တော် အကြံပြုချင်ပါတယ်။

ဒါဟာ ၁၉၉၉ ခုနှစ်က Toys R Us ကုမ္ပဏီ ကြုံခဲ့ရတဲ့ ကျဆုံးမှုမျိုးနဲ့ ဆင်တူနိုင်ပါတယ်၊ ဆိုလိုတာကတော့ ရက်ပေါင်းများစွာ စနစ်တွေ ပြတ်တောက်တာနဲ့ performance ပြဿနာတွေကြောင့် ဖောက်သည်တွေရဲ့ အချက်အလက်တွေနဲ့ အော်ဒါတွေပါ အန္တရာယ် ရှိလာနိုင်တာမျိုးပေါ့။

Steve၊ ကျွန်တော် နောက် မိနစ်အနည်းငယ်အတွင်း ဖုန်းဆက်ပါ့မယ်။

လေးစားစွာဖြင့်၊
Bill

ကျွန်တော် စိတ်ကို ပြန်စုစည်းလိုက်ပြီး Steve ဆီကို ဖုန်းခေါ်လိုက်တဲ့အခါ သူက ပထမဆုံး အကြိမ် မြူးသံမှာတင် ဖုန်းကိုင်လိုက်ပါတယ်။

“Steve၊ Bill ပါ။ ကျွန်တော် ခင်ဗျားတို့ သုံးယောက်ဆီကို အီးမေးလ် ပို့ထားပါတယ်။ ဒီ rollout က ဘယ်လောက်အထိ ဆိုးဆိုးရွားရွား ဖြစ်နေလဲဆိုတာ ကျွန်တော် ထပ်ပြီး ပြောစရာ မလိုတော့ဘူး ထင်ပါတယ်။ ဒါဟာ ငါတို့ကို တကယ့်ကို ပြန်ပြီး ဒုက္ခပေးလာပါလိမ့်မယ်။ William လည်း သဘောတူပါတယ်။ မနက်ဖြန် မနက် ၈ နာရီမှာ စတိုးဆိုင်တွေ ဖွင့်တဲ့အချိန်မှာ rollout က ပြီးမှာ မဟုတ်ဘူးလို့ ကျွန်တော့်အဖွဲ့က အရမ်း စိုးရိမ်နေကြတယ်။ အဲဒါက စတိုးဆိုင်တွေမှာ အရောင်းအဝယ် လုပ်တာကို နှောင့်ယှက်နိုင်သလို ဝဘ်ဆိုဒ်ကိုလည်း ရက်ပေါင်းများစွာ ပြတ်တောက်သွားစေနိုင်ပါတယ်။

“ဒီ ဘေးဒုက္ခကြီးကို ရပ်တန့်ဖို့ အချိန်မနှောင်းသေးပါဘူး” လို့ ကျွန်တော် တောင်းပန်လိုက်တယ်။ “ကျရှုံးခဲ့မယ်ဆိုရင် အင်တာနက်ကရော စတိုးဆိုင်တွေကပါ အော်ဒါတွေ လက်ခံလို့ ရမှာ မဟုတ်တော့ဘူး။ ကျရှုံးခဲ့မယ်ဆိုရင် အော်ဒါ အချက်အလက်တွေနဲ့ customer records (ဖောက်သည် မှတ်တမ်းများ) တွေပါ ပျက်စီးသွားနိုင်ပြီး ဖောက်သည်တွေကို ဆုံးရှုံးရပါလိမ့်မယ်။ တစ်ပတ်လောက် ရွှေ့လိုက်တာက ဖောက်သည်တွေကို စိတ်ပျက်စေတာပဲ ရှိမယ်၊ ဒါပေမဲ့ သူတို့ကတော့ ပြန်လာကြဦးမှာပါ!” လို့ ပြောလိုက်တာပေါ့။

Steve က ဖုန်းထဲမှာ အသက်ပြင်းပြင်း ရှူလိုက်ပြီး— “အခြေအနေ ဆိုးတယ်ဆိုတာတော့ ဟုတ်ပါတယ်၊ ဒါပေမဲ့ အခုချိန်မှာ ငါတို့မှာ ရွေးချယ်စရာ မရှိဘူး။ ဆက်သွားရမယ်။ Marketing က Phoenix ရရှိနိုင်ပြီဖြစ်ကြောင်း သတင်းစာတွေမှာ ကြော်ငြာတွေ ထည့်ထားပြီးပြီ။ အဲဒါတွေက ပိုက်ဆံ ပေးပြီးသားဖြစ်သလို နိုင်ငံတစ်ဝှမ်းက အိမ်တွေဆီကိုလည်း ရောက်ကုန်ကြပြီလေ။ ငါတို့ရဲ့ မိတ်ဖက် ကုမ္ပဏီတွေကလည်း အကုန်လုံး အဆင်သင့် ဖြစ်နေကြပြီ” လို့ ပြန်ဖြေပါတယ်။

ကျွန်တော် အံ့သြမှင်တက်သွားပြီး— “Steve၊ ဒါဆို ဘယ်လောက်အထိ ဆိုးမှ ဒီ release ကို ရွှေ့ဆိုင်းမှာလဲ။ ကျွန်တော် ပြောနေတာက အခု လုပ်နေတာဟာ တကယ့်ကို အန္တရာယ်ကြီးတဲ့ အလုပ်မျိုးဖြစ်နေပြီဆိုတာပဲ!” လို့ ပြောလိုက်တယ်။

သူ ခဏ ငြိမ်သွားတယ်— “ငါ တစ်ခု ပြောမယ်။ တကယ်လို့ မင်းက Sarah ကို ဒီ rollout ကို ရွှေ့ဆိုင်းဖို့ နားချနိုင်ရင် ငါတို့ ထပ်ဆွေးနွေးကြတာပေါ့။ မဟုတ်ရင်တော့ ဆက်လုပ်ကြရုံပဲ” တဲ့။

“ခင်ဗျား မနောက်ပါနဲ့ဗျာ။ ဒီ kamikaze (ကိုယ့်ကိုယ်ကိုယ် သတ်သေမယ့်) ပြသနာကြီးကို ဖန်တီးခဲ့တာ သူပဲလေ။”

ကျွန်တော် ကိုယ့်ကိုယ်ကိုယ် မထိန်းနိုင်ဘဲ Steve ရဲ့ ဖုန်းကို ချပစ်လိုက်မိတယ်။ ခဏလောက်တော့ ပြန်ခေါ်ပြီး တောင်းပန်ရင် ကောင်းမလားလို့ စဉ်းစားမိသေးတယ်။

ကျွန်တော် ဘယ်လောက်ပဲ မုန်းနေပါစေ၊ ဒီရူးသွပ်မှုကို ရပ်တန့်ဖို့အတွက် ကုမ္ပဏီအတွက် နောက်ဆုံးတစ်ခါတော့ ကြိုးစားကြည့်သင့်တယ်လို့ ခံစားမိတယ်။ အဲဒါကတော့ Sarah နဲ့ ကိုယ်တိုင် သွားတွေ့ပြီး စကားပြောဖို့ပဲပေါ့။

Phoenix war room ထဲကို ပြန်ရောက်တဲ့အခါမှာတော့ တင်းမာမှုတွေနဲ့ ကြောက်စိတ်တွေကြောင့် ချွေးပြန်နေကြတဲ့ လူတွေရဲ့ အနံ့တွေနဲ့ အရမ်းကို အိုက်စပ်နေပါတယ်။ Sarah ကတော့ တစ်ယောက်တည်း ထိုင်ပြီး laptop မှာ စာရိုက်နေလေရဲ့။

ကျွန်တော် သူ့ကို လှမ်းခေါ်လိုက်တယ်— “Sarah၊ စကားပြောလို့ ရမလား။”

သူက သူ့ဘေးက ကုလားထိုင်ကို ညွှန်ပြပြီး— “ရတာပေါ့။ ဘာဖြစ်လို့လဲ” လို့ မေးပါတယ်။

ကျွန်တော်က အသံကို နှိမ့်လိုက်ပြီး— “စင်္ကြံလမ်းမှာ သွားပြောရအောင်” လို့ ပြောလိုက်တယ်။

ကျွန်တော်တို့ နှစ်ယောက် တိတ်တိတ်ဆိတ်ဆိတ်ပဲ အပြင်ဖက် လမ်းလျှောက်လာကြတုန်း ကျွန်တော်က— “ဒီကနေ ကြည့်ရင် release က ဘယ်လို အခြေအနေ ရှိတယ်လို့ ထင်သလဲ” လို့ မေးလိုက်တယ်။

သူက ဝေ့ဝိုက်ပြီး— “ငါတို့တွေ မြန်မြန်ဆန်ဆန် (nimble) လုပ်ဖို့ ကြိုးစားတဲ့အခါမှာ ဒီလိုမျိုးတွေ ဖြစ်တတ်တာ မင်း သိပါတယ် မဟုတ်လား။ နည်းပညာနဲ့ ပတ်သက်လာရင်တော့ မမျှော်လင့်ထားတဲ့ အရာတွေ အမြဲ ရှိနေတာပဲလေ။ အကယ်၍ မင်းက ကြက်ဥအော်မ်လက် (omelets) စားချင်တယ်ဆိုရင်တော့ ကြက်ဥတွေကို ခွဲရမှာပဲပေါ့” လို့ ပြန်ဖြေပါတယ်။

“ဒါက မင်း ကြုံနေကျ rollout တွေထက်တော့ ပိုပြီး ဆိုးလိမ့်မယ်လို့ ငါ ထင်တယ်။ ငါ့အီးမေးလ်ကို မင်း မြင်တယ် မဟုတ်လား။”

သူက ခပ်အေးအေးပဲ— “ဟုတ်တာပေါ့။ ပြီးတော့ ငါ့ရဲ့ ပြန်စာကိုကော မင်း မြင်ရဲ့လား” တဲ့။

ဒုက္ခပဲ။

ကျွန်တော်က— “ဟင့်အင်း။ ဒါပေမဲ့ မင်း မရှင်းပြခင်မှာ မင်း လုပ်နေတာက စီးပွားရေး လုပ်ငန်းအပေါ် ဘယ်လောက်အထိ သက်ရောက်မှုရှိပြီး ဘယ်လောက်အထိ အန္တရာယ်ကြီးလဲဆိုတာကို မင်း နားလည်စေချင်တယ်” လို့ ပြောပြီး Steve ကို ပြောခဲ့တဲ့ စကားတွေကိုပဲ ထပ်ပြောလိုက်ပါတယ်။

Sarah ကတော့ ထုံးစံအတိုင်းပဲ စိတ်မလှုပ်ရှားပါဘူး။ ကျွန်တော် စကားဆုံးတာနဲ့ သူက— “ငါတို့ အားလုံး Phoenix ကို ဒီအဆင့် ရောက်အောင် အသည်းအသန် ကြိုးစားခဲ့ကြတာ။ Marketing က အဆင်သင့်၊ Development က အဆင်သင့်။ မင်းကလွဲလို့ လူတိုင်း အဆင်သင့်ပဲ။ ငါ မင်းကို အရင်ကလည်း ပြောခဲ့ပြီးပြီ၊ ဒါပေမဲ့ မင်းကတော့ နားမထောင်ဘူး—ပြီးပြည့်စုံမှု (perfection) ဆိုတာ ကောင်းမွန်မှု (good) ရဲ့ ရန်သူပဲလေ။ ငါတို့တွေ ဆက်သွားရမယ်” လို့ ပြောပါတယ်။

ဒီလောက်အထိ အချိန်ဖြုန်းနေတာကို အံ့သြမိရင်း ခေါင်းယမ်းလိုက်ကာ— “မဟုတ်ဘူး… အရည်အချင်း မရှိမှု (lack of competence) ကသာ ကောင်းမွန်မှုရဲ့ ရန်သူပဲ။ ငါ့စကားကို မှတ်ထားလိုက်။ မင်းရဲ့ ဒီလို တုံးအတဲ့ ဆုံးဖြတ်ချက်တွေကြောင့် ငါတို့တွေဟာ ရက်ပေါင်းများစွာ၊ ဒါမှမဟုတ် ရက်သတ္တပတ်ပေါင်းများစွာ ဒုက္ခလိုက်ရှင်းနေရလိမ့်မယ်” လို့ ပြောလိုက်မိတယ်။

NOC ဆီကို ဒေါသတကြီး ပြန်လာတဲ့အခါမှာတော့ Sarah ရဲ့ အီးမေးလ်ကို ဖတ်လိုက်ရပြီး ပိုပြီးတော့တောင် ဒေါသထွက်သွားတယ်။ ပြန်စာပို့ပြီး မီးလောင်ရာ လေပင့်တာမျိုး မလုပ်မိအောင် ကိုယ့်ကိုယ်ကိုယ် ထိန်းလိုက်ရတယ်။ ပြီးတော့ အဲဒါကို ဖျောက်ပစ်ချင်တဲ့ စိတ်ကိုလည်း ထိန်းလိုက်ရတယ်—နောက်ပိုင်းကျရင် ကိုယ့်ကိုယ်ကိုယ် ကာကွယ်ဖို့အတွက် အဲဒါက လိုအပ်လာနိုင်လို့လေ။

အကြောင်းအရာ: ပြန်ကြားချက်—အရေးပေါ်—Phoenix deployment တကယ့်ကို အခက်အခဲ တွေ့နေပါသည်၊ ကျွန်တော့်၏ အကြံပြုချက်မှာ—၁ ပတ် ရွှေ့ဆိုင်းရန်။

မင်းကလွဲလို့ လူတိုင်း အဆင်သင့် ဖြစ်နေပါပြီ။ Marketing, Dev, Project Management အဖွဲ့တွေ အကုန်လုံး သူတို့ရဲ့ အစွမ်းကုန် ကြိုးစားခဲ့ကြပြီးပြီ။ အခုကတော့ မင်းရဲ့ အလှည့်ပဲ။
ငါတို့ သွားရမယ်!
Sarah

ရုတ်တရက်ဆိုရင်ပဲ Paige ကို ဘာမှ မပြောရသေးတာ နာရီပေါင်းများစွာ ရှိနေပြီဆိုတာ သိလိုက်ရပြီး ထိတ်လန့်သွားမိတယ်။ သူ့ဆီကို မက်ဆေ့ခ်ျ အမြန် ပို့လိုက်တယ်—

ညကတော့ ပိုပိုပြီး ဆိုးလာတယ်။ အနည်းဆုံး နောက်ထပ် နာရီအနည်းငယ်လောက်တော့ ဒီမှာ ရှိဦးမယ်။ မနက်မှ ပြန်တွေ့မယ်နော်။ ချစ်တယ်။ ကံကောင်းပါစေလို့ ဆုတောင်းပေးပါဦး ကလေးရေ။

တစ်ယောက်ယောက်က ပုခုံးကို ပုတ်လိုက်လို့ လှည့်ကြည့်လိုက်တော့ Wes ကို တွေ့ရတယ်။ “ဆရာ… ကျွန်တော်တို့မှာ တကယ့်ကို ကြီးမားတဲ့ ပြဿနာ တစ်ခု ရှိနေပြီ။”

သူ့မျက်နှာက အမူအရာကို ကြည့်ရုံနဲ့တင် ကျွန်တော် တကယ်ကို ထိတ်လန့်သွားမိတယ်။ ထရပ်လိုက်ပြီး သူ့နောက်ကနေ အခန်းရဲ့ တခြား တစ်ဖက်ကို လိုက်သွားလိုက်တယ်။

“ည ၉ နာရီလောက်မှာ ပြန်လှည့်လို့ မရတော့တဲ့ အခြေအနေ (point of no return) ကို ရောက်ခဲ့တာ မှတ်မိလား။ ကျွန်တော် Phoenix database conversion ရဲ့ အခြေအနေကို လိုက်ကြည့်နေတာ၊ အဲဒါက ကျွန်တော်တို့ ထင်ထားတာထက် အဆပေါင်းများစွာ ပိုပြီး နှေးနေတယ်။ အဲဒါက လွန်ခဲ့တဲ့ နာရီပေါင်းများစွာကတည်းက ပြီးရမှာ၊ ဒါပေမဲ့ အခုအထိ ၁၀ ရာခိုင်နှုန်းပဲ ပြီးသေးတယ်။ ဆိုလိုတာကတော့ အချက်အလက်တွေ အကုန်လုံး ပြောင်းလဲဖို့က အင်္ဂါနေ့အထိ ကြာဦးမှာပဲ။ ငါတို့တော့ သွားပြီ။”

ကျွန်တော် ထင်ထားတာထက် ပိုပြီး ပင်ပန်းနေလို့လားတော့ မသိဘူး၊ သူ ပြောတာကို ကျွန်တော် နားမလည်တော့ဘူး။ “အဲဒါက ဘာလို့ ပြဿနာ ဖြစ်ရတာလဲ” လို့ မေးလိုက်တယ်။

Wes က ထပ်ပြီး ရှင်းပြတယ်— “POS စနစ်တွေ ပြန်တက်လာဖို့အတွက် အဲဒီ script က အရင်ဆုံး ပြီးဖို့ လိုတယ်။ script ကို ရပ်လို့လည်း မရသလို ပြန်စလို့လည်း မရဘူး။ ပိုမြန်အောင် လုပ်ဖို့လည်း ဘယ်လိုမှ မဖြစ်နိုင်ဘူး။ Phoenix ကိုတော့ အလုပ်လုပ်အောင် ငါတို့ လုပ်လို့ ရမှာပါ၊ ဒါပေမဲ့ စတိုးဆိုင်တွေက POS စနစ်တွေကတော့ မသေချာဘူး—စမ်းသပ်ဖို့အတွက် ငါတို့ ဓာတ်ခွဲခန်းမှာ POS စက်တွေလည်း မရှိဘူးလေ။”

ဒုက္ခပါပဲ။

ကျွန်တော် နှစ်ခါ စဉ်းစားပြီးမှ— “Brent ကော?” လို့ မေးလိုက်တယ်။

သူက ခေါင်းယမ်းပြီး— “သူ့ကို မိနစ်အနည်းငယ်လောက် ကြည့်ခိုင်းပြီးပြီ။ တစ်ယောက်ယောက်က database indexing (ဒေတာဘေ့စ် အညွှန်း) ကို စောစောစီးစီး ဖွင့်လိုက်တဲ့အတွက် inserts (အချက်အလက် ထည့်သွင်းခြင်း) တွေက နှေးသွားတာလို့ သူ ထင်တယ်။ အခုချိန်မှာတော့ အချက်အလက်တွေ မပျက်စီးစေဘဲ ဒါကို လုပ်လို့ ရတာ ဘာမှ မရှိတော့ဘူး။ သူ့ကို Phoenix deployment ဆီပဲ ပြန်လွှတ်လိုက်တယ်” လို့ ပြောပါတယ်။

“တခြား အရာတွေကော ဘယ်လို ရှိလဲ” လို့ ကျွန်တော် အခြေအနေ အားလုံးကို သိချင်တာနဲ့ မေးလိုက်တယ်။ “performance ပိုင်းမှာရော တိုးတက်မှု ရှိလား။ database maintenance tools (ဒေတာဘေ့စ် ထိန်းသိမ်းရေး ကိရိယာများ) တွေကော ဘယ်လိုလဲ။”

“performance ကတော့ ဆိုးနေတုန်းပဲ” လို့ သူက ပြောတယ်။ “memory leak (မှတ်ဉာဏ် ယိုစိမ့်မှု) ကြီးတစ်ခု ရှိနေတယ်လို့ ထင်ရတယ်၊ အဲဒါက user တွေ မသုံးခင်မှာတောင် ဖြစ်နေတာနော်။ ငါ့လူတွေကတော့ ပေါက်ကွဲမသွားအောင်လို့ server တွေကို နာရီအနည်းငယ်ခြားတစ်ခါ reboot (ပြန်ဖွင့်) လုပ်ပေးရလိမ့်မယ်လို့ ထင်နေကြတယ်။ အသုံးမကျတဲ့ developer တွေ…”

သူ ဆက်ပြောတယ်— “ငါတို့ server နောက်ထပ် ၁၅ လုံးလောက်တော့ ရှာတွေ့ထားပါတယ်၊ တချို့က အသစ်တွေဖြစ်သလို တချို့ကတော့ ကုမ္ပဏီရဲ့ တခြား နေရာတွေကနေ ဆွဲထုတ်လာတာပေါ့။ ဒါပေမဲ့ မင်း ယုံချင်ယုံ၊ မယုံချင်နေ—အဲဒါတွေကို တပ်ဆင်ဖို့ data center မှာ နေရာ မရှိတော့ဘူး။recabling (ကြိုးတွေ ပြန်သွယ်တာ) နဲ့ rack တွေကို ပြန်စီရဦးမှာ။ Patty က သူ့လူတွေကို အဲဒါတွေ ကူညီပေးဖို့အတွက် လှမ်းခေါ်ထားတယ်” တဲ့။

ကျွန်တော် တကယ်ကို အံ့သြသွားပြီးမှ အရှေ့ကို ကိုင်းပြီး အော်ရယ်မိသွားတော့တယ်။ “အိုး… ဘုရားသခင်။ deploy လုပ်ဖို့ server တွေတော့ ရှာတွေ့ပါပြီ၊ ဒါပေမဲ့ အဲဒါတွေကို ထားဖို့ နေရာ ရှာမတွေ့ဘူးတဲ့။ အံ့သြစရာပါပဲ။ ငါတို့မှာ ကံကို မကောင်းနိုင်ဘူး!” လို့ ပြောလိုက်တာပေါ့။

Wes က ခေါင်းယမ်းပြီး— “ငါ့သူငယ်ချင်းတွေဆီက ဒါမျိုး ဇာတ်လမ်းတွေ ကြားဖူးပါတယ်၊ ဒါပေမဲ့ ဒါကတော့ deployment ကျဆုံးမှုတွေထဲမှာ အဆိုးဆုံး ဖြစ်လာလိမ့်မယ် ထင်တယ်” လို့ ပြောပါတယ်။

သူ ဆက်ပြောတယ်— “စိတ်ဝင်စားစရာ အကောင်းဆုံး အချက်ကတော့… ငါတို့တွေ virtualization (စနစ်များကို အတုပြုလုပ်ခြင်း) မှာ ပိုက်ဆံတွေ အများကြီး ရင်းနှီးမြှုပ်နှံခဲ့ကြတာက ဒီလို ကိစ္စတွေကနေ ကာကွယ်ပေးဖို့လေ။ ဒါပေမဲ့ Development က performance ပြဿနာတွေကို မပြင်နိုင်တဲ့အခါ virtualization ကိုပဲ အပြစ်ဖို့ကြတယ်လေ။ ဒါကြောင့် ငါတို့တွေ အရာရာတိုင်းကို physical servers (တကယ့် ဆာဗာအစစ်များ) ပေါ်ကို ပြန်ရွှေ့ခဲ့ကြရတာလေ!”

ပြီးတော့ Chris က virtualization က ငါတို့ကို ကူညီပေးမှာမို့လို့ ဒီ deployment ကို အတင်းအကျပ် လုပ်ခိုင်းခဲ့တာလေ။

ကျွန်တော် မျက်ရည်တွေကို သုတ်လိုက်ပြီး ရယ်နေတာကို အတင်း ထိန်းလိုက်ရတယ်။ “ဒါဆို developer တွေ ကတိပေးခဲ့တဲ့ database support tools တွေကော ဘယ်လိုလဲ။”

Wes ရဲ့ အပြုံးတွေက ချက်ချင်း ပျောက်ကွယ်သွားတယ်— “တကယ့် အမှိုက်တွေပဲ။ Phoenix ကနေ ဖြစ်လာတဲ့ အမှားတွေကို ငါ့လူတွေက database ထဲမှာ တစ်ခုချင်းစီ လိုက်ပြင်ပေးနေရတယ်။ ပြီးတော့ ပစ္စည်းဖြည့်တင်းတာတွေကိုလည်း လက်နဲ့ပဲ (manually) လိုက်လုပ်ပေးနေရတယ်။ ဒီလိုမျိုး manual work (လက်နဲ့လုပ်တဲ့အလုပ်) တွေ Phoenix မှာ ဘယ်လောက်တောင် လိုဦးမှာလဲဆိုတာ ငါတို့ ခုထိ လေ့လာနေတုန်းပဲ။ အဲဒါက အမှားတွေ အများကြီး ဖြစ်နိုင်သလို လူအင်အားလည်း အများကြီး လိုလိမ့်မယ်” လို့ ပြောပါတယ်။

ပျက်စီးနေတဲ့ application က လုပ်ရမယ့် အလုပ်တွေကို ငါ့လူတွေက တာဝန်ယူပြီး လုပ်နေရတာကို တွေးမိတော့ ကျွန်တော် စိတ်မကောင်းဖြစ်မိတယ်။ စာရင်းစစ် (auditors) တွေဟာ အချက်အလက်တွေကို အစီအမံ မရှိဘဲ၊ ထိန်းချုပ်မှု မရှိဘဲ တိုက်ရိုက် လိုက်ပြင်တာကို အကြောက်ဆုံး မဟုတ်လား။

“မင်း တကယ်ကို ကောင်းကောင်း လုပ်နိုင်ခဲ့ပါတယ်။ ငါတို့ရဲ့ ထိပ်တန်း ဦးစားပေးကတော့ မပြီးသေးတဲ့ database conversion ကြောင့် စတိုးဆိုင်က POS စနစ်တွေအပေါ် ဘယ်လိုမျိုး သက်ရောက်မှု ရှိမလဲဆိုတာကို ရှာဖွေဖို့ပဲ။ အဲဒါကို သေသေချာချာ သိတဲ့လူတစ်ယောက်ကို ရှာပြီး သူ့ရဲ့ အမြင်ကို မေးကြည့်လိုက်ပါ။ လိုအပ်ရင်တော့ Sarah ရဲ့ အဖွဲ့က retail operations (အရောင်းဆိုင် လည်ပတ်မှု) ကို ကိုင်တွယ်တဲ့ လူတစ်ယောက်ကို ဖုန်းဆက်လိုက်။ ပိုကောင်းတာကတော့ ငါတို့ ကိုယ်တိုင် စစ်ဆေးနိုင်အောင်လို့ POS စက်တစ်လုံးနဲ့ server တစ်ခုကို ရှာကြည့်လိုက်ပါ” လို့ ကျွန်တော် မှာလိုက်တယ်။

“နားလည်ပါပြီ” လို့ Wes က ခေါင်းညိတ်ပြီး— “အဲဒါအတွက် သင့်တော်မယ့် လူတစ်ယောက်ကို ငါ သိပါတယ်” လို့ ပြောပါတယ်။

သူ ထွက်သွားတာကို ကြည့်ပြီး ကျွန်တော် ဘယ်နေရာမှာ အသုံးဝင်ဆုံး ဖြစ်မလဲဆိုတာကို လိုက်ကြည့်နေမိတယ်။

မနက်ခင်း အလင်းရောင်ဟာ ပြတင်းပေါက်ကနေ ဝင်လာနေပြီး ကော်ဖီခွက်တွေ၊ စာရွက်တွေနဲ့ တခြား အမှိုက်တွေကို ထင်ထင်ရှားရှား မြင်နေရပါပြီ။ ထောင့်တစ်နေရာမှာတော့ developer တစ်ယောက်က ကုလားထိုင်တွေအောက်မှာ အိပ်ပျော်နေလေရဲ့။

ကျွန်တော် မျက်နှာသစ်ဖို့နဲ့ သွားတိုက်ဖို့ ရေချိုးခန်းကို အမြန် ပြေးသွားလိုက်တယ်။ နည်းနည်းတော့ လန်းသွားသလို ရှိပေမဲ့ တစ်ညလုံး မအိပ်ဘဲ လုပ်ခဲ့ရတာကတော့ နှစ်ပေါင်းများစွာ ရှိနေပါပြီ။

Maggie Lee ကတော့ Sarah ရဲ့ အောက်မှာ အလုပ်လုပ်တဲ့ Senior Director of Retail Program Management ပါ။ သူက မနက် ၇ နာရီ အရေးပေါ် အစည်းအဝေးကို စတင်လိုက်ပြီး လူ ၃၀ နီးပါးလောက် အခန်းထဲမှာ ပြည့်ကျပ်နေပါတယ်။ သူက ပင်ပန်းနေတဲ့ အသံနဲ့— “ဒါဟာ သူရဲကောင်းဆန်ဆန် ကြိုးစားခဲ့ကြတဲ့ ညတစ်ညပါပဲ၊ Phoenix အတွက် အစွမ်းကုန် ကြိုးစားခဲ့ကြတဲ့ လူတိုင်းကို ကျေးဇူးတင်ပါတယ်” လို့ ပြောပါတယ်။

“ရှင်တို့ သိတဲ့အတိုင်းပဲ၊ ဒီအရေးပေါ် အစည်းအဝေး ခေါ်ရတဲ့ အကြောင်းရင်းကတော့ database conversion မှာ တစ်ခုခု မှားယွင်းသွားလို့ပါ” လို့ သူက ဆက်ပြောတယ်။ “အဲဒါကြောင့် စတိုးဆိုင်တွေက POS စနစ်တွေ အကုန်လုံး ကျနေလိမ့်မယ်၊ ဆိုလိုတာကတော့ စတိုးဆိုင်တွေမှာ ငွေသိမ်းစက်တွေ အလုပ်လုပ်မှာ မဟုတ်ဘူး။ အဲဒီတော့ အရာအားလုံးကို လက်နဲ့ပဲ (manual) လုပ်ကြရပါလိမ့်မယ်။”

သူက ထပ်ပြီး— “သတင်းကောင်းကတော့ Phoenix website ကတော့ လည်ပတ်နေပြီဆိုတာပါပဲ” လို့ ပြောပြီး ကျွန်တော့်ဘက်ကို လက်ညှိုးထိုးပြကာ— “ဒါကို ဖြစ်မြောက်အောင် လုပ်ပေးခဲ့တဲ့ Bill နဲ့ IT Operations အဖွဲ့တစ်ခုလုံးကို ကျေးဇူးတင်ပါတယ်” လို့ ပြောလိုက်တာပေါ့။

ကျွန်တော် စိတ်တိုတိုနဲ့— “ကျွန်တော်ကတော့ Phoenix အစား အဲဒီ POS စနစ်တွေကိုပဲ အလုပ်လုပ်စေချင်တယ်။ NOC ထဲမှာတော့ တကယ့်ကို ရုတ်ရုတ်သဲသဲ ဖြစ်နေပြီ။ စတိုးဆိုင်တွေက လူတွေက သူတို့ စနစ်တွေ အလုပ်မလုပ်ဘူးလို့ အော်ဟစ်နေကြလို့ ကျွန်တော်တို့ ဖုန်းတွေက တစ်နာရီလုံးလုံး ထငြိမ်တာ မရှိသေးဘူး။ စတိုးဆိုင် ၁၂၀ က ဝန်ထမ်းတွေရဲ့ voicemail တွေနဲ့ ကျွန်တော့်ဖုန်းက ပြည့်နေပြီ။ ဖုန်းတွေကို ကိုင်ပေးဖို့အတွက် လူတွေ ပိုပြီး လိုအပ်ပါလိမ့်မယ်” လို့ ပြောလိုက်တယ်။

ကျွန်တော့်စကားကို ထောက်ခံသလိုမျိုး စားပွဲပေါ်က ဖုန်းတစ်လုံး တုန်ခါလာပြန်တယ်။

“ငါတို့တွေ ကြိုတင်ပြီးတော့ (proactive) အစီအစဉ်ဆွဲရမယ်” လို့ ကျွန်တော် Sarah ကို ပြောလိုက်တယ်။ “စတိုးဆိုင်တွေက လူတိုင်းဆီကို အခြေအနေ အကျဉ်းချုပ်နဲ့ POS စနစ်တွေ မပါဘဲနဲ့ အလုပ်တွေကို ဘယ်လို လုပ်ဆောင်ရမလဲဆိုတဲ့ ညွှန်ကြားချက်တွေကို အမြန်ဆုံး ပို့ပေးဖို့ လိုတယ်။”

Sarah က ခဏလောက် ကြောင်သွားပြီးမှ— “ဒါ ကောင်းတဲ့ အကြံပဲ။ မင်း အဲဒီ အီးမေးလ်ရဲ့ မူကြမ်းကို အရင် ရေးလိုက်ပါလား၊ အဲဒီနောက်မှ ငါတို့ ဆက်လုပ်ကြတာပေါ့” လို့ ပြောပါတယ်။

ကျွန်တော် အံ့သြသွားပြီး— “ဘာ! ကျွန်တော်က စတိုးဆိုင် မန်နေဂျာမှ မဟုတ်တာ! မင်းတို့ အဖွဲ့ကပဲ အရင် ရေးလိုက်ပါ၊ ပြီးမှ Chris နဲ့ ကျွန်တော်က မှန်မမှန် ပြန်စစ်ပေးမယ်လေ” လို့ ပြန်ပြောလိုက်တယ်။

Chris ကလည်း ခေါင်းညိတ်ပြပါတယ်။

Sarah က အခန်းကို ဝေ့ကြည့်လိုက်ပြီး— “အိုကေ… နောက် နာရီအနည်းငယ်အတွင်းမှာ တစ်ခုခု အဆင်သင့် ဖြစ်အောင် လုပ်လိုက်ပါ့မယ်” လို့ ပြောပါတယ်။

“မင်း နောက်နေတာလား” လို့ ကျွန်တော် အော်လိုက်တယ်။ “အရှေ့ဘက်ကမ်းခြေက စတိုးဆိုင်တွေ ဖွင့်ဖို့ မိနစ်ပိုင်းပဲ လိုတော့တာလေ—အခုချက်ချင်းပဲ တစ်ခုခု ပို့ပေးဖို့ လိုတာ!”

“ကျွန်မ ကိုင်တွယ်လိုက်ပါ့မယ်” လို့ Maggie က လက်ညှိုးထောင်ပြီး ပြောပါတယ်။ သူက ချက်ချင်းပဲ laptop ကိုဖွင့်ပြီး စာရိုက်တော့တာပါပဲ။

ကျွန်တော် ခေါင်းကို လက်နှစ်ဖက်နဲ့ အုပ်လိုက်ပြီး ခေါင်းကိုက်တာ သက်သာမလားလို့ ကြည့်ရင်း ဒီ rollout က ဒါထက် ပိုပြီး ဘယ်လောက်တောင် ဆိုး ဦးမှာလဲလို့ စဉ်းစားနေမိပါတယ်။

စနေနေ့ နေ့လည် ၂ နာရီ ရောက်တဲ့အခါမှာတော့ အခြေအနေက ကျွန်တော် ထင်ထားတာထက် အများကြီး ပိုပြီး ဆိုးနေပါပြီ။

စတိုးဆိုင် အားလုံးဟာ manual fallback mode (လက်နဲ့လုပ်တဲ့စနစ်) နဲ့ပဲ လည်ပတ်နေကြရတယ်။ အရောင်းအဝယ် အားလုံးကို လက်နဲ့လှည့်ရတဲ့ credit card imprint machines တွေနဲ့ လုပ်နေကြရပြီး မှတ်တမ်းတွေကိုတော့ ဖိနပ်ဗူးတွေထဲမှာ သိမ်းထားကြရတာပေါ့။

စတိုးဆိုင် မန်နေဂျာတွေကတော့ သူတို့ရဲ့ ဝန်ထမ်းတွေကို အနီးအနားက ရုံးသုံးပစ္စည်းဆိုင်တွေဆီ လွှတ်ပြီး carbon paper တွေ လိုက်ရှာခိုင်းနေရသလို ဘဏ်တွေဆီကိုလည်း အကြွေတွေ သွားလဲခိုင်းနေရပါတယ်။

Phoenix website ကို သုံးနေတဲ့ ဖောက်သည်တွေကလည်း စနစ်က ကျနေတာ ဒါမှမဟုတ် အရမ်း နှေးလွန်းလို့ သုံးလို့ မရတာတွေကြောင့် ညည်းတွားနေကြပါတယ်။ ငါတို့တွေဟာ Twitter မှာတောင် trending topic (လူပြောအများဆုံး အကြောင်းအရာ) ဖြစ်သွားခဲ့ပါတယ်။ တီဗွီနဲ့ သတင်းစာ ကြော်ငြာတွေ မြင်ပြီး ဝန်ဆောင်မှုကို စမ်းသုံးချင်နေကြတဲ့ ငါတို့ရဲ့ ဖောက်သည်တွေကတော့ အိုင်တီ ကျရှုံးမှု (big IT fail) အပေါ်မှာ အပြစ်တင်နေကြပါပြီ။

အွန်လိုင်းကနေ အော်ဒါမှာနိုင်ခဲ့တဲ့ ဖောက်သည်တွေကျတော့လည်း စတိုးဆိုင်ကို ပစ္စည်းသွားယူတဲ့အခါမှာ ပြဿနာ တက်ကြပြန်ရော။ Phoenix ဟာ အရောင်း အချက်အလက်တွေကို ကျပန်း ပျောက်ဆုံးစေသလို တချို့ ကိစ္စတွေမှာဆိုရင် ဖောက်သည်တွေရဲ့ ခရက်ဒစ်ကတ်တွေကို နှစ်ခါ၊ သုံးခါလောက် ဖြတ်တောက်နေတာကို တွေ့လိုက်ရတယ်။

အရောင်း အချက်အလက်တွေ (sales order data) ပျက်စီးသွားမှာကို စိုးရိမ်တဲ့အတွက် Finance (ဘဏ္ဍာရေး) က Ann က ရုံးကို ရောက်လာပြီး သူ့အဖွဲ့ကို အခန်းရဲ့ တစ်ဖက်မှာ war room အသစ်တစ်ခု ဖွင့်ခိုင်းလိုက်ပါတယ်။ စတိုးဆိုင်တွေက လာတဲ့ ပြဿနာရှိတဲ့ အော်ဒါတွေကို ကိုင်တွယ်ဖို့ပေါ့။ နေ့လည်ပိုင်းမှာဆိုရင်တော့ စတိုးဆိုင်တွေကနေ fax နဲ့ ပို့လာတဲ့ ဒေါသထွက်နေတဲ့ ဖောက်သည် ရာပေါင်းများစွာရဲ့ စာရွက်တွေဟာ ပုံနေပါပြီ။

Ann ကို ကူညီဖို့အတွက် Wes က အင်ဂျင်နီယာတွေကို ခေါ်ပြီး လွဲမှားနေတဲ့ အော်ဒါတွေကို ပြုပြင်ပေးမယ့် tools အချို့ကို အမြန် လုပ်ပေးခိုင်းရပါတယ်။

NOC စားပွဲကို တတိယအကြိမ်မြောက် ဖြတ်လျှောက်တဲ့အခါမှာတော့ ကျွန်တော်ဟာ အရမ်း ပင်ပန်းနေပြီဖြစ်လို့ ဘယ်သူ့ကိုမှ မကူညီနိုင်တော့ဘူးဆိုတာ သိလိုက်ရတယ်။ အခု နေ့လည် ၂:၃၀ ရှိပါပြီ။

Wes က တစ်ဖက်က လူတစ်ယောက်နဲ့ ငြင်းခုန်နေတာကြောင့် သူ ပြီးတဲ့အထိ ကျွန်တော် စောင့်နေလိုက်တယ်။ “ဒါဟာ ရက်ပေါင်းများစွာ ကြာမယ့် ပြဿနာကြီးဆိုတာ ငါတို့ အခု လက်ခံလိုက်ရအောင်။ မင်း အခြေအနေ ဘယ်လို ရှိလဲ” လို့ ကျွန်တော် မေးလိုက်တယ်။

သူက သန်းဝေလိုက်ပြီး— “ကျွန်တော် တစ်နာရီလောက်တော့ အိပ်လိုက်ရပါသေးတယ်။ ဟာ… ဆရာ့ မျက်နှာက တကယ်ကို ကြည့်ရဆိုးနေပြီ။ အိမ်ပြန်ပြီး နာရီအနည်းငယ်လောက် သွားအိပ်လိုက်ပါဦး။ ဒီနေရာကိုတော့ ကျွန်တော် ထိန်းထားနိုင်ပါတယ်။ တစ်ခုခု ဖြစ်ရင်တော့ ဖုန်းဆက်လိုက်ပါ့မယ်” လို့ ပြန်ဖြေပါတယ်။

မငြင်းနိုင်တော့တာနဲ့ပဲ ကျွန်တော် သူ့ကို ကျေးဇူးတင်ကြောင်း ပြောပြီး ထွက်လာခဲ့တယ်။

ဖုန်းသံ ကြားလိုက်ရလို့ ကျွန်တော် လန့်နိုးသွားတယ်။ ဖုန်းကို ကောက်ကိုင်လိုက်တော့ ညနေ ၄:၃၀ ရှိပါပြီ။ Wes ဆီက ဖုန်းပါ။

ကျွန်တော် သတိပြန်ဝင်လာအောင် ခေါင်းကို ခဏလောက် ယမ်းလိုက်ပြီး— “ဘာဖြစ်လို့လဲ” လို့ မေးလိုက်တယ်။

သူ့အသံကို ကြားလိုက်ရတယ်— “သတင်းဆိုးပါပဲ။ အကျဉ်းချုပ် ပြောရရင်တော့… Phoenix website က ဖောက်သည်တွေရဲ့ credit card numbers တွေကို ပေါက်ကြား (leaking) စေနေတယ်ဆိုတာ Twitter တစ်ခွင်မှာ ပြန့်နေပြီ။ Screenshot တွေတောင် ပါနေတာ။ ဖောက်သည်တွေက သူတို့ရဲ့ shopping cart (ပစ္စည်းထည့်တဲ့ခြင်း) ကို ဖျက်လိုက်တဲ့အခါ စနစ်က session crash ဖြစ်သွားပြီး အောင်မြင်ခဲ့တဲ့ နောက်ဆုံး အော်ဒါရဲ့ credit card နံပါတ်ကို ပြနေတာတဲ့။”

ကျွန်တော် ကုတင်ပေါ်ကနေ ဝုန်းခနဲ ထလိုက်ပြီး ချက်ချင်းပဲ ရေချိုးခန်းဆီကို ပြေးသွားလိုက်တယ်။ “John ဆီ ဖုန်းဆက်လိုက်။ သူတော့ ရူးသွားမှာပဲ။ အဲဒါအတွက် စာရွက်စာတမ်းတွေ အများကြီး လုပ်ရမှာဖြစ်သလို ရဲတွေကိုပါ အကြောင်းကြားရတာမျိုး ဖြစ်နိုင်တယ်။ ရှေ့နေတွေလည်း ပါလာမှာပေါ့။”

Wes က— “သူ့ဆီ ဖုန်းဆက်ပြီးပါပြီ။ သူနဲ့ သူ့အဖွဲ့ လာနေကြပြီ။ သူကတော့ တကယ်ကို စိတ်တိုနေတာ။ *Pulp Fiction* ထဲက လူလိုပဲ။ စာရင်းရှင်းရမယ့်နေ့ အကြောင်းနဲ့ ပြင်းထန်တဲ့ ဒေါသနဲ့ လူတွေကို နှိမ်နင်းမယ့်အကြောင်း တွေတောင် သူ ပြောနေသေးတယ်” လို့ ပြန်ဖြေပါတယ်။

ကျွန်တော် ရယ်လိုက်မိတယ်။ John Travolta နဲ့ Samuel L. Jackson တို့ရဲ့ အဲဒီ အခန်းကို ကျွန်တော် အကြိုက်ဆုံးလေ။

ငါတို့ရဲ့ အေးအေးဆေးဆေး နေတတ်တဲ့ CISO (သတင်းအချက်အလက် လုံခြုံရေး အရာရှိ) က အဲဒီလို ဖြစ်လာလိမ့်မယ်လို့ ကျွန်တော် မထင်ခဲ့မိပါဘူး။ ဒါပေမဲ့ လူတွေ ပြောကြသလိုပဲပေါ့—အေးဆေးတဲ့ လူတွေကိုတော့ ပိုပြီး သတိထားရမယ်လေ။

ကျွန်တော် အမြန် ရေချိုးလိုက်တယ်။ မီးဖိုချောင်ထဲကို ပြေးဝင်ပြီး သားလေး စားလေ့ရှိတဲ့ string cheese လေး ငါးချောင်းလောက် ယူလိုက်ကာ ကားပေါ် ဆက်တိုက် တက်မောင်းလာခဲ့တော့တယ်။

လမ်းမကြီးပေါ် ရောက်တဲ့အခါ Paige ဆီကို ဖုန်းခေါ်လိုက်တယ်။ “ကလေးရေ… မင်း ဘယ်မှာလဲ။ ငါ အခု အလုပ်မှာ၊ ကလေးတွေကတော့ ငါ့အမေနဲ့ ရှိနေတယ်” တဲ့။

ကျွန်တော်က— “ငါ အိမ်မှာ တစ်နာရီလောက် ရှိခဲ့သေးတယ်။ ကုတင်ပေါ် ရောက်တာနဲ့ အိပ်ပျော်သွားတာ၊ ဒါပေမဲ့ Wes ခုနက ဖုန်းဆက်တယ်လေ။ Phoenix website က လူတွေရဲ့ ခရက်ဒစ်ကတ် နံပါတ်တွေကို တစ်ကမ္ဘာလုံး မြင်အောင် ပြနေတာတဲ့။ ဒါဟာ တကယ့်ကို ကြီးမားတဲ့ security breach (လုံခြုံရေး ဖောက်ဖျက်မှု) ပဲ၊ အဲဒါကြောင့် ငါ အခု ရုံးကို ပြန်မောင်းလာနေတယ်” လို့ ပြောလိုက်တာပေါ့။

သူ့ရဲ့ သက်ပြင်းချသံကို ကြားလိုက်ရတယ်— “မင်း အဲဒီမှာ လုပ်နေတာ ၁၀ နှစ်ကျော်ပြီ၊ ဒီလိုမျိုး အချိန်တွေအထိ မင်း တစ်ခါမှ အလုပ်မလုပ်ခဲ့ဖူးပါဘူး။ ဒီ ရာထူးတိုးတာကို ငါ တကယ်ပဲ သဘောကျရဲ့လား မသိတော့ဘူး” တဲ့။

“ငါလည်း မင်းလိုပဲ ခံစားရပါတယ် ကလေးရေ…” လို့ ကျွန်တော် ပြောလိုက်ရတော့တာပေါ့။
